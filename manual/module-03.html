FM-03 · Architecture

Multi-Agent Architectures

Six topologies compared on the one axis that matters — the coordination tax versus what context isolation actually buys you — typed handoff semantics, a full-stack orchestrator-workers build, and the steelmanned case for staying single-agent.

PUBLISHED 2026-07-11 DEPTH ~5,500 words PREREQS Module 02 LAB orchestrator-workers researcher vs. one well-tooled agent, evals decide

§ 3.0 Orientation

The trace for one "evaluate this vendor" request shows a coordinator spawning six subagents. Four of them independently open the same 38,000-token compliance PDF — none knew a sibling had already fetched it, because none of them can see each other's tool calls. Two return summaries that disagree on the contract's renewal date, because one read a cached copy from an earlier turn and the prompt cache had gone stale. Total cost for the request: just under 320,000 tokens and 94 seconds. A single agent with the same three tools and a bigger context window answered the identical question, correctly, in 11,000 tokens and 7 seconds. The team had built a multi-agent system because the tutorial said parallel subagents are faster — nobody had priced what six agents cost to coordinate versus what isolating each one's context was supposed to buy back.

Before frameworks made "spin up a crew of agents" one line of code, teams facing a task too big for one context window had exactly two options: chain fixed-shape calls together (Module 00 §0.2's prompt-chaining and routing patterns, still one control loop, one context) or write a monolithic agent with a bigger tool belt (Module 02) and hope the window held. Multi-agent architectures are the third option — genuinely separate agents, each with its own multi-turn loop and its own context window, coordinating to solve one task. Module 00 §0.2 already introduced orchestrator-workers as a workflow pattern, but there the "workers" were single LLM calls directed by one coordinating call — still one process, one control loop, context shared by construction. This module is about what changes once workers become full agents: independent context windows, independent tool loops, and a coordination problem that has to be engineered, not assumed away.

The throughline: multi-agent systems have exactly one honest justification — context isolation — and exactly one honest cost — the coordination tax spent buying that isolation. Every topology in this module is a different way of trading one against the other. Get that trade wrong, as in the cold open above, and you've paid the tax without collecting the isolation's benefit.

§ 3.1 The coordination tax: context isolation is the point, not a side effect

Here is the only reason to reach for separate agents instead of one bigger agent: each subagent gets a context window free of every other subagent's tool noise, free of subtasks it doesn't need, and — critically — free of the context poisoning and attention dilution that Module 04 covers in depth (a long, noisy transcript degrades a model's ability to use the parts of it that still matter). If a task's subtasks are large enough or independent enough that combining them in one context would blow the window or bury the signal, isolation earns its keep. If they aren't, you're paying for isolation you didn't need.

What you pay for that isolation is the coordination tax: every token and turn spent packaging context for a handoff, deduplicating work two agents both decided to do, and reconciling outputs that disagree because they never saw each other's reasoning. A rough accounting, useful in any design review:

total_cost ≈ Σ(per_worker_cost) + dispatch_overhead + synthesis_overhead + coordination_tax

where coordination_tax ≈ redundant_fetches + conflict_resolution + reconciliation_of_disagreement

Multi-agent wins only when the reduction in per_worker_cost from a smaller, focused context — fewer tokens per call, fewer distractor tools, less attention dilution — outweighs everything on the right of that first plus sign. Nobody measures the right-hand side before shipping; that is the single most common root cause behind a multi-agent system that costs more and performs the same as the single agent it replaced.

§ 3.2 Six topologies, priced on the tax

Every multi-agent system is one of a small number of coordination shapes. Two earn a diagram here because their mechanics differ most from the workflow-level patterns Module 00 already covered; the rest are priced in the table that follows.

Orchestrator-workersDispatch briefs + synthesize findings — moderate, paid onceFull, across independent workersTask decomposes upfront into non-overlapping, independent subtasks
HierarchicalCompounds — every extra level adds its own dispatch+synthesis taxFull within each levelSubtasks are themselves decomposable (research → per-topic sub-research)
Peer-swarmHigh — agents negotiate/claim from a shared queue with no fixed leaderFull, opportunisticNo natural coordinator; resilience to one agent's failure matters more than efficiency

PipelineLowest — no negotiation, no fan-out, one handoff per stageNone — strictly sequentialSteps are known, ordered, and each stage's output is exactly the next stage's input

Pipeline deserves one clarifying note: it is the multi-agent-process version of the same idea Module 01 §1.2 built as a single agent's deterministic DAG scheduler. The difference is where the boundary sits — §1.2's nodes were function calls inside one process; a pipeline topology's stages are separate agents, possibly separate services, so the "typed proposal boundary" from Module 02 §2.2 has to actually cross a process or network boundary between S2 and S3, not just a Python function call.

§ 3.3 Handoff semantics: what actually crosses the isolation boundary

Isolation is only real if you resist the easiest way to break it: handing the next agent your entire transcript "to be safe." Three tiers exist, and most working systems use two of them deliberately rather than one of them by accident.

The brief and the artifact are both instances of Module 02 §2.2's typed proposal boundary, applied to inter-agent handoffs instead of tool calls: the orchestrator proposes work as a validated object, and a worker's result is accepted only as a validated object, never a trusted blob of prose.

from pydantic import BaseModel, Field

class SubagentBrief(BaseModel):
    subagent_id: str
    task: str
    context_refs: list[str]              # pointers the worker fetches itself — never the inlined transcript
    constraints: list[str] = Field(default_factory=list)

class Finding(BaseModel):
    subagent_id: str
    claim: str
    evidence_refs: list[str]
    confidence: float = Field(ge=0, le=1)

context_refs is the detail that prevents the cold open's redundant-fetch problem: a brief carries pointers (file paths, document IDs, row keys), not inlined text, so the orchestrator can dedupe references before any worker runs — see §3.4's dispatch().

§ 3.4 Full-stack build: an orchestrator-workers researcher

The topology from FIG 3.1's left panel, built end to end. Three layers: the agent runtime dispatches deduplicated briefs to isolated workers and synthesizes only their typed findings; the API streams per-worker progress over SSE; the frontend renders a live tree. A fourth layer — one Postgres table — is the difference between "restart from zero" and "resume" when the process crashes mid-run.

Layer 1 — agent runtime: dispatch with pre-run dedupe, isolated workers (Python)

# pip install anthropic pydantic
from pydantic import BaseModel, Field
from concurrent.futures import ThreadPoolExecutor, as_completed
import anthropic, json

class SubagentBrief(BaseModel):
    subagent_id: str
    task: str
    context_refs: list[str]
    constraints: list[str] = Field(default_factory=list)

class Finding(BaseModel):
    subagent_id: str
    claim: str
    evidence_refs: list[str]
    confidence: float = Field(ge=0, le=1)

client = anthropic.Anthropic()

WORKER_SYSTEM = """You are a research worker. You receive ONE brief and nothing else —
no sibling worker's output, no orchestrator reasoning. Fetch only the context_refs listed,
respect every constraint, and respond with JSON matching:
{"subagent_id": str, "claim": str, "evidence_refs": [str], "confidence": float}"""

def fetch_ref(path: str) -> str:
    with open(path, encoding="utf-8") as f:
        return f.read()[:4000]

def run_worker(brief: SubagentBrief) -> Finding:
    context = "\n\n".join(fetch_ref(r) for r in brief.context_refs)
    resp = client.messages.create(
        model="claude-sonnet-5", max_tokens=1024, system=WORKER_SYSTEM,
        messages=[{"role": "user", "content":
            f"{brief.task}\n\nContext:\n{context}\n\nConstraints: {brief.constraints}"}],
    )
    return Finding.model_validate(json.loads(resp.content[-1].text))  # typed boundary — same rule as M01 §1.5 / M02 §2.2

def dispatch(briefs: list[SubagentBrief]) -> list[Finding]:
    claimed: set[str] = set()
    for b in briefs:                                  # first brief wins each ref — this is the coordination-tax fix
        b.context_refs = [r for r in b.context_refs if r not in claimed]
        claimed.update(b.context_refs)
    with ThreadPoolExecutor(max_workers=len(briefs)) as pool:
        futures = [pool.submit(run_worker, b) for b in briefs]
        return [f.result() for f in as_completed(futures)]

def synthesize(findings: list[Finding]) -> str:
    bullets = "\n".join(f"- {f.claim} (confidence {f.confidence})" for f in findings)
    resp = client.messages.create(model="claude-sonnet-5", max_tokens=1024,
        messages=[{"role": "user", "content": f"Synthesize one answer from these findings:\n{bullets}"}])
    return resp.content[-1].text                       # orchestrator never sees a worker's raw transcript

Layer 2 — API/transport: streaming worker completions over SSE (Python, FastAPI)

# pip install fastapi uvicorn sse-starlette
from fastapi import FastAPI
from sse_starlette.sse import EventSourceResponse
from concurrent.futures import ThreadPoolExecutor, as_completed
import asyncio, json, queue

app = FastAPI()
JOBS = {
    "vendor-abc": [
        SubagentBrief(subagent_id="w1", task="Summarize security posture",
                      context_refs=["docs/vendor-abc/security.md"]),
        SubagentBrief(subagent_id="w2", task="Summarize pricing terms",
                      context_refs=["docs/vendor-abc/pricing.md"]),
        SubagentBrief(subagent_id="w3", task="Summarize SLA terms",
                      context_refs=["docs/vendor-abc/sla.md"]),
    ]
}

def run_all(briefs: list[SubagentBrief], q: "queue.Queue"):
    with ThreadPoolExecutor(max_workers=len(briefs)) as pool:
        futures = {pool.submit(run_worker, b): b for b in briefs}
        for f in as_completed(futures):
            q.put(f.result())
    q.put(None)                                         # sentinel: every worker finished

@app.get("/research/{job_id}/events")
async def research_events(job_id: str):
    async def stream():
        briefs = JOBS[job_id]
        yield {"event": "dispatch", "data": json.dumps([b.model_dump() for b in briefs])}
        q: queue.Queue = queue.Queue()
        loop = asyncio.get_event_loop()
        loop.run_in_executor(None, run_all, briefs, q)   # workers run concurrently in the background
        findings = []
        while True:
            finding = await asyncio.to_thread(q.get)      # drains in arrival order, not dispatch order
            if finding is None:
                break
            findings.append(finding)
            yield {"event": "subagent_done", "data": json.dumps(finding.model_dump())}
        yield {"event": "complete", "data": json.dumps({"answer": synthesize(findings)})}
    return EventSourceResponse(stream())

The fourth layer — data/infra — is one table: create table findings (job_id uuid, subagent_id text, claim text, evidence_refs jsonb, confidence real, created_at timestamptz default now(), primary key (job_id, subagent_id)). A crashed orchestrator resumes by re-dispatching only the briefs whose subagent_id isn't already a row — not every worker from scratch. Module 09 builds the full checkpoint/resume harness on top of exactly this shape; here it's enough that the SSE stream and the scheduler's writes go through the same table, so a client that reconnects mid-run sees consistent state instead of racing the workers.

§ 3.5 The steelmanned case against multi-agent

Cognition's widely-read "Don't Build Multi-Agents" argues the failure mode plainly: context degrades at every handoff, a distributed trace across N agents is dramatically harder to debug than one agent's linear transcript, and most demos showing multi-agent "wins" are actually workflows — fixed decomposition, no real coordination problem — wearing a multi-agent label. All three points are correct as stated, and all three are exactly the coordination tax from §3.1, named from the debugging side instead of the cost side.

Anthropic's own account of building a production multi-agent research system is the honest counterweight: for genuinely open-ended, breadth-first research tasks, a lead agent coordinating parallel subagents beat single-agent Claude by 90.2% on their internal eval, largely because the task was naturally decomposable into independent, context-heavy searches that would have blown one context window trying to hold all the raw search results at once. The same post is candid about the tax: early versions spawned 50 subagents for simple queries that needed three, agents scoured the web for sources that didn't exist, and subagents distracted each other with excessive status updates before the team fixed it with tighter prompts and explicit fan-out limits.

Both are true at once. The variable that decides which side you land on is whether your subtasks are read-heavy and independent — little cross-talk needed, each subtask's context doesn't fit comfortably alongside the others — or whether they share most of their context, in which case one agent with better tools and a bigger window is strictly cheaper and strictly easier to debug.

SignalFavors multi-agentFavors one well-tooled agent
Subtask context overlapLow — each subtask needs different, large source materialHigh — subtasks reference the same few documents
Debuggability requirementSecondary to latency/throughputPrimary — one linear trace is non-negotiable
Task shapeGenuinely open-ended, breadth matters more than a fixed planFixed, known steps — that's Module 00's workflow taxonomy, not this module
Cost toleranceLatency/throughput gain is worth N× the token spendCost-per-task is the constraint

§ 3.6 Failure modes

Failure modes — multi-agent architectures
  1. Redundant context fetches (the cold open). Symptom: two or more subagent traces within the same request show identical tool calls against the same file or record. Mitigation: dedupe context_refs across all briefs before dispatch, as in §3.4's dispatch() — not after the fact.
  2. Lost-in-the-brief. The orchestrator's brief omits a constraint the worker needed, so the worker's output is internally consistent and looks correct in its own trace, but violates a rule visible only in context the orchestrator never passed down. Symptom: an escalation or incident where the worker "did exactly what it was told" and the told was incomplete. Mitigation: treat brief-writing as its own reviewed artifact (the same discipline Module 02 §2.2 applies to tool schemas), not a string an orchestrator improvises per call.
  3. Coordination deadlock/livelock in blackboard or peer-swarm topologies. Two agents each wait for the other's write, or repeatedly overwrite each other's contribution to the shared state. Symptom: the blackboard's value for a given key oscillates between two states across turns, or a key's write-count climbs with the task's overall completion signal flat. Mitigation: version the blackboard (optimistic concurrency, not last-write-wins) and cap the number of opportunistic write attempts per agent per task.
  4. Runaway subagent spawning. A coordinator (or worse, a hierarchical coordinator's sub-coordinator) recursively decomposes without a fan-out cap. Anthropic's own account of this: early versions of their research system spawned 50 subagents for queries that needed three. Symptom: active-subagent count climbs faster than task complexity justifies, visible as a step-function jump in the cost-per-task dashboard. Mitigation: a hard fan-out limit enforced in the deterministic dispatcher, never left to the planner's discretion.
  5. Debate false consensus. N debating agents converge on unanimous agreement not because the answer is right but because they share the same training-induced blind spot, or because a majority vote overrides the one dissenting agent that happened to be correct. Symptom: the debate "passes" (unanimous or majority) with no dissent logged, yet golden-set accuracy on that task class doesn't improve — or gets worse — compared to a single well-prompted agent.
Principal's Judgment Call — multi-agent or one well-tooled agent?

Reach for multi-agent when subtasks are genuinely independent, each needs context large or specialized enough that combining them would blow one window or dilute attention, and you're willing to pay dispatch, synthesis, and reconciliation overhead for the isolation. Stay with one agent, better tooled, when subtasks overlap in what they need to know, when a single linear trace matters more than throughput, or — the tell that actually settles most reviews — when you can't name what a worker's isolated context buys you beyond "it felt like the modern way to build it."

Measure before committing either way: run the task both as an orchestrator-workers system and as one agent with the union of the workers' tools, on the same golden set, and compare accuracy, cost, and latency. §3.8's lab is that comparison. Teams that skip it are the ones who end up in this module's cold open.

Principal's Judgment Call — which topology, once you've committed to multi-agent

Reach for orchestrator-workers or hierarchical when the decomposition is knowable before dispatch — you can sketch which subtasks exist and that they don't depend on each other's output. Reach for pipeline when the steps are strictly ordered and each stage's input is exactly the prior stage's output — it's the lowest-tax topology precisely because it negotiates nothing. Reach for blackboard only when the decomposition is genuinely emergent — you don't know the shape of the problem until partial results reveal it — and accept that its tax (contention, staleness) is the price of not having to plan upfront. Reach for debate only after you've measured a single agent's error rate on the specific failure mode in question; if you haven't measured it, you're guessing at whether N× the cost is worth it.

§ 3.7 Lab — orchestrator-workers researcher vs. one well-tooled agent

LAB 03 · Estimated 4–6 hours

Mission: build §3.4's orchestrator-workers researcher for real, then build a single agent with the union of all three workers' tools and a bigger context window, and let evals — not intuition — decide which one you'd ship.

Spec:

  1. Pick a research task family with 3–5 genuinely independent sub-questions over real source documents (a vendor evaluation, a multi-file codebase audit, a compliance review — anything where the sub-questions don't need each other's answers to be answered).
  2. Implement §3.4's orchestrator-workers version exactly, including the pre-dispatch dedupe. Wire it to the SSE endpoint and either the React component or a minimal HTML page so a run is observable live.
  3. Implement a single-agent version: one context window, one tool loop, access to every source document the three workers had.
  4. Run both against 5 real task instances. Record total tokens, wall-clock latency, and an accuracy score against a golden answer you write by hand for each instance.

Acceptance criteria:

  • ✅ curl -N http://localhost:8000/research/vendor-abc/events streams subagent_done events in arrival order — confirm at least one run where the events arrive out of dispatch order (worker 3 finishes before worker 1).
  • ✅ A table: architecture × (accuracy, total tokens, p50 latency) across the 5 task instances, for both versions.
  • ✅ A written verdict, 5–8 sentences, naming which architecture you'd ship and citing the specific numbers — not "the multi-agent one felt more thorough."
  • ✅ One induced failure captured in a transcript: force two briefs to claim an overlapping context_ref before the dedupe fix, show the redundant fetch, then show the fix removing it.

Stretch: add a fan-out cap to the dispatcher and a test that proves a pathological brief-generator (one that tries to spawn 20 workers for a 3-subtask job) gets capped, not honored.

§ 3.8 Consultant's checklist

Questions to ask any team running a multi-agent system
  • What does context isolation buy this specific task — named concretely, not "it's more scalable"?
  • Has anyone measured the coordination tax (redundant fetches, reconciliation, dispatch/synthesis overhead) against a single-agent baseline on the same task?
  • What crosses a handoff boundary — a full transcript, a typed brief, or a typed artifact? If it's a full transcript, why is this multi-agent at all?
  • Is there a hard fan-out cap enforced in deterministic code, or is subagent count left to a planner's discretion?
  • For any blackboard or peer-swarm system: what happens when two agents write the same key in the same round — last-write-wins, or versioned/optimistic concurrency?
  • For any debate topology: what's the measured error rate of a single agent on the failure mode being guarded against, and does N× the cost actually buy a proportional accuracy gain?

§ 3.9 Design-review questions (staff+)

Interview / design-review bank
  1. A team proposes five agents for a task you suspect one agent could do. What's your first question? Strong answers ask what each proposed agent's isolated context contains that the others' don't — if the answer is "they all need most of the same source material," that's the tell for one agent with better tools, not five agents negotiating.
  2. Your orchestrator-workers system's cost tripled after adding a fourth worker type, with no accuracy change. Diagnose. Strong answers check for overlapping context_refs across the new worker type and the existing three first (the cold-open failure), then check whether the new worker type introduced a blackboard-style shared resource that the others now contend over.
  3. Design the handoff for an orchestrator that dispatches work to workers built on a different framework than the orchestrator itself. Strong answers reach for an artifact-mediated handoff — a typed brief/finding schema serialized as JSON crossing a process or network boundary, framework-agnostic by construction — and explicitly flag that once the workers are a different team's or a different org's agents, this becomes Module 10's A2A territory, not this module's.
  4. Steelman both sides: "multi-agent systems are strictly harder to debug than single agents." Strong answers concede the trace is genuinely harder to read linearly and that Cognition's critique is correct as far as it goes, then counter that a well-typed Finding/Brief boundary makes each worker's contribution independently auditable in a way a single agent's 40-turn transcript often isn't — landing on "harder to read as one trace, easier to audit as discrete typed contributions," not a flat yes/no.
  5. A blackboard-based system intermittently produces different final answers for the identical input. Where do you look first? Strong answers suspect write-order nondeterminism on the shared state before suspecting model nondeterminism — check whether two agents wrote conflicting keys in the same round and whether the resolution policy is deterministic (versioned) or accidental (whichever thread's write lands last).
