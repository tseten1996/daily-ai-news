Mental Models: What an Agent Actually Is

The honest definition, the workflow-vs-agent taxonomy, the autonomy spectrum, and the bitter lesson — the four mental models every other module builds on.

PUBLISHED 2026-07-10 DEPTH ~4,500 words PREREQS none LAB build a no-framework agent

§ 0.0 Orientation

Most confusion about agentic systems traces back to fuzzy definitions. "Agent" gets used for everything from an autocomplete feature to a fleet of autonomous processes, and architecture debates go in circles because the participants are picturing different things. This module fixes the vocabulary. By the end you should be able to look at any "AI agent" — a product, a framework demo, a vendor pitch — and answer three questions precisely: where does control flow live, what can it actually do, and what happens when it's wrong?

Everything else in this manual — context engineering, MCP, harnesses, multi-agent systems, evals — is elaboration on the four models in this module.

§ 0.1 The agent loop: LLM + tools + loop

Strip away the branding and an agent is this: a language model, given a set of tools, called in a loop, where the model's output determines what happens next. Each iteration: assemble context → run inference → if the model requested a tool, execute it and append the result to context → repeat until the model produces a final answer (or a guard stops it).

The classic framing — perceive → reason → act → observe — maps directly onto engineering artifacts. Perceive is context assembly (system prompt, conversation, tool results, retrieved data). Reason is a single inference pass. Act is your code executing a tool call. Observe is appending the tool's output back into context. There is no additional magic layer; if you can read a while loop, you can read an agent.

The loop has exactly one property that makes it different from every other system you've built: control flow is decided at runtime, by a probabilistic component. That single property is the source of all agentic power (it handles paths you didn't enumerate) and of every failure mode in this manual (it takes paths you didn't intend). Hold onto that framing — most of the remaining fifteen modules are techniques for keeping the benefit while capping the damage.

The whole thing in code

Here is a complete, working agent — no framework. The Python track shows the raw loop, which you should write at least once in your life; the TypeScript track uses the Claude Agent SDK, where the loop, tools, and permissions come pre-built as a harness; the Java track uses Spring AI, which runs the tool loop inside the framework. All three are the same diagram from FIG 0.1.

# pip install anthropic
import anthropic

client = anthropic.Anthropic()

TOOLS = [{
    "name": "read_file",
    "description": "Read a UTF-8 text file and return its contents.",
    "input_schema": {
        "type": "object",
        "properties": {"path": {"type": "string", "description": "Relative file path"}},
        "required": ["path"],
    },
}]

def read_file(path: str) -> str:
    try:
        with open(path, encoding="utf-8") as f:
            return f.read()[:20_000]
    except OSError as e:
        # Errors are observations, not exceptions: the model sees them and adapts.
        return f"ERROR: {e}"

messages = [{"role": "user", "content": "Summarize README.md in 3 bullets."}]

MAX_TURNS = 8                      # guard: the model never controls this
for _ in range(MAX_TURNS):
    resp = client.messages.create(
        model="claude-sonnet-5",
        max_tokens=2048,
        system="You are a careful engineering assistant. Use tools when needed.",
        tools=TOOLS,
        messages=messages,
    )
    messages.append({"role": "assistant", "content": resp.content})
    if resp.stop_reason != "tool_use":
        break                      # the model chose to stop — the loop's only soft exit
    results = []
    for block in resp.content:
        if block.type == "tool_use":
            output = read_file(**block.input)
            results.append({"type": "tool_result",
                            "tool_use_id": block.id,
                            "content": output})
    messages.append({"role": "user", "content": results})

print(resp.content[-1].text)
// npm install @anthropic-ai/claude-agent-sdk
import { query } from "@anthropic-ai/claude-agent-sdk";

// The SDK ships the loop from FIG 0.1 as a production harness:
// context assembly, a built-in tool belt (Read, Bash, Grep, ...),
// permission gating, and session persistence are already there.
for await (const message of query({
  prompt: "Summarize README.md in 3 bullets.",
  options: {
    allowedTools: ["Read"],   // constrain the action space (see §0.2)
    maxTurns: 8,              // same guard as the Python loop
  },
})) {
  if (message.type === "result") console.log(message.result);
}
// Spring AI — the framework runs the tool loop internally.
@Component
class FileTools {
    @Tool(description = "Read a UTF-8 text file and return its contents")
    String readFile(String path) throws IOException {
        return Files.readString(Path.of(path));
    }
}

// The .tools(...) call registers the schema; ChatClient loops on
// tool_use responses until the model produces a final answer.
String answer = ChatClient.create(chatModel)
        .prompt("Summarize README.md in 3 bullets.")
        .tools(new FileTools())
        .call()
        .content();

Read the Python version closely, because three of its lines carry most of the discipline: the MAX_TURNS guard (the model must never own its own termination), the ERROR: return inside the tool (tool failures are observations the model can react to, not exceptions that kill the loop), and the truncation at 20k characters (every tool output is a context-budget decision — Module 4).

Failure modes — the agent loop
  1. Unbounded looping / goal drift. The model keeps acting without converging — re-reading files, retrying variations, "one more check." Mitigation: hard turn caps, token budgets, and a progress heuristic (if the last N observations are near-duplicates, stop). Never rely on the model to notice it's stuck.
  2. Error spirals. A failing tool call gets retried with the same arguments forever. Mitigation: return instructive error messages (an error message is a prompt — "file not found; directory listing: […]" beats "ENOENT"), and cap identical retries in the executor.
  3. Context flooding. A tool returns 400KB of JSON and every subsequent inference pays for it — cost balloons and reasoning quality degrades (context rot, Module 4). Mitigation: truncate, paginate, and summarize at the tool boundary.
  4. Premature stop. The model answers confidently without calling any tool — plausible text, zero grounding. Mitigation: instructions alone are weak; verify outputs against the trajectory (did it actually read the file it's citing?) — Module 11.
  5. Silent misuse. Arguments pass the JSON schema but are semantically wrong (path: "/etc/passwd"). Schema validation is necessary, not sufficient — pair it with allowlists and permission gates (Modules 9, 12).
Principal's Judgment Call — who owns the loop?

The loop is 40 lines; nobody needs a framework for it. The real question is who owns the harness around the loop — permissions, sandboxing, tracing, retries, session resume, cost caps. That's thousands of lines of undifferentiated engineering.

My rule: if the agent is your product (you're selling the agent experience), own the loop and build the harness deliberately — you'll need control over every layer. If the agent augments your product or your team's workflow, adopt a harness (Claude Agent SDK, or Claude Code itself for internal work) and spend your innovation budget on tools, context, and evals instead. Teams that hand-roll harnesses for internal agents are usually rebuilding permission gating badly, six months at a time.

§ 0.2 Workflows vs. agents: the taxonomy that prevents overbuilding

Anthropic's "Building Effective Agents" essay drew the industry's most useful line: workflows are systems where your code decides the path and the model fills in steps; agents are systems where the model decides the path. Between "one prompt" and "full agent" sit five workflow patterns that solve most real problems at a fraction of the cost and risk. Senior engineers overbuild here more than anywhere else.

PatternPath decided byUse whenCanonical failure
Prompt chainingCode (fixed sequence)Task decomposes cleanly into ordered steps; add gate checks between themError propagation — a bad step 1 poisons steps 2–N; latency stacks linearly
RoutingCode (after a classify step)Distinct input categories that deserve specialized handlingAmbiguous inputs bounce between routes; categories drift over time — always keep a fallback route to a human or general handler
ParallelizationCode (fan-out / fan-in)Independent subtasks, or when N diverse opinions beat one (voting)Aggregation is the hard part everyone underestimates; N× cost for marginal gain if lenses aren't genuinely diverse
Orchestrator-workersModel decides the subtasks, code runs the delegationSubtask structure is unknowable upfront (research, multi-file changes)Orchestrator context bloat; workers duplicate effort or lack context — handoff design is the whole game (Module 3)
Evaluator-optimizerCode (generate → critique loop)Quality is verifiable against a rubric and iteration measurably improves itSycophantic convergence — the evaluator starts approving everything; give it teeth (deterministic checks) and cap iterations
AgentModel, at runtimeThe path genuinely can't be enumerated — open-ended debugging, exploration, multi-step tasks over unknown terrainAll of §0.1's failure modes, times autonomy
Principal's Judgment Call — default to workflows

The engineering instinct that "more general = better architecture" is inverted here. A workflow is testable, traceable, and boring in exactly the ways production systems should be boring. An agent buys you runtime flexibility and charges you determinism.

My default: start with the simplest pattern that could clear the quality bar, and only escalate on eval evidence — "the workflow version scores 71%, the agent version scores 89% on the same golden set" is an argument; "agents are the future" is not. In review, when someone proposes an agent, ask them to sketch the workflow version first. If they can enumerate the paths on a whiteboard, it's a workflow. The corollary also holds: if you find yourself adding the ninth conditional branch to a workflow, the paths have stopped being enumerable — let the model drive.

§ 0.3 The autonomy spectrum: matching trust to evidence

"Should we build an agent?" is the wrong question. The right one: how much of the control loop are we ready to hand over, given the blast radius when it's wrong? Autonomy is a dial, not a switch, and the spectrum has four useful stops.

StopHuman's roleBlast radius of one bad actionWhat must be true firstExample
CopilotInitiates and reviews everything≈ zero — nothing ships unreviewedOutput quality clears "useful more often than annoying"Code completion; draft-an-email
WorkflowDesigns the path; monitors aggregatesOne step's error, bounded by gates between stepsGolden-set evals per step; error handling at every gateTicket triage; document extraction pipeline
Semi-autonomousSets the task; approves irreversible actionsBounded by sandbox + approval gatesSandboxing, permission model, action allowlists, trajectory loggingCoding agent in a repo (Claude Code's default posture)
AutonomousSets goals; audits outcomesWhatever its credentials can reach — design assuming a bad dayEval maturity + reversibility + monitoring + incident playbooks. Earned, not declaredScheduled research/maintenance agents with scoped credentials

Two principles fall out of the table. First, autonomy is earned by evidence, not enabled by a flag — each step right should be justified by eval scores and operational history at the previous step, the way you'd earn the right to deploy on Fridays. Second, reversibility is the cheapest safety property you can buy: an agent that operates on branches, drafts, staging environments, and soft-deletes can be trusted with far more autonomy than one that touches production state, at the same model quality. Architect the substrate to be reversible and the autonomy debate gets dramatically easier (Module 12 formalizes this).

§ 0.4 The bitter lesson, applied to agents

Rich Sutton's "bitter lesson" from seventy years of AI research: general methods that leverage more compute eventually beat systems built on hand-encoded human knowledge — and the hand-encoded knowledge, which felt so clever when added, becomes the ceiling. The agentic translation: every piece of scaffolding you write is a bet against the next model release.

Concretely, scaffolding means: hard-coded task decompositions, rigid state machines wrapping the model, elaborate prompt contortions to force behavior, regex post-processors that "fix" model output, and multi-agent topologies invented to work around one model's weakness. Teams built all of these in 2023–2024 to make weak models usable, and much of it turned to dead weight — or an active handicap — as models improved. The 2026 version of the mistake is the same, just fancier.

But "no scaffolding" is cargo-cult in the other direction. The resolution is to sort every piece of structure into one of two buckets:

  • Risk controls — typed output boundaries, schema validation, permission gates, sandboxes, evals, audit logs. These manage the consequences of a probabilistic component in a deterministic system. They do not depreciate when models improve; a smarter model still shouldn't be able to drop your production tables.
  • Capability workarounds — manual decomposition because the model can't plan; forced re-prompting loops because it misses instructions; output post-processing because it can't format. These encode today's model weaknesses and depreciate with every release. They're sometimes necessary — but they're technical debt with a known creditor.
Principal's Judgment Call — design for deletion

I ask teams to tag every scaffold in the architecture doc as RISK-CONTROL or WORKAROUND, with a one-line justification. Risk controls get maintained like any security infrastructure. Workarounds get an expiry review: on every major model release, re-run the eval suite with the workaround disabled. If the naked model now clears the bar, delete the scaffold — and treat the deletion as a win in the changelog, not a refactor to defer.

The failure mode this prevents is real and expensive: a 2024-era pipeline of seven prompts, three regex fixers, and a voting layer that today underperforms a single well-contexted model call — but nobody dares touch it because it "works." Scaffolding you can't delete is architecture you don't understand.

§ 0.5 Lab — build loopbot, a no-framework agent

LAB 00 · Estimated 2–4 hours

Mission: build a file-question-answering agent over any real repository, using only a model provider's SDK — no agent framework. You will write the loop from FIG 0.1 and hit every guard and failure mode in §0.1 on purpose.

Spec:

  1. Two tools: list_files(directory) and read_file(path), both returning errors as string observations (never exceptions), both truncating output to a byte budget you choose.
  2. Guards: a max-turn cap, a running token/cost counter printed per turn, and a duplicate-call detector (same tool + same args twice in a row → inject a "you already tried this" observation).
  3. Transcript: log every message — including tool results — to a JSONL file. This is your primitive trace (Module 11 upgrades it).

Acceptance criteria:

  • ✅ Answers "which file defines <some function> and what does it do?" correctly on a repo of ≥ 50 files, citing the actual path.
  • ✅ Given a question about a nonexistent file, recovers gracefully (lists the directory, adapts) instead of crashing or spiraling — show the transcript that proves it.
  • ✅ Never exceeds the turn cap; prints total cost at exit.
  • ✅ Run the same task with a frontier model and a small/cheap model; write five sentences on the behavioral difference you observed (this intuition is Module 13's cost-routing groundwork).

Stretch: add an evaluator-optimizer pass — a second model call that grades the answer against the cited file's actual content and triggers one retry on failure. Notice how quickly "just add a critic" doubles your cost. That tension is Module 11.

§ 0.6 Consultant's checklist

Questions to ask any team claiming they "have agents"
  • For each capability: where does control flow live — code or model? Can they answer per-feature, or only with a marketing diagram?
  • What's the simplest taxonomy pattern that would clear the quality bar, and was it tried and measured before escalating to an agent?
  • What are the stop conditions on every loop? Who enforces turn caps and budgets — the model (bad) or the harness (good)?
  • What is the blast radius of the worst single tool call the system can make today? Is that action reversible, gated, or merely "unlikely"?
  • Which scaffolds are risk controls and which are model workarounds? When were the workarounds last re-validated against a current model?
  • What eval evidence would justify moving one step right on the autonomy spectrum — and does that evidence exist, or just enthusiasm?
  • What does one completed task cost (p50 and p99), and does anyone look at that number weekly?

§ 0.7 Design-review questions (staff+)

Interview / design-review bank
  1. You're asked to build an AI code-review assistant. Which taxonomy pattern do you reach for first, and what evidence would push you toward a full agent? Strong answers start at workflow (chaining or parallel lenses with an aggregator), name the gate checks, and demand eval deltas before granting the model control flow.
  2. Your agent's task success rate dropped after upgrading to a newer, stronger model. Diagnose. Strong answers suspect capability workarounds tuned to the old model's behavior (prompt contortions, decomposition, post-processors), reach for trajectory diffs on the golden set, and A/B the scaffolds off.
  3. When is a 200-line while-loop the right architecture versus a graph framework? Strong answers frame it as who-owns-the-harness (§0.1 sidebar) plus control-flow complexity, team skill distribution, human-in-the-loop needs, and observability requirements — not tribal framework preference.
  4. Argue both sides: "multi-agent systems are premature for most organizations." Strong answers cite coordination tax and error compounding (Module 3 preview) against context-isolation wins, and land on a conditional position tied to eval maturity.
  5. An agent needs write access to a production database. Where do you place the human approval gate, and why there? Strong answers put a typed proposal boundary in front of execution (agent proposes structured operations; a deterministic engine validates against an allowlist; irreversible categories require approval) rather than "review the SQL string."
