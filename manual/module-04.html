FM-04 · Architecture

Context Engineering

Budgeting the window across in-context, retrieved, and computed sources; compaction and tiered external memory with TTLs for sessions that outlive any one window; and how to diagnose poisoning, distraction, and context rot before they collapse a long-running agent.

PUBLISHED 2026-07-13 DEPTH ~5,000 words PREREQS Module 02 LAB give loopbot working memory + compaction; prove 100-turn survival

§ 4.0 Orientation

An on-call remediation agent was six hours into a database failover incident when a human reviewer noticed something wrong: at turn 118 it proposed restarting prod-web — a service it had already restarted twice before, each time justifying the action with "a deploy is in progress; restart is safe once it completes." That justification was true at turn 4. It had been false since turn 40, when a human posted in the incident channel — ingested by the agent as an ordinary tool result — that the deploy was frozen pending a rollback review and prod-web must not be touched. The freeze notice was message 6 of what had, by turn 118, become an 830,000-token transcript: every kubectl get pods, every log grep, every Slack reply, all still resident because nothing had ever left. The constraint that should have overridden the model's guess was technically still "in context" — sitting in the middle of a haystack the model could no longer reliably search. It hadn't forgotten the freeze notice so much as lost the ability to retrieve it over the noise of everything that came after, and it filled the gap with the most recent plausible pattern instead: restart the thing, that's what you do when a pod looks unhealthy.

Module 00's turn caps would not have stopped this — the agent wasn't looping, it was steadily doing new, individually-reasonable-looking work. Module 02's typed proposal boundary would have stopped the actual restart at a policy gate, and that control is still necessary — nothing in this module replaces it. But this module is about the layer beneath both of those: what goes into the context window in the first place, what gets deliberately thrown away as a session runs long, and what gets moved somewhere a lookup can find reliably instead of leaving it to a 830K-token scan. Get this layer right and the freeze notice survives to turn 118 as a first-class, retrievable fact. Get it wrong and every other control in this manual spends its budget defending against a problem context engineering should have prevented upstream.

Three disciplines, escalating in structure the way Module 02's did: budget what enters the window at all (§4.1), compact what's accumulated before it rots (§4.2), and route what must outlive any single window into tiered external memory (§4.3). §4.5 names the failure modes precisely enough to find them in a trace, and the lab proves the fix survives 100 turns — not a demo.

§ 4.1 The context window as a budget, not a container

Every token an agent sees on a given turn comes from exactly three sources, and conflating them is the root of most context-engineering mistakes. In-context is whatever's literally sitting in the messages array right now — the system prompt, the conversation so far, every tool result that's already been returned. Retrieved is anything fetched on demand, present only because something in the current turn asked for it — a RAG chunk, a memory-tool file read, a database row. Computed is a derived artifact that stands in for something larger — a summary, an extracted fact, a structured note — cheaper to hold than the source material and lossy by construction. A well-engineered agent treats all three as draws against one shared budget, not as three separate pools; the question for every piece of information is never "can it fit" but "does keeping it here, verbatim, earn its place over a pointer to it."

Every one of those numbers will be stale within a quarter — treat the table as an illustration of order-of-magnitude, not a spec sheet, and verify against the vendor's current docs before citing a figure in a proposal (AS-OF note, §4.10). The number that matters more than any of them is qualitative: windows got bigger; the reason to manage them didn't go away, because the failure mode it prevents moved. In the 8K–32K era, the dominant failure was a hard error — the request simply didn't fit, and you found out immediately. At 200K–2M tokens, the dominant failure is soft: Liu et al.'s "Lost in the Middle" found that retrieval accuracy over long contexts follows a U-shape — highest when the needed fact sits at the very start or the very end of the input, and measurably worse when it sits in the middle, even on models explicitly built for long contexts. Later replications sharpen this further: the U-shape mostly holds while the window is under roughly half full; past that point, the pattern shifts toward straightforward recency bias — the model increasingly favors whatever is closest to the end, regardless of where the important fact actually sits. Either way, the practical upshot is the same one the incident-response agent lived through: a fact doesn't have to be truncated to become unreachable. It just has to be far enough from the places the model's attention actually searches.

The engineering consequence: default every piece of information to the cheapest tier that still does the job. Tool output the model only needed to confirm success can be truncated to a status line the moment it's read (computed). A document the agent might reference again belongs behind a retrieval call, not inlined in full (retrieved). Only facts the agent must have available on every subsequent turn, without a lookup, earn a permanent seat in-context — and even those should be as compact as the task allows. §4.2 and §4.3 are the two mechanisms that enforce this discipline once a session runs long enough that it stops being optional.

§ 4.2 Compaction: summarize and continue, before rot sets in

Compaction is the discipline of periodically converting the older part of a long-running conversation into a smaller, computed stand-in — a summary — while keeping the most recent turns verbatim and carrying forward a small set of facts that must survive untouched. It is not a last resort for when the window is about to overflow; per §4.1, the trigger for compaction should fire well before the hard limit, at the point where retrieval quality has started to erode, because that point arrives earlier than most teams assume.

Anthropic ships two vendor-managed primitives for exactly this problem: context editing (beta context-management-2025-06-27), which clears old tool-use/tool-result pairs once a token threshold is crossed and replaces them with a placeholder so the model knows something was removed; and compaction (beta compact-2026-01-12), which summarizes the whole conversation server-side as it nears the window limit. Anthropic reports roughly 29% token savings from context editing alone and 39% from context editing plus the memory tool (§4.3) over an unmanaged baseline, and describes cutting token consumption by roughly 84% on a hundred-turn web-search task while letting the run complete instead of failing on context exhaustion. Treat those figures as vendor-reported and directional — verify current numbers before quoting them in a proposal (AS-OF, §4.10).

Layer 1 — agent runtime: a typed compacted-context object (Python)

The pinned-facts list is the load-bearing part — it's exactly Module 01 §1.5's typed output boundary applied to compaction: the summary is prose and allowed to be lossy, but anything a downstream turn will treat as settled must be schema-validated, not trusted freehand.

# pip install anthropic pydantic
from pydantic import BaseModel, Field
import anthropic

client = anthropic.Anthropic()

class PinnedFact(BaseModel):
    key: str
    value: str
    established_turn: int
    evidence_ref: str          # what tool result or message proved this — never a bare assertion

class CompactedContext(BaseModel):
    session_id: str
    summary: str                       # lossy prose — never load-bearing on its own
    pinned_facts: list[PinnedFact]      # must survive every compaction verbatim
    last_k_turns: list[dict]
    compacted_at_turn: int
    tokens_before: int
    tokens_after: int

COMPACT_THRESHOLD_TOKENS = 60_000   # trip well below the model's window — rot, not overflow, is the target
KEEP_LAST_TURNS = 8                  # recent turns need no summarization; keep them exact

def estimate_tokens(messages: list[dict]) -> int:
    return sum(len(str(m)) for m in messages) // 4   # swap for a real tokenizer/count endpoint in production

def compact(session_id: str, messages: list[dict],
            pinned: list[PinnedFact], turn: int) -> CompactedContext | None:
    tokens_before = estimate_tokens(messages)
    if tokens_before < COMPACT_THRESHOLD_TOKENS:
        return None                     # not due yet — most turns take this path
    head, tail = messages[:-KEEP_LAST_TURNS], messages[-KEEP_LAST_TURNS:]
    pinned_block = "\n".join(f"- {p.key}: {p.value} (turn {p.established_turn})" for p in pinned)
    resp = client.messages.create(
        model="claude-sonnet-5", max_tokens=1024,
        system=("Summarize this agent transcript for a fresh continuation. Preserve every "
                 "constraint, decision, and open question. The pinned facts below are ALREADY "
                 "guaranteed to survive — do not restate them, and do not contradict them."),
        messages=[{"role": "user", "content": f"Pinned facts:\n{pinned_block}\n\nTranscript:\n{head}"}],
    )
    summary = resp.content[-1].text
    return CompactedContext(
        session_id=session_id, summary=summary, pinned_facts=pinned, last_k_turns=tail,
        compacted_at_turn=turn, tokens_before=tokens_before,
        tokens_after=estimate_tokens(tail) + len(summary) // 4,
    )
// npm install @anthropic-ai/sdk
import Anthropic from "@anthropic-ai/sdk";
const client = new Anthropic();

// Server-side compaction + context editing: the vendor runs the summarizer and
// the eviction policy; you configure thresholds and what to keep verbatim.
// Same shape as the Python version, without owning the summarizer call yourself.
const resp = await client.beta.messages.create({
  model: "claude-sonnet-5",
  max_tokens: 2048,
  betas: ["compact-2026-01-12", "context-management-2025-06-27"],
  context_management: {
    edits: [{
      type: "clear_tool_uses_20250919",
      trigger: { type: "input_tokens", value: 60000 },   // same threshold logic as COMPACT_THRESHOLD_TOKENS
      keep: { type: "tool_uses", value: 8 },              // keep the last 8 tool turns verbatim
    }],
  },
  messages,
});
// Beta surface — header names and default thresholds are exactly the kind of
// thing to re-verify against current docs before shipping (§4.10 AS-OF note).
// Spring AI's built-in ChatMemory implementations are windowed by default:
// MessageWindowChatMemory drops the oldest messages once a size cap is hit —
// it does NOT summarize or pin anything. That's a materially different failure
// profile from the Python/TypeScript versions above: windowing loses the
// freeze notice permanently and silently, with no summary trace left behind.
// A compacting memory needs its own implementation:
@Component
class CompactingChatMemory implements ChatMemory {
    private static final int THRESHOLD_TOKENS = 60_000;
    private final Map> store = new ConcurrentHashMap<>();
    private final Compactor compactor;   // wraps a summarizer call + pinned-facts schema check

    @Override
    public void add(String conversationId, List messages) {
        var existing = store.computeIfAbsent(conversationId, k -> new ArrayList<>());
        if (estimateTokens(existing) > THRESHOLD_TOKENS) {
            existing = compactor.summarizeAndPin(existing);   // same shape as compact() above
        }
        existing.addAll(messages);
        store.put(conversationId, existing);
    }
}
Principal's Judgment Call — compact this agent, or isolate the next chunk of work?

Reach for compaction when the task is one continuous identity that must remember what came before — a single incident, one customer conversation, one agent's ongoing shift. Reach for Module 03's context isolation (a fresh subagent, a clean handoff) when the next unit of work doesn't actually need the accumulated history at all — a genuinely independent subtask is cheaper to hand a typed brief than to compact 800K tokens down to fit.

The tell: isolation forfeits continuity in exchange for a clean slate. If the honest answer to "does the next step need to remember turn 4's freeze notice" is yes, you cannot solve that with a fresh subagent unless you hand it that fact explicitly in the brief — which is compaction wearing a different hat, not an alternative to it. Isolation and compaction aren't competing solutions to the same problem; they answer different questions ("does this need to be one continuous session?" vs. "how do I keep one continuous session small?").

§ 4.3 Memory tiers: conversation, scratchpad, episodic, semantic, procedural

Compaction manages one session's window. Some facts need to outlive the session entirely — a customer's stated preference, a runbook learned from a past incident, a standing organizational policy. These belong in external memory, and treating "memory" as one undifferentiated bucket is how teams end up either re-deriving the same facts every session (nothing persisted) or drowning every session in irrelevant history (everything persisted, nothing scoped). Five tiers, ordered by how long a fact should live and how it should be looked up, cover almost every real case:

TierLifetimeStorageExample
ConversationThe live session onlyThe messages array itself — never written to an external storeThe last K raw turns after compaction (§4.2)
ScratchpadOne task/sessionExternal KV, short-medium TTL"Step 3 of 7 done, waiting on the migration job"
EpisodicWeeks to monthsKV or document store, longer TTL, often similarity-indexed"Last time this alert fired, restarting didn't help — clearing the connection pool did"
SemanticLong-lived, revalidated not clock-expiredKV/DB, versioned, long or no TTL"prod-web deploys freeze every Friday after 3pm"
ProceduralCurated, changes rarelyFiles/skills — authored, not auto-expiredA runbook: the db-failover checklist

This maps closely onto Packer et al.'s MemGPT, which frames an LLM's context the way an OS frames RAM: a small, fast working set (main context) backed by unlimited, slower archival storage, with explicit function calls paging facts between the two. Letta, MemGPT's production successor, collapses this into a three-tier core/archival/recall model that maps cleanly onto the five tiers above (core ≈ conversation + pinned facts, archival ≈ semantic + procedural, recall ≈ episodic). Anthropic's own memory tool is a production instance of the same idea at the API layer: a client-side /memories directory Claude reads and writes via view/create/str_replace/insert/delete/rename commands, with just-in-time retrieval as the default posture — Claude checks the directory listing before reading any file in full, rather than front-loading everything. Anthropic's stated pattern for long-running agents is to run both mechanisms together: compaction keeps the active window small without client-side bookkeeping, while memory preserves exactly the facts that must survive a compaction event untouched — which is precisely why §4.2's pinned_facts and this section's tiers are the same discipline applied at two different lifetimes.

Layer 2 — data/infra: a Redis-backed tiered store with TTLs (Python)

# pip install redis
import redis, json

r = redis.Redis(host="localhost", port=6379, decode_responses=True)

TIER_TTL_SECONDS = {
    # conversation lives only in-context — never written here, see §4.2/§4.3 table
    "scratchpad": 60 * 60 * 4,            # 4h — dies with the session unless promoted
    "episodic":   60 * 60 * 24 * 30,      # 30 days — one task's outcome, for similar-task recall
    "semantic":   None,                    # long-lived; revalidated against source, not clock-expired
    "procedural": None,                    # curated playbooks — no auto-expiry
}

def remember(session_id: str, tier: str, key: str, value: str) -> None:
    redis_key = f"agent:{session_id}:{tier}:{key}"
    r.set(redis_key, value)
    ttl = TIER_TTL_SECONDS[tier]
    if ttl:
        r.expire(redis_key, ttl)

def recall(session_id: str, tier: str, key: str) -> str | None:
    return r.get(f"agent:{session_id}:{tier}:{key}")

def recall_tier(session_id: str, tier: str) -> dict[str, str]:
    keys = r.keys(f"agent:{session_id}:{tier}:*")
    return {k.rsplit(":", 1)[-1]: r.get(k) for k in keys}

# The freeze notice from §4.0, written the moment it's confirmed (evidence_ref
# points at the Slack message tool result that proved it — see PinnedFact above):
remember("incident-4471", "semantic", "deploy_freeze",
         json.dumps({"value": "prod-web deploy frozen since turn 4 pending rollback "
                                "approval — do not restart prod-web",
                     "evidence_ref": "slack:C0123/p1720000000000"}))
Principal's Judgment Call — what earns a place in memory, and what's its TTL?

Pin or store a fact only when it's both expensive/risky to reconstruct and its absence would cause an unsafe or costly decision — a deploy freeze, a customer's stated hard constraint, a runbook step learned the hard way. Everything else should fall out of context and be regenerated on demand; a memory system that hoards everything is context rot with extra infrastructure, plus its own staleness bugs.

Set the TTL from the fact's real-world half-life, not from storage convenience. A deploy freeze doesn't expire on a clock — it expires when the freeze lifts, so it belongs in the semantic tier with a revalidation check against the live source before any high-stakes action, not a fixed TTL that might silently outlive or under-live the actual freeze window. Reserve clock-based TTLs (scratchpad, episodic) for facts whose relevance genuinely does decay with time regardless of any external event.

§ 4.4 Full-stack: the same context object across runtime, data, and API

The pattern from Module 01 §1.2 and Module 03 §3.4 generalizes here: whatever object the runtime produces should be the same object the API serves and a human (or curl) can inspect — not re-derived per layer. A thin FastAPI layer over the Redis store from §4.3 turns "what does this agent remember" from a debugging mystery into a one-line command, and gives the lab (§4.7) something observable that isn't just a log line.

Layer 3 — API/transport: exposing memory and pinned facts for inspection (Python, FastAPI)

# pip install fastapi uvicorn   ·   run: uvicorn memory_api:app --port 8000
from fastapi import FastAPI
import json

app = FastAPI()

@app.get("/agents/{session_id}/memory")
def get_memory(session_id: str):
    return {tier: recall_tier(session_id, tier) for tier in TIER_TTL_SECONDS}

@app.get("/agents/{session_id}/pinned")
def get_pinned(session_id: str):
    raw = recall(session_id, "semantic", "deploy_freeze")
    return json.loads(raw) if raw else None

§ 4.5 Diagnosing a degraded session: poisoning, distraction, and rot

Failure modes — context engineering
  1. Context poisoning. A hallucinated or injected claim gets stated once, and because it now reads as "the agent's own prior statement" in the transcript, later turns treat it as established fact and build on it — the error compounds instead of staying contained. Symptom: a fabricated detail (a file path, a metric, a customer name) appears once with no tool verification behind it, then gets cited as given in subsequent turns. Mitigation: never let compaction or a memory write promote a claim into pinned_facts or a semantic-tier entry without an evidence_ref pointing at the tool result that actually proved it — an assertion with no evidence is not eligible for permanence.
  2. Distraction. A long run accumulates many instances of the same class of action (repeated restarts, repeated similar tool calls), and their sheer repetition in the transcript out-weighs a single, more important instruction from earlier — the model pattern-matches to "what usually happens next" rather than reasoning about the specific constraint that should override it. Symptom: the agent keeps repeating an action whose frequency in the transcript has grown, even after being told to stop, and the ratio of stated reasoning to repeated action collapses. Mitigation: compact repeated-action clusters into one aggregated note ("attempted N times, all failed the same way") rather than letting age alone drive what gets summarized — repetition, not just staleness, is a compaction trigger.
  3. Compaction amnesia. The summarizer drops or under-weights a fact that wasn't flagged as pin-worthy at write time — the §4.0 failure exactly. Symptom: an established constraint stops being honored right after a compaction event; grepping the transcript shows the fact's first appearance and its last honored turn straddling a compaction boundary. Mitigation: make pinning an explicit, cheap action available every turn rather than something inferred after the fact by a summarizer working under token pressure, and require a pin check at any point with irreversible stakes — restarting a production service is exactly the kind of action that should force the check, echoing Module 02 §2.4's phase-gating discipline.
  4. TTL staleness. A semantic-tier fact gets served past the point it stopped being true because its TTL was set for storage convenience rather than the fact's real-world half-life. Symptom: the agent acts on a memory read that's contradicted by a live tool result it just fetched, and trusts the stale memory because it's phrased confidently or was written more recently than the underlying fact actually changed. Mitigation: attach a revalidation rule — not just a TTL — to any semantic fact with an external ground truth, and treat memory as a hint that earns trust from freshness, never a substitute for a live check before a high-stakes action.

§ 4.6 Progressive disclosure as the default posture

Every mitigation above shares one shape: load less by default, expand only when the agent's own behavior signals it needs more. The memory tool's just-in-time retrieval (§4.3) is the clearest instance — Claude checks the /memories directory listing before reading any file in full, rather than front-loading every file's contents on every turn. The same discipline applies to retrieval and computed context generally: default to a pointer or a one-line summary, and only expand to the full source when a follow-up question or a failed check indicates the summary wasn't enough. Skills (Module 08) apply exactly this pattern at the harness level — a short front-matter description loaded always, the full skill body loaded only when selected — and that module covers the authoring mechanics in depth; here the point is narrower: progressive disclosure is the reason a bigger window doesn't make context engineering optional. A 1M-token window without this discipline just means you can afford to stuff more low-relevance material in before hitting a hard error — you arrive at the soft-rot problem from §4.1 on a fuller, more expensive context, not a smaller one.

Principal's Judgment Call — adopt the vendor's compaction API, or hand-roll your own?

Adopt the vendor primitive (context editing, compaction) when you're already committed to that provider's stack and your summarization needs are generic — age-based, tool-result-heavy, no unusual pinning logic. You get server-side, cache-friendly compaction with no orchestration code to maintain, and the vendor's benchmark numbers (§4.2) are a reasonable prior for what to expect.

Hand-roll when you need domain-specific pinning (evidence-gated facts with an audit trail, as in this module's code), custom TTL logic across tiers, or the identical behavior across more than one model provider. The tell: if "what counts as pin-worthy" is a business rule your risk owner needs to approve — the same discipline Module 02 §2.2 applied to the auto-approve threshold — that logic belongs in code you own, with the vendor primitive handling the generic bulk-summarization underneath it if you want to layer them.

§ 4.7 Lab — give loopbot working memory and compaction; survive 100 turns

LAB 04 · Estimated 4–6 hours

Mission: extend Module 00's loopbot with §4.2's compaction manager and §4.3's Redis-backed memory tiers, then run it through a scripted 100-turn incident-investigation task and prove — via transcript and curl, not a passing unit test — that a fact established at turn 4 is still correctly honored at turn 100.

Spec:

  1. Build a synthetic 100-turn task harness that feeds loopbot a stream of tool outputs (logs, status checks) simulating an incident. At turn 4, inject a constraint ("service X is frozen, do not restart it") with a verifiable evidence_ref. Ensure the raw transcript crosses your COMPACT_THRESHOLD_TOKENS at least 3 times before turn 100.
  2. Wire in §4.2's compaction manager: pinned facts pass through every compaction verbatim; everything else gets summarized. Wire in §4.3's Redis store for the scratchpad/episodic/semantic tiers, and §4.4's FastAPI endpoints.
  3. Log every turn, every compaction event (before/after token counts), and every memory read/write to a JSONL transcript — this is your evidence, not a side effect.

Acceptance criteria:

  • ✅ curl -s localhost:8000/agents/incident-4471/pinned at turn 100 returns the turn-4 constraint verbatim, with its original evidence_ref unchanged.
  • ✅ The transcript shows at least 3 compaction events; for each, tokens_after is under 25% of tokens_before — the savings are measured, not asserted.
  • ✅ Run the identical 100-turn task a second time with pinning disabled (facts flow only through the lossy summarizer). Show, via a transcript diff, that the agent violates the constraint — attempts the forbidden restart — somewhere after the first compaction event that should have carried it forward. This is the side-by-side proof that pinning is load-bearing, not decorative.
  • ✅ A golden check at turn 100: grade the agent's final action against a rubric that specifically checks whether it recalled the freeze (cited the pinned fact) rather than re-derived it by re-reading the entire transcript. Write 5 sentences on whether quality held across the full 100 turns.

Stretch: at the end of the run, write an episodic-tier entry summarizing what resolved the incident. Start a second synthetic run for a similar incident and confirm loopbot retrieves that episodic note at turn 5 instead of re-deriving the same fix from scratch.

§ 4.8 Consultant's checklist

Questions to ask any team running long-lived or long-running agents
  • Where is the line between "in context" and "needs a lookup" drawn today — deliberately, or wherever the last engineer stopped adding tool results?
  • Has anyone measured this agent's own accuracy as a function of context fullness, or is "it has a 1M-token window" being treated as equivalent to "it's fine"?
  • What triggers compaction — a measured token threshold, or does the session just error out and restart from zero?
  • Which facts are pinned, who decided they were pin-worthy, and does that decision carry an evidence reference or just an assertion?
  • What's the TTL on every external memory write, and was it chosen from the fact's real-world half-life or copied from another tier's default?
  • Can you produce, right now, a curl command that shows what this agent "remembers" about a specific past decision? If not, the memory is unauditable — for anything high-stakes, that's equivalent to not having it.
  • When a summarizer call fails or times out mid-compaction, does the session lose everything, or degrade gracefully to "no compaction this round, retry next turn"?

§ 4.9 Design-review questions (staff+)

Interview / design-review bank
  1. A team upgrades to a model with a 5x larger context window and removes their existing summarization step to "let the model handle it." Accuracy silently drops. Diagnose. Strong answers reject "bigger window solves retrieval" outright, point to lost-in-the-middle-style degradation starting well before the window is full, and ask for accuracy-vs-context-fullness measurements on the team's own traces before accepting "the model can just hold it all" as an architecture decision.
  2. Design the memory layer for a long-running support agent that must remember a customer's stated preferences across sessions but must never act on stale billing information. Strong answers put preferences in the semantic tier with a long TTL, but give billing-adjacent facts a revalidation rule instead of a TTL — recheck the live source before any money-touching action — tying directly back to Module 02 §2.2: memory informs the proposal, live data gates execution.
  3. A teammate proposes fixing a 100-turn quality-collapse problem by switching to a fresh subagent every 20 turns instead of adding compaction. Steelman both, then decide. Strong answers concede isolation genuinely sidesteps rot (Module 03's argument), but note it forfeits continuity — the fresh subagent doesn't know about turn 4's constraint unless it's handed a brief containing it, which is compaction's job under another name. Land on: compaction for one continuous identity that must remember; fresh subagents when the next chunk of work doesn't need continuity at all — not a blanket either/or.
  4. Your compaction summarizer occasionally fabricates a detail that wasn't in the transcript it condensed. Diagnose and fix. Strong answers name this as context poisoning at the summarization boundary itself — the summarizer is a probabilistic step whose output later turns will trust as settled. Apply Module 01 §1.5's boundary discipline: validate any pinned-facts list's evidence_ref against the actual transcript before accepting it, and never let free-text summary content silently become an unverified fact a later turn cites as given.
  5. How do you decide whether a fact belongs in the system prompt (Module 02 §2.3), a pinned fact, or semantic memory? Strong answers use scope and lifetime as the test: system prompt facts hold for this agent regardless of session (role, non-negotiables); pinned facts hold for this specific task and are dangerous to lose (survive compaction, scoped to the session); semantic memory holds across sessions but should be looked up when relevant, not force-loaded everywhere. The tell: if hard-coding it into the system prompt would apply to sessions where it's irrelevant, it belongs in memory, not the prompt.
