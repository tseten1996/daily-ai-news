FM-04 · Architecture

Context Engineering

The window is a budget, not a warehouse: compaction and server-side context editing, four memory tiers with a real store behind them, the pathology of degraded sessions, and progressive disclosure.

PUBLISHED 2026-07-12 DEPTH ~5,200 words PREREQS Module 02 LAB loopbot survives a 100-turn session

§ 4.0 Orientation

Around turn 55 of a schema-migration session, the agent re-read a config file it had already read three times, "discovered" a constraint it had itself written down at turn 12, and then — worse — cited a table name that never existed, because a hallucinated name from turn 30 had been sitting in the transcript ever since, quietly promoted from guess to fact by repetition. Nothing crashed. Cost per turn kept climbing, answer quality kept sliding, and no log line said why. If you've operated a long-running agent, you've watched this: the model didn't get dumber, its context did.

Module 00 flagged tool-output truncation as a context-budget decision and Module 02 §2.3 showed how to layer the system prompt and inject dynamic context per turn. This module is where those threads become a discipline: treating everything inside the window as a budgeted, curated, perishable resource. Four moves cover the territory — budget the window (§4.1), edit and compact it as sessions run long (§4.2), move durable knowledge out of it into memory tiers (§4.3), and recognize the specific ways sessions degrade so you can diagnose them from a trace instead of vibes (§4.4). Progressive disclosure (§4.5) ties the moves together into the load-bearing default.

§ 4.1 The window is a budget, not a warehouse

Every token in the window pays rent twice: once in dollars (input tokens on every subsequent call — an agent that carries 150k tokens through 40 more turns bills those tokens 40 more times) and once in attention. Model quality degrades as input grows, and it starts degrading well before the advertised limit: Chroma's context-rot study measured all 18 frontier models it tested getting measurably worse on tasks as input length grew, even trivially simple tasks, long before hitting any hard cap. A 200k-token window is not 200k tokens of equally usable space; it's a small high-fidelity core plus a long tail of increasingly mushy recall. Treat vendor window sizes as an upper bound on what fits, never as a target for what to load.

Budgeting starts with knowing what the window actually holds. For a mid-session agent it's roughly: system prompt and tool definitions (fixed overhead — 20 tool schemas can quietly cost 10–15k tokens before anything happens), conversation turns, tool results (usually the dominant and fastest-growing segment), and whatever you injected — retrieved documents, memory, scratch state. If you can't print this breakdown per turn, you're not engineering context, you're hoping. The runtime code in §4.3 emits exactly this BudgetReport after every turn, and the transport layer streams it to a gauge you can watch.

Principal's Judgment Call — the 2M-token window doesn't repeal this module

Every window-size jump (as of this writing, 2M-token production windows are arriving — see the Trends Board) re-raises the same proposal: "just load everything and skip the retrieval machinery." Reach for the big window when the task is genuinely one-shot comprehension of a large static corpus — analyze this codebase, review this contract set — where curation cost exceeds the one-time token bill and nothing needs to persist across sessions. Stay with curation for anything iterative or long-running: an agent loop re-sends its context every turn, so a stuffed window multiplies cost per turn, slows every inference, and marinates in its own context rot.

The decision rule I use in reviews: multiply intended window fill by expected turns. A 1.5M-token context on a 3-turn analysis is a reasonable $10 of comprehension. The same context on a 60-turn agent is a four-figure bill for quality that Chroma's data says will be worse at turn 60 than a curated 40k context. Big windows are a capability; they are not an architecture.

§ 4.2 Compaction and context editing: surviving the long session

A long-running agent has exactly three options when the transcript approaches the budget: die (hit the cap mid-task), amputate (drop old turns wholesale and lose state), or compact — replace the bulk of the transcript with a dense summary and continue with the summary plus recent turns. Compaction is how harnesses like Claude Code survive multi-hour sessions, and as of early 2026 it's also a server-side API primitive, so you no longer have to hand-roll the summarize-and-reinitiate dance to get it.

Two distinct mechanisms, often confused, that compose:

  • Context editing (clearing) — surgically remove the bodies of old tool results (and optionally thinking blocks) while keeping the conversational skeleton. Cheap, lossy only where you choose, ideal for the "tool results dominate growth" problem in FIG 4.1. The model sees a placeholder where the 40k-token file dump used to be — and can re-fetch it if it genuinely needs it again.
  • Compaction (summarize-and-continue) — replace the transcript wholesale with a model-written summary when a token threshold trips, then keep looping on top of the summary. Heavier, touches everything, and its quality depends entirely on what the summarization is instructed to preserve.

Both mechanisms in one call, using the Claude API's server-side context management (the TypeScript track shows the harness view — the Claude Agent SDK ships this behavior as defaults you tune rather than code you write):

# pip install anthropic — betas as of 2026-07: verify current headers before shipping
import anthropic

client = anthropic.Anthropic()

resp = client.beta.messages.create(
    model="claude-sonnet-5",
    max_tokens=4096,
    betas=["context-management-2025-06-27", "compact-2026-01-12"],
    tools=TOOLS,
    messages=messages,
    context_management={
        "edits": [
            {   # fires repeatedly: strip old tool-result bodies, keep the skeleton
                "type": "clear_tool_uses_20250919",
                "trigger": {"type": "input_tokens", "value": 60_000},
                "keep": {"type": "tool_uses", "value": 5},
                "clear_at_least": {"type": "input_tokens", "value": 15_000},
                "exclude_tools": ["memory"],   # never clear the memory tool's results
            },
            {   # fires rarely: whole-transcript summary under an explicit contract
                "type": "compact_20260112",
                "trigger": {"type": "input_tokens", "value": 150_000},
                "instructions": (
                    "Summarize this session for an agent that will continue it. "
                    "Preserve verbatim: the goal and acceptance criteria; every "
                    "decision and why; every discovered constraint; paths of all "
                    "artifacts written; what is done vs. open; unresolved errors. "
                    "Compress everything else aggressively."
                ),
            },
        ]
    },
)

applied = getattr(resp, "context_management", None)
if applied:
    print("edits applied:", applied)   # cleared_tool_uses, cleared_input_tokens, ...

    // written to disk (NOTES.md, plan files) — the filesystem is the agent's
    // external memory, re-readable after any summary. §4.3 formalizes this.

  if (message.type === "result") console.log(message.result);

The single most important line in the Python example is the instructions string. A default summary optimizes for "reads like a good summary." Your agent needs it to optimize for resumability — which is why the preservation contract names categories (decisions, constraints, artifact paths, open items) instead of trusting the summarizer's taste. The second most important line is exclude_tools: ["memory"]: §4.3's memory tool is only useful if editing doesn't eat its outputs. And the cheapest insurance of all sits outside the API: write durable state to disk before it's at risk. An agent that keeps a NOTES.md and a plan file loses almost nothing to compaction, because the summary only has to remember that the files exist, not their contents.

Failure modes — editing & compaction
  1. Compaction amnesia. The one constraint that mattered ("staging uses schema v2, prod is still v1") gets averaged into "worked on schema versions." Symptom in the trace: immediately after a compaction block, the agent re-asks or re-derives something it demonstrably knew, or violates a constraint it had been respecting for 40 turns. Mitigation: preservation-contract instructions + durable notes on disk + an eval that runs tasks across a forced compaction boundary (Lab, step 4).
  2. Summarizing the poison. Compaction laminates errors: a hallucinated "fact" in the transcript gets restated in the summary, now stripped of the shaky phrasing that might have let the model doubt it. Symptom: a wrong claim that appears with hedges early in a session reappears post-compaction as flat assertion. Mitigation: ground summaries in verifiable artifacts (file paths, IDs the runtime can check) and keep provenance — "per tool result X" — in the contract.
  3. Clearing what the model still needed. Aggressive clear_tool_uses drops a file body the model was mid-way through editing; it re-fetches (paying the turn back) or, worse, edits from memory of the file. Symptom: re-reads of recently cleared items, or diffs that don't match the file on disk. Mitigation: raise keep, tune the trigger to fire well before pressure is critical, and exclude tools whose outputs are session-critical.
  4. Cache-invalidation surprise. Editing mutates the prefix, so prompt-cache hits drop and per-turn cost can rise right after an edit fires even though the context shrank. Symptom: cost-per-turn sawtooth aligned with applied_edits events. Mitigation: fire edits in larger, rarer batches (clear_at_least) instead of tiny frequent trims; watch cache-hit rate next to token counts on the same dashboard (Module 11).
Principal's Judgment Call — clear first, compact late

Reach for tool-result clearing as your first and default lever: it targets the segment that actually grows (FIG 4.1), it's reversible in the only sense that matters (the model can re-fetch), and its blast radius is one stale payload. Reach for compaction only when the conversational history itself is the pressure — sessions long enough that even a skeleton transcript is heavy — and treat every compaction event as a small operational incident: log it, count it, eval across it.

If your agent compacts more than once or twice per task, that's not a tuning problem — the task is decomposing wrong. Split it into sessions with explicit artifact handoffs (Module 03 §3.3's brief/finding discipline applies within a single agent's lifetime too), or move state out of the transcript into memory (§4.3). Compaction is a pressure-relief valve, not a lifestyle.

§ 4.3 Memory tiers: moving knowledge out of the window

Everything in §4.2 buys headroom inside one session. Memory is the complementary move: put durable knowledge outside the window, in stores the agent reads and writes through tools, so it survives compaction, session restarts, and process crashes. The taxonomy that's become standard sorts memory by what it remembers, and it matters because each tier wants a different store, a different retrieval trigger, and a different forgetting policy:

TierRemembersLives inEnters contextForgetting policy
WorkingThe current task: plan, recent turns, scratch stateThe window itself + scratch filesAlways (it is the context)Compaction / session end
EpisodicWhat happened: past sessions, outcomes, transcriptsJSONL transcripts, session tableRetrieved when a similar task recurs ("have I done this before?")TTL by age; keep outcomes, drop play-by-play
SemanticFacts and preferences: "prefers staging deploys," "the API rate limit is 50/s"Key-value / vector store (Postgres + pgvector below)Small core pinned in-context; the rest retrieved by similaritySupersede-on-conflict + last-accessed decay
ProceduralHow to behave: learned rules, corrections, "always run the linter before committing"Instruction files (NOTES.md, CLAUDE.md-style) the agent editsIn-context, folded into the system layer (Module 02 §2.3)Human-reviewed edits; version-controlled

The mechanism connecting the tiers to the loop is deliberately boring: memory is just tools. The Claude API's memory tool (memory_20250818) is the cleanest reference shape — the model gets view / create / str_replace / insert / delete / rename commands against a /memories directory, and your code implements the backend, which is exactly the point: the same tool surface can sit on tmpfiles for a prototype and Postgres for production. Anthropic's published numbers for the combination — context editing plus memory tool — are a 39% improvement over baseline on their agentic eval, and an 84% token reduction on a 100-turn task that otherwise exhausts the window (as-of figures; re-verify before quoting in a design doc). The framework ecosystem (LangMem, Mem0, Letta, Zep) packages these tiers with managed extraction and retrieval; the judgment call below is whether you need them.

Layer 1 — agent runtime: the memory-tool handler and the instrumented loop (Python)

# pip install anthropic psycopg[binary]
from dataclasses import dataclass, asdict
from pathlib import Path
import anthropic, json

MEMORY_TOOL = {"type": "memory_20250818", "name": "memory"}

class FileMemoryBackend:
    """Prototype backend: /memories on disk. Same interface as the
    Postgres backend in Layer 2 — swap without touching the loop."""
    def __init__(self, root: str = "./memories"):
        self.root = Path(root); self.root.mkdir(exist_ok=True)

    def handle(self, cmd: dict) -> str:
        p = lambda rel: (self.root / rel.removeprefix("/memories").lstrip("/"))
        try:
            match cmd["command"]:
                case "view":
                    t = p(cmd.get("path", "/memories"))
                    return t.read_text() if t.is_file() else "\n".join(
                        sorted(x.name for x in t.iterdir())) or "(empty)"
                case "create":
                    p(cmd["path"]).write_text(cmd["file_text"]); return "ok"
                case "str_replace":
                    f = p(cmd["path"]); body = f.read_text()
                    if body.count(cmd["old_str"]) != 1:
                        return "ERROR: old_str must match exactly once"
                    f.write_text(body.replace(cmd["old_str"], cmd["new_str"])); return "ok"
                case "delete":
                    p(cmd["path"]).unlink(); return "ok"
                case _:
                    return f"ERROR: unsupported command {cmd['command']!r}"
        except OSError as e:
            return f"ERROR: {e}"          # observations, not exceptions (M00 rule)

@dataclass
class BudgetReport:                        # the object that crosses every layer
    turn: int; input_tokens: int; output_tokens: int
    cache_read_tokens: int; edits_applied: int

def run_session(task: str, memory: FileMemoryBackend, on_budget) -> str:
    client = anthropic.Anthropic()
    messages = [{"role": "user", "content": task}]
    for turn in range(100):                # the Lab's 100-turn bar
        resp = client.beta.messages.create(
            model="claude-sonnet-5", max_tokens=4096,
            betas=["context-management-2025-06-27", "compact-2026-01-12"],
            tools=[MEMORY_TOOL, *TOOLS],
            messages=messages,
            context_management=CONTEXT_EDITS,        # the §4.2 config
        )
        applied = getattr(resp, "context_management", None)
        on_budget(BudgetReport(
            turn=turn,
            input_tokens=resp.usage.input_tokens,
            output_tokens=resp.usage.output_tokens,
            cache_read_tokens=getattr(resp.usage, "cache_read_input_tokens", 0) or 0,
            edits_applied=len(applied.applied_edits) if applied else 0,
        ))
        messages.append({"role": "assistant", "content": resp.content})
        if resp.stop_reason != "tool_use":
            return resp.content[-1].text
        results = []
        for block in resp.content:
            if block.type != "tool_use": continue
            out = (memory.handle(block.input) if block.name == "memory"
                   else run_tool(block.name, block.input))
            results.append({"type": "tool_result",
                            "tool_use_id": block.id, "content": out})
        messages.append({"role": "user", "content": results})
    return "ERROR: turn cap reached"

Layer 2 — data/infra: the production backend and the forgetting policy (SQL + Python)

Memory that only grows is a liability — stale facts become §4.4's poison. The store therefore carries expiry and access metadata from day one, and a scheduled job enforces forgetting deterministically; forgetting is a policy decision, never something the model is trusted to remember to do.

create extension if not exists vector;

create table memories (
  id            uuid primary key default gen_random_uuid(),
  agent_id      text not null,
  scope         text not null check (scope in ('episodic','semantic','procedural')),
  path          text not null,            -- mirrors the memory tool's file paths
  content       text not null,
  embedding     vector(1024),             -- for similarity recall on semantic tier
  supersedes    uuid references memories(id),
  created_at    timestamptz not null default now(),
  last_accessed timestamptz not null default now(),
  expires_at    timestamptz,              -- null = pinned (procedural tier)
  unique (agent_id, path)
);
create index on memories using hnsw (embedding vector_cosine_ops);

-- the forgetting job (cron, e.g. nightly): deterministic, logged, boring
delete from memories where expires_at < now();
delete from memories
 where scope = 'episodic'
   and last_accessed < now() - interval '90 days';
-- superseded semantic facts go last so conflicts resolve toward the newest
delete from memories m
 using memories newer
 where newer.supersedes = m.id;
# Same interface as FileMemoryBackend — the loop cannot tell them apart.
import psycopg

class PgMemoryBackend:
    def __init__(self, dsn: str, agent_id: str):
        self.conn = psycopg.connect(dsn, autocommit=True)
        self.agent = agent_id

    def handle(self, cmd: dict) -> str:
        try:
            match cmd["command"]:
                case "view":
                    rows = self.conn.execute(
                        "update memories set last_accessed = now() "
                        "where agent_id = %s and path like %s "
                        "returning path, content",
                        (self.agent, cmd.get("path", "/memories") + "%"),
                    ).fetchall()
                    return "\n".join(f"== {p} ==\n{c}" for p, c in rows) or "(empty)"
                case "create":
                    self.conn.execute(
                        "insert into memories (agent_id, scope, path, content, expires_at) "
                        "values (%s, %s, %s, %s, now() + interval '90 days') "
                        "on conflict (agent_id, path) do update "
                        "  set content = excluded.content, last_accessed = now()",
                        (self.agent, self._scope(cmd["path"]), cmd["path"], cmd["file_text"]),
                    )
                    return "ok"
                case _:
                    return "ERROR: unsupported command"
        except psycopg.Error as e:
            return f"ERROR: {e}"

    def _scope(self, path: str) -> str:   # convention: /memories//...
        for s in ("episodic", "semantic", "procedural"):
            if f"/{s}/" in path: return s
        return "semantic"

Layer 3 — API/transport: streaming the BudgetReport (Python, FastAPI)

The on_budget callback from Layer 1 lands here. The rendering side is Module 01 §1.2's EventSource pattern verbatim — same subscription shape, a stacked bar instead of a checklist — so it isn't repeated; the acceptance test is that curl -N shows the pressure live.

# pip install fastapi uvicorn sse-starlette
from fastapi import FastAPI
from sse_starlette.sse import EventSourceResponse
import asyncio, json
from dataclasses import asdict

app = FastAPI()
queues: dict[str, asyncio.Queue] = {}

def on_budget_factory(session_id: str):
    q = queues.setdefault(session_id, asyncio.Queue())
    loop = asyncio.get_event_loop()
    def on_budget(report):                     # called from the agent thread, per turn
        loop.call_soon_threadsafe(q.put_nowait, asdict(report))
    return on_budget

@app.get("/sessions/{session_id}/budget")
async def budget_stream(session_id: str):
    q = queues.setdefault(session_id, asyncio.Queue())
    async def stream():
        while True:
            report = await q.get()
            yield {"event": "budget", "data": json.dumps(report)}
    return EventSourceResponse(stream())

Failure modes — memory
  1. Stale-memory poisoning. A fact that was true in March ("the rate limit is 50/s") steers a July session into repeated tool failures. Symptom: the agent confidently acts on a value no current tool result supports, and the memory store shows last_accessed updating while created_at is ancient. Mitigation: TTLs by tier, supersede-on-conflict writes, and provenance in the record so the agent can weigh age.
  2. Memory hoarding. The agent writes everything "to be safe"; recall then returns twenty near-duplicate notes and the useful one drowns — you've rebuilt context rot one query away. Symptom: memory reads whose results exceed the tool-result budget they were meant to save. Mitigation: write-time discipline in the tool description (what not to store), dedup on write, and the forgetting job.
  3. Retrieval mismatch. The fact is in the store but similarity search doesn't surface it at the moment of need — embeddings match topics, not tasks. Symptom in evals: memory-hit rate high on paraphrase probes, low on real task replays. Mitigation: retrieve by structure first (path/scope conventions, exact keys), similarity second; evaluate recall on replayed sessions, not synthetic queries (Module 11).
  4. Procedural drift. The agent edits its own instructions (procedural tier) into something subtly worse — an over-general rule learned from one bad episode. Symptom: behavior change with no deploy; the instruction file's diff history shows the moment. Mitigation: procedural memory goes through review — version control and a human gate, exactly like any other config that steers production behavior.
Principal's Judgment Call — memory framework, or files and a table?

Reach for a memory framework (LangMem, Mem0, Letta, Zep) when memory is a product feature with hard requirements — cross-session personalization at scale, temporal knowledge graphs, managed extraction pipelines you'd otherwise staff — and benchmark it on your own latency budget first: published numbers differ wildly (as of this writing, one popular option's p95 recall latency is measured in tens of seconds, which disqualifies it from interactive paths regardless of accuracy). Stay with the boring stack — instruction files in git, a memories table like Layer 2, transcripts in JSONL — when memory is an agent implementation detail: it's inspectable with psql and cat, its forgetting policy is a cron job you wrote, and there's no second vector database to operate.

The tell: if you can't state which tier (episodic/semantic/procedural) a proposed memory feature serves and what its forgetting policy is, you don't need a framework yet — you need this module's table. Frameworks package the tiers; they can't decide them for you.

§ 4.4 The pathology of degraded sessions

When a long session goes wrong, "the model got confused" is not a diagnosis. The failure taxonomy popularized by Drew Breunig gives you four named diseases, each with a distinct trace signature and a distinct cure — plus a fifth, older one. Learn to tell them apart the way you tell apart a memory leak from a connection-pool leak: the fix for one makes the other worse.

PathologyWhat happensTrace signatureFirst-line fix
PoisoningAn error or hallucination enters context and gets built upon — repetition promotes it to established factA wrong claim recurs across turns, losing its hedges each time; post-compaction it's asserted flat (§4.2)Quarantine: validate tool outputs at the boundary (M02's typed proposals), ground claims in checkable artifacts, re-verify before compaction
DistractionContext grows so large the model leans on the accumulated history instead of reasoning — it repeats past actions rather than planning new onesAction loops: near-duplicate tool calls late in session that early turns wouldn't have made; quality slides as tokens climb with no single bad inputShrink: clearing + compaction (§4.2); for research-shaped work, partition across subagents — context isolation is M03 §3.1's whole reason to exist
ConfusionIrrelevant content — unused tools, off-task documents — degrades output; the model tries to honor everything it was givenTool calls to tools that have nothing to do with the task; answers that drag in unrelated retrieved contentCurate: fewer tools per session (M02's action-space rule), retrieval thresholds, don't load "might need it" content (§4.1 table)
ClashContradictory information sits unresolved in one window — old plan vs. new instruction, stale memory vs. fresh tool resultThe agent alternates between two behaviors across turns, or follows the superseded instruction after the correctionSupersede explicitly: remove/rewrite the stale half (context editing, memory supersedes), don't just append the correction and hope recency wins
Lost-in-the-middleContent placed mid-window gets less attention than the edges — the classic long-context positional effectInstructions honored when they're first/last in the prompt, missed when the same text sits at position 60k of 120kPlacement: critical constraints live at the top (system layer) or in the latest turn; long documents get summarized headers

The diagnostic workflow when a session degrades: pull the transcript (you have one — Module 00's JSONL requirement was for this day), find the earliest turn where behavior went wrong, and read backwards asking "what in the window at that moment explains this?" Poisoning shows you the bad fact's first appearance. Distraction shows nothing wrong except size. Confusion shows the irrelevant block. Clash shows both contradicting pieces. This is exactly debugging from a heap dump, and it's why trace tooling (Module 11) treats the full per-turn context as the primary artifact, not just the final answer.

§ 4.5 Progressive disclosure: the default that makes the rest rare

Everything so far treats context pressure after it exists. The strongest move prevents it: give the agent the minimum to start, plus the means to get more — identifiers instead of payloads, indexes instead of documents, summaries with drill-down tools instead of full dumps. The agent that receives "there are 214 files; here's the tree; use read_file" stays sharp for 100 turns; the agent that receives 214 files inline is degraded by turn five, and no amount of §4.2 machinery fully claws that back.

Progressive disclosure is already threaded through the manual — it's why Module 00 truncates tool outputs at the boundary, why Module 02 §2.3 injects dynamic context per-turn instead of front-loading, and why Module 03's orchestrator sends briefs rather than transcripts. Two forward pointers complete the picture: skills (Module 08) are progressive disclosure for knowledge — a one-line description in context, a full playbook loaded only when triggered — and agentic search vs. pre-indexed retrieval (Module 06) is the same budget decision applied to data-source integration. The unifying test for any context payload: would the agent fetch this itself if given a tool and a reason? If yes, ship the tool, not the payload.

§ 4.6 Lab — loopbot survives 100 turns

LAB 04 · Estimated 3–5 hours

Mission: take loopbot (Module 00, hardened in Module 02) and give it working memory, context editing, and compaction — then prove, with the budget gauge and an eval, that it survives a 100-turn session without quality collapse.

Spec:

  1. Wire in the memory tool with the file backend (§4.3 Layer 1); add the context_management config from §4.2 with thresholds low enough to actually fire in your session (clearing at ~30k, compaction at ~80k).
  2. Emit a BudgetReport per turn and serve it over the §4.3 Layer 3 SSE endpoint. Store per-turn reports in a JSONL next to the transcript.
  3. Build a 100-turn task: a question list over a repo, fed one at a time, where five late questions (turns 85–100) depend on facts established before turn 20 ("earlier you found the config default — what was it?").
  4. Run three configurations against the same task: (a) baseline, no management — let it die or degrade; (b) editing + compaction, no memory; (c) editing + compaction + memory tool. Score the five late-recall questions in each.
  5. Induce one pathology on purpose (§4.4): plant a contradictory "fact" mid-session and document, from the transcript, whether it poisons, clashes, or gets superseded.

Acceptance criteria:

  • ✅ curl -N http://localhost:8000/sessions/lab04/budget streams live budget events, and the plotted input_tokens curve visibly saw-tooths where edits fired — the pressure-relief is observable, not asserted.
  • ✅ Configuration (a) either exhausts the window or scores ≤2/5 on late-recall; configuration (c) scores ≥4/5. Show the table.
  • ✅ The memory directory after the run contains at least one semantic note that was written before compaction fired and read after it — cite the turn numbers from the transcript.
  • ✅ A five-sentence writeup of the induced pathology naming which §4.4 signature it matched and which fix resolved it.

Stretch: swap FileMemoryBackend for the Postgres backend (Layer 2) without changing the loop, run the forgetting job mid-experiment, and show the agent gracefully re-deriving an expired fact instead of failing.

§ 4.7 Consultant's checklist

Questions to ask any team running long-lived agents
  • Can they print the window's composition — system, tools, conversation, tool results, injected — for any turn of any session? Right now, or "we'd have to add logging"?
  • What fires when the context fills: a hard death, wholesale truncation, or managed editing/compaction? Who chose the thresholds, and against what eval?
  • What is the compaction preservation contract, verbatim? Does an eval run tasks across a compaction boundary, or is summary fidelity taken on faith?
  • For each memory the system keeps: which tier is it, where does it live, and what is its forgetting policy? "It's in the vector DB" answers none of those three.
  • Is there any deterministic forgetting at all, or does memory only grow? Who reviewed the procedural tier's last self-edit?
  • When a long session degraded last quarter, did anyone name the pathology (poisoning / distraction / confusion / clash) from the trace — or was the fix "restart it"?
  • What's the largest single tool result that can enter context today, in tokens? Is that a number or a shrug?

§ 4.8 Design-review questions (staff+)

Interview / design-review bank
  1. Your agent's answer quality drops sharply after ~45 minutes of session time, with no errors logged. Walk me through the diagnosis. Strong answers reach for the transcript and per-turn token counts first, distinguish the §4.4 pathologies by trace signature (poisoning has a first-bad-fact; distraction correlates only with size; clash has two contradicting entries), and check whether a compaction event precedes the cliff.
  2. A teammate proposes loading your entire 800k-token monorepo context into the new 2M window "so the agent stops missing things." Argue it. Strong answers concede the one-shot comprehension case, then price the iterative case (tokens × turns), cite degradation-before-the-limit, and counter-propose tree + grep/read tools — progressive disclosure — with an eval to settle it.
  3. Design the compaction instructions for a customer-support agent that handles multi-hour escalations. Strong answers write a preservation contract with categories (customer identifiers, commitments made to the customer, steps already tried, open ticket state), demand verbatim preservation of promises, and add a post-compaction eval probing exactly those categories.
  4. When does memory belong in the system prompt versus a store behind a tool? Strong answers use size × stability: small and always-relevant (identity, top preferences, procedural rules) is pinned in-context where it's cacheable; large or conditionally relevant is behind retrieval; and anything the model itself should update flows through the tool so writes are observable and gated.
  5. Argue both sides: "context editing should be invisible to the model — never tell it things were removed." Strong answers note that placeholders the model can see let it re-fetch deliberately (transparency aids planning), while invisible edits preserve conversational flow but risk the model referencing content that no longer exists; land on visible-but-cheap markers plus exclude_tools for anything it must never lose.
