FM-05 · Architecture

Frameworks: The Honest Comparison

LangGraph, PydanticAI, Claude Agent SDK, OpenAI Agents SDK, CrewAI, AG2/Microsoft Agent Framework, Spring AI, Mastra, and smolagents, compared on control-flow ownership, checkpoint/resume, typed I/O, and lock-in — not GitHub stars.

PUBLISHED 2026-07-15 DEPTH ~5,800 words PREREQS Module 02 LAB port one workflow across two frameworks + no framework; write the migration-cost memo

§ 5.0 Orientation

A team's CrewAI-based document-review crew has been shipping for four months. Support opens a ticket: a review that used to take ninety seconds now takes eleven minutes, and the on-call engineer's first move is to open the trace — except the trace shows six inter-agent messages for a task that should have taken two. Nobody remembers deciding on six; the crew's role-based agents kept "checking in" with each other because nothing capped it, and the framework's own execution log doesn't match what actually ran closely enough to say why. The fix isn't a bug fix. It's a rewrite, because the team picked the framework in a two-day spike based on which one had the best onboarding tutorial, not on which one they could operate at the failure mode they eventually hit.

That's the failure this module exists to prevent: choosing a framework the way you'd choose a menu item, then discovering eighteen months later that the choice was actually an architecture decision with a migration cost attached. Module 00 built the raw loop and asked who should own the harness around it. This module answers that question concretely — across nine current options, on the axes that actually predict pain, with code showing what changes and what doesn't when a framework's internal state has to reach a browser.

§ 5.1 What a framework buys — and what teams did before one existed

Before there was a market of agent frameworks, teams that needed more than Module 00's forty-line loop built their own harness: a hand-rolled state machine to track which step an agent was on, a database table to persist that state so a crashed process could resume, a bespoke retry wrapper around every tool call, and — if the product needed a human to approve anything — a queue and an approval UI built from scratch. None of that is agent-specific engineering; it's the same undifferentiated plumbing every stateful, long-running system needs, and most teams built a worse version of it than they'd have built for a payments system, because it felt like a side quest to the "real" work of prompting the model well.

A framework's honest pitch is: we already built the plumbing, wired it to a specific control-flow model, and tested the edges you haven't hit yet. That's a real trade, and for the plumbing pieces — checkpointing, retry policy, streaming, a permission model, session persistence — most teams are better off adopting one than re-deriving it. What a framework does not buy you is a better prompt, a better tool schema, or a better eval; those are still entirely your problem, and no framework's marketing page will tell you that plainly.

Principal's Judgment Call — framework or no framework

Reach for a framework when your control flow needs at least two of: checkpoint/resume across a process restart, a human-in-the-loop pause that can last hours or days, streaming intermediate state to a caller, or a permission model you don't want to hand-write. Those are the pieces that are genuinely hard to get right the first time, and a mature framework has already hit the edge cases you haven't.

Stay with no framework — Module 00's loop, extended by hand — when your control flow is one or two steps deep, runs to completion in seconds, and has no pause points. The tell in review: if you can draw the entire control flow on a whiteboard in under a minute and nobody argues about an edge, you don't need a framework's abstraction between you and the model's raw output. Teams that add LangGraph to a single tool-use loop are usually buying abstraction they'll pay for in indirection and never redeem in capability.

§ 5.3 Two different bets: explicit graph state vs. typed single agent

Underneath the marketing, every framework in §5.2 is making one of two structural bets. LangGraph, LangGraph4j, and Mastra's workflow layer bet that your control flow itself — the branches, loops, and pause points — deserves to be a first-class, inspectable, checkpointed object. PydanticAI, Spring AI, and the vendor SDKs bet that the hard part is the single agent's tool loop and its output contract, and that orchestration around it is just code you write. Neither bet is wrong; they're optimizing for different failure modes, and picking wrong shows up as friction six months in, not at week one.

The typed-output half of Bet 2 is a framework-level guarantee of something Module 02 already made you build by hand: a validated boundary between the model's probabilistic output and the deterministic code that acts on it (Module 02 §2.2's typed proposal boundary). PydanticAI, Spring AI's structured-output converters, and OpenAI's Guardrails all sell a version of that same boundary as a built-in — worth adopting for the plumbing, but the design discipline underneath it is the one Module 02 already taught, not a new idea.

# pip install langgraph langgraph-checkpoint-postgres
from typing import TypedDict, Literal
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.postgres import PostgresSaver

class ReviewState(TypedDict):
    diff: str
    findings: list[str]
    risk: Literal["low", "medium", "high"]
    approved: bool

def review_node(state: ReviewState) -> ReviewState:
    findings, risk = run_review_model(state["diff"])   # your model call
    return {**state, "findings": findings, "risk": risk}

def gate_node(state: ReviewState) -> ReviewState:
    return state   # execution pauses here; see interrupt_before below

graph = StateGraph(ReviewState)
graph.add_node("review", review_node)
graph.add_node("gate", gate_node)
graph.set_entry_point("review")
graph.add_edge("review", "gate")
graph.add_conditional_edges(
    "gate", lambda s: END if s["approved"] else "review"
)

with PostgresSaver.from_conn_string(DB_URL) as checkpointer:
    checkpointer.setup()
    app = graph.compile(checkpointer=checkpointer, interrupt_before=["gate"])
    # thread_id is the resume key: kill the process, restart it days later,
    # invoke again with the same thread_id, and the graph picks up at "gate".
    app.invoke(
        {"diff": diff_text, "findings": [], "risk": "low", "approved": False},
        config={"configurable": {"thread_id": pr_id}},
    )
# pip install pydantic-ai
from typing import Literal
from pydantic import BaseModel
from pydantic_ai import Agent

class ReviewVerdict(BaseModel):
    findings: list[str]
    risk: Literal["low", "medium", "high"]

review_agent = Agent(
    "claude-sonnet-5",
    output_type=ReviewVerdict,          # the framework validates this — no schema drift
    system_prompt="Review this diff for correctness bugs only. No style comments.",
)

result = review_agent.run_sync(diff_text)
verdict: ReviewVerdict = result.output  # typed; downstream code never sees a raw string

# There is no graph, no checkpointer, no pause primitive here — the human-approval
# gate and the resume-after-restart logic are entirely your code's responsibility,
# exactly as they'd be with Module 00's raw loop. PydanticAI bought you the
# validated-output boundary; it did not buy you the state machine around it.
if verdict.risk != "low":
    queue_for_human_approval(pr_id, verdict)  # your table, your queue, your resume path

Read the comment at the bottom of the PydanticAI tab twice — it's the whole comparison in one sentence. LangGraph's checkpointer means the pause-and-resume machinery in the first tab is thirty lines you didn't write. The second tab's queue_for_human_approval is exactly that machinery, and PydanticAI will not write it for you; you get the typed boundary and nothing else. Neither is a bug. They're answering different questions about where the twenty lines the vendor SDKs also skip should live.

§ 5.4 Full-stack: a framework's checkpoint reaching a browser without leaking its shape

The interesting full-stack question isn't "how do you stream agent progress to a UI" — Module 01 §1.2 already built that SSE endpoint and its EventSource counterpart, and Module 03 §3.4 reused the identical pattern for a different payload; neither is repeated here. The interesting question for this module is narrower: when the state being streamed is a framework's own checkpointed object, how much of the framework's internal shape should be allowed to cross the wire? Get this wrong and the API contract — and every frontend built against it — silently depends on LangGraph's internal field names, which is exactly the lock-in problem §5.5 names next.

Layer 1 — data/infra: the checkpointer replaces a hand-rolled table

Module 01 §1.2 and Module 03 §3.4 each hand-rolled a Postgres table (plan_nodes, findings) to persist progress, because neither had a framework underneath them. Here, LangGraph's PostgresSaver from §5.3's first tab is that table — it owns its own schema, migrations, and thread-keyed rows. That's a real win: one less table for your team to design, index, and back up. It's also the trade this module keeps returning to — you no longer control that schema, which becomes §5.6's first failure mode.

Layer 2 — agent runtime: stream node completions as generic events

The only new code at the API boundary is the translation step in FIG 5.2's middle box — pulling LangGraph's typed state out of its own shape before it ever reaches the SSE endpoint:

async for chunk in graph_app.astream(
    initial_state, config={"configurable": {"thread_id": pr_id}},
    stream_mode="updates",             # one event per completed node
):
    node_name, node_state = next(iter(chunk.items()))
    await queue.put({                  # <- the ONLY framework-shaped read in the request path
        "node": node_name,
        "status": "waiting_approval" if node_name == "gate" else "done",
        "findings": node_state.get("findings"),
    })

That dict is the same generic {node, status, findings?} event shape Module 01's SSE endpoint already serializes and Module 01's EventSource handler already renders — plug it into that endpoint's existing queue.put(...) call and the frontend needs zero changes. Swap LangGraph for PydanticAI-plus-your-own-queue next quarter and the only file that changes is this one.

§ 5.5 Keeping any framework at arm's length

Lock-in isn't a framework being hard to remove — every nontrivial dependency is somewhat hard to remove. It's a framework's types leaking past the module boundary where you first adopted it, so that removing it means touching code that has nothing to do with agents. The mitigation is the oldest trick in the book, ports-and-adapters, applied deliberately at exactly two seams: keep tool implementations as plain functions that a thin adapter registers with whichever framework's schema format is current, and keep the API/transport contract in a generic shape (FIG 5.2's translation boundary) instead of serializing the framework's own state object directly.

Do this and a framework migration becomes "rewrite the adapter and the graph/agent definition," which is real work but boundable in a sprint. Skip it — call the framework's types directly from your route handlers and your domain logic — and the same migration becomes "grep the codebase for every place `ReviewState` or `RunResult` appears," which is how a two-day framework spike turns into a quarter-long rewrite.

Principal's Judgment Call — hybrid framework stacks

Reach for a hybrid — PydanticAI's typed agents dropped in as LangGraph nodes, which is a genuinely common 2026 pattern — when you want validated output and checkpointed control flow, and your team already knows both tools well enough that the integration is a day, not a design project.

Stay with a single framework when your team is small enough that "two frameworks to onboard new engineers on" is a real cost, not a rounding error. The hybrid pattern is a sign of a mature stack, not a starting point — teams that reach for it in month one are usually solving a problem they don't have yet at the cost of a problem (two sets of upgrade cadences, two failure surfaces) they now definitely do.

Principal's Judgment Call — LangGraph4j vs. Spring AI for JVM shops

Reach for LangGraph4j when the control flow itself is graph-shaped and needs the same checkpoint/resume/human-pause story the Python ecosystem gets from LangGraph — it mirrors that model closely, including checkpoint savers for Postgres, MySQL, Redis, and DynamoDB.

Reach for Spring AI's ChatClient advisor chain when the team is already deeply Spring-idiomatic and the agentic piece is fundamentally a single-agent tool loop threaded through existing @Service beans — its 2.0 line moved tool-calling into a composable advisor chain specifically so it reads like the rest of a Spring app, not like a bolted-on DSL. Don't reach for LangGraph4j just because "that's what the Python team uses" — matching your JVM team's existing idioms is worth more than cross-language symmetry.

Failure modes — frameworks
  1. Checkpoint schema drift across a version upgrade. A minor framework bump changes how the checkpointer serializes your state object; threads paused before the upgrade fail to deserialize on resume. Symptom: a batch of long-lived human-approval threads all fail with a deserialization or missing-key error the moment someone tries to approve them, correlating exactly with a deploy that "only" bumped a framework version. Mitigation: pin the framework version for any service holding long-lived paused state, and treat checkpoint schema like a database migration — versioned and tested before the bump ships.
  2. Framework retries wrap a non-idempotent tool call. A built-in retry policy re-runs a node after a transient error, assuming the node is idempotent; the node isn't, and the second run repeats a side effect (a charge, an email, a write to an external system). Symptom: duplicate downstream records or duplicate customer-facing actions that correlate with a caught-and-retried exception in the trace, not with two separate legitimate invocations. Mitigation: make every tool call idempotent at the boundary (idempotency keys, upserts) before trusting any framework's retry default — the framework cannot know which of your side effects are safe to repeat.
  3. Role-based coordination without a cap. A CrewAI-style crew's agents "check in" with each other with no bound on inter-agent messages; token cost and latency scale with conversation length nobody set a limit on, and the framework's own execution trace doesn't cleanly show which message caused which outcome. Symptom: a cost-per-task dashboard (Module 13 builds this) shows one topology costing five to ten times a single-agent equivalent for the same eval score, and root-causing a slow run from the trace alone doesn't work. Mitigation: cap inter-agent turns explicitly, or fall back to an orchestrator-workers topology (Module 03 §3.2) with a bounded, code-owned dispatch instead of open-ended peer chatter.
  4. Vendor-SDK lock-in surfacing at the worst time. Business logic and tool implementations get written directly against a vendor SDK's types (Claude Agent SDK's permission/hook model, OpenAI Agents SDK's Handoffs) with no adapter boundary; a pricing, procurement, or model-availability change forces an emergency "can we switch providers" evaluation. This is not hypothetical — Anthropic announced moving Agent SDK usage onto separate metered billing in May 2026, then paused the change the day it was due to take effect after developer pushback, which is exactly the kind of surprise a tightly-coupled integration has no slack to absorb. Symptom: a provider-swap spike that should take a sprint takes a quarter because tool logic, prompts, and framework primitives are all in the same files. Mitigation: §5.5's ports-and-adapters seam, applied before you need it, not after.

§ 5.6 Lab — port one workflow, three ways, and price the migration

LAB 05 · Estimated 4–6 hours

Mission: implement the same typed-review workflow from §5.3 three ways — no framework (extend Module 00's loop by hand), LangGraph with a checkpointer, and PydanticAI — prove behavioral parity, then time-box a real migration attempt and write down exactly what broke.

Spec:

  1. All three implementations take the same diff text and return the same shape: a findings list and a low/medium/high risk classification.
  2. The LangGraph implementation must use a real checkpointer (Postgres or SQLite) with interrupt_before=["gate"], exactly as in §5.3.
  3. Pick whichever implementation you'd actually ship and wire it to Module 01 §1.2's SSE endpoint via §5.4's translation boundary, so a run is observable live — either in a browser or with curl -N http://localhost:8000/reviews/<id>/events.
  4. Time-box thirty minutes to port your chosen implementation to a fourth framework (Spring AI if you built Python, or vice versa) and stop the clock the moment something doesn't translate cleanly.

Acceptance criteria:

  • ✅ A shared golden test file of five diffs produces the same finding count and risk classification (not necessarily identical wording) across all three implementations — behavioral parity, not just three programs that compile.
  • ✅ Kill the LangGraph process mid-run with kill -9 after the review node completes but before gate; restart it with the same thread_id and confirm it resumes at gate instead of re-running review.
  • ✅ curl -N against the SSE endpoint (or the browser component) shows live node-by-node events for at least one implementation, using the exact event shape from §5.4.
  • ✅ A half-page migration-cost memo naming precisely which lines didn't port cleanly in the thirty-minute port attempt (state shape, tool schema format, permission model, or the pause primitive) and an honest hour estimate to migrate a real production agent between your two chosen frameworks.

§ 5.7 Consultant's checklist

Questions to ask any team that just picked a framework
  • Which two or more of checkpoint/resume, long human-in-the-loop pauses, streaming, or a permission model does this workload actually need? If the honest answer is zero or one, why is there a framework here at all?
  • Where does business logic live — in plain functions a thin adapter registers with the framework, or directly inside the framework's own node/agent/tool types?
  • What is the plan the day the checkpointer's schema changes across a version bump — is it tested, or discovered in production?
  • For any multi-agent topology built inside a framework: is inter-agent communication bounded by code, or open-ended and hoped to converge?
  • If this vendor's SDK became unavailable or repriced tomorrow, what fraction of the codebase would need to change, and has anyone actually measured that instead of estimating it?
  • Was the framework chosen after a spike that stressed the workload's actual failure modes, or after a tutorial that stressed the happy path?

§ 5.8 Design-review questions (staff+)

Interview / design-review bank
  1. A team wants to migrate a working CrewAI prototype to production and proposes LangGraph as the target. What do you ask before approving that? Strong answers separate "the framework was wrong" from "the topology chosen inside it was wrong" (Module 03 §3.2's coordination tax applies regardless of framework), ask for eval evidence of where CrewAI concretely fails today — latency, cost, or missing resumability — and only endorse LangGraph if the gap is specifically checkpointing/HITL, not just general dissatisfaction.
  2. Your org standardized on Claude Agent SDK for internal tools. A new product initiative needs to stay model-agnostic for procurement reasons. How do you architect for that without doubling the build? Strong answers apply §5.5's ports-and-adapters seam: business logic and tool implementations stay in plain functions behind a thin internal interface, the SDK's permission/hook wiring stays isolated at the harness edge, and the team accepts that a future switch means rewriting the edges, not the domain logic.
  3. A LangGraph agent's checkpointer fails to deserialize after a routine framework version bump, and forty paused human-approval threads are stuck. Diagnose, then propose the process change that prevents recurrence. Strong answers name checkpoint schema drift as the root cause (§5.6's first failure mode), propose a scripted migration for the stuck threads, and going forward treat checkpointer schema like a database migration — versioned, tested against the pinned framework version before any bump ships to a service holding long-lived state.
  4. When would you deliberately choose CrewAI over LangGraph for a production system, despite this module's cost and observability criticisms? Strong answers name genuine fits — fast prototyping with non-engineers who think in roles, small bounded topologies where the coordination tax doesn't bite, or an org with existing CrewAI-specific tooling that offsets the weaknesses — and explicitly reject "it's more popular" or "it was first" as reasons.
  5. How do you decide, for a JVM shop, between LangGraph4j and Spring AI's advisor chain? Strong answers use control-flow shape as the test, not language symmetry with a Python team: LangGraph4j when the workflow itself is graph-shaped and needs checkpoint/resume/HITL; Spring AI's advisor chain when the agentic piece is a single-agent tool loop that should read like the rest of an idiomatic Spring codebase (§5.5's second sidebar).
