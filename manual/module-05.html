Frameworks: The Honest Comparison

Ten frameworks, one axis-by-axis table, the same tool-call loop ported four ways — and the migration-cost numbers nobody puts in the pitch deck.

PUBLISHED 2026-07-14 DEPTH ~6,200 words PREREQS Module 02 LAB port one workflow across two frameworks + no-framework; write the migration-cost memo

§ 5.0 Orientation

Module 00's loopbot shipped as roughly 120 lines: a while-loop, a tool dispatch table, a turn cap, a JSONL transcript. It worked. Then a PM asked for a pause-for-approval step before any destructive tool call, an SRE asked for the run to survive a pod restart without starting over, and a frontend engineer asked for live progress in the UI instead of tailing a log file. None of those are exotic asks — they're Module 01's typed proposal boundary, Module 09's checkpoint/resume, and Module 01 §1.2's SSE pattern, respectively. But now three people are asking for three capabilities in the same sprint, and "we'll add it to the while-loop" starts to sound like a framework's job description read back to you one feature at a time. That's the actual moment teams reach for a framework — not because the loop is hard, but because the surrounding capabilities compound faster than a hand-rolled harness can absorb them under deadline pressure.

This module is not a listicle. It's an axis-by-axis comparison of the frameworks a senior engineer will actually be asked to choose among in 2026 — LangGraph (and LangGraph4j for the JVM), PydanticAI, Claude Agent SDK, OpenAI Agents SDK, CrewAI, AG2 (the community-maintained AutoGen fork), Spring AI, Semantic Kernel, smolagents, and Mastra — plus the no-framework baseline, which stays a first-class option throughout, not a strawman. The same minimal tool-call loop gets ported across four of them in §5.3 so the differences are concrete, not adjectival. As of this writing (2026-07-14), version numbers and API surfaces below are moving targets; the axes they move along — control flow, state, streaming, HITL, typed I/O, lock-in — are stable enough to plan a decision around.

§ 5.1 What a framework actually buys — and what it costs

Strip the marketing and every agent framework is selling some subset of six capabilities, all of which Module 00–04 already showed you how to build by hand: (1) a control-flow model for sequencing steps and branching, (2) state persistence and checkpointing so a killed process resumes instead of restarting, (3) streaming primitives for surfacing partial progress, (4) human-in-the-loop interrupt points, (5) a typed input/output contract at the model boundary, and (6) observability — traces, spans, cost accounting. None of these is beyond a competent team's ability to build. What a framework actually buys is that someone else already built them, tested them against edge cases you haven't hit yet, and will keep maintaining them — for the price of adopting that framework's opinion about how each one should work.

The corollary: don't evaluate a framework by counting its features. Evaluate it by which of the six axes above you actually need this quarter, and how much you'll pay in the framework's own opinions to get them. A framework that nails checkpointing but forces its state shape into every node's function signature is a great trade if you need checkpointing and a bad one if you don't.

Three patterns worth naming explicitly. First, the provider-native SDKs (Claude Agent SDK, OpenAI Agents SDK) buy you the tightest integration with their own model family's features — hooks, guardrails, computer use — at the explicit cost of portability; picking one is a bet on that vendor, not a neutral technical choice. Second, "typed I/O" is not evenly distributed: PydanticAI and Mastra treat it as the framework's entire reason to exist, while CrewAI and AG2 treat it as something you bolt on, which matters enormously once Module 01's verification funnel (§1.5) is the thing standing between a malformed plan and a scheduler crash. Third, checkpointing quality tracks almost exactly with how graph-shaped the framework is — LangGraph/LangGraph4j and Mastra checkpoint at node boundaries because their control-flow model has explicit boundaries to checkpoint at; conversational frameworks (AG2, CrewAI) don't, because a multi-turn negotiation doesn't have a clean "resume point" the way a DAG node does.

§ 5.3 The same loop, four ways

Abstractions are cheap to describe and expensive to feel. Below is the same minimal capability — Module 00's tool-call turn, propose an action, validate it, execute it — implemented with no framework, with LangGraph, with the Claude Agent SDK, and with Spring AI + LangGraph4j. Read them side by side, not for syntax trivia, but for where each one puts the seam between "the model's idea" and "code that actually runs it."

from langgraph.graph import StateGraph, END, START
from typing import TypedDict, Annotated
import operator

class LoopState(TypedDict):
    messages: Annotated[list, operator.add]
    turn: int

def assistant(state: LoopState) -> LoopState:
    resp = client.messages.create(model=MODEL, system=SYSTEM,
                                   messages=state["messages"], tools=TOOLS)
    return {"messages": [resp], "turn": state["turn"] + 1}

def tools_node(state: LoopState) -> LoopState:
    call = next(b for b in state["messages"][-1].content if b.type == "tool_use")
    action = ProposedAction.model_validate(call.input)      # same boundary, framework-hosted
    result = TOOL_IMPL[action.tool](action.args)
    return {"messages": [{"role": "user", "content": [
        {"type": "tool_result", "tool_use_id": call.id, "content": result}]}]}

def route(state: LoopState) -> str:
    return "tools" if state["messages"][-1].stop_reason == "tool_use" else END

graph = StateGraph(LoopState)
graph.add_node("assistant", assistant)
graph.add_node("tools", tools_node)
graph.add_edge(START, "assistant")
graph.add_conditional_edges("assistant", route, {"tools": "tools", END: END})
graph.add_edge("tools", "assistant")
app = graph.compile()   # checkpointer + interrupt added in §5.4

Notice what's actually different: the no-framework version and the LangGraph version have identical tool-dispatch logic — the only thing LangGraph adds is a graph wrapper around the same two functions, which is exactly why it's a low-regret adoption for teams already comfortable with the loop shape. The Claude Agent SDK version is shorter because the loop itself is managed — you get hooks into it rather than owning it, which is a better trade when you want a coding-agent-shaped harness (Module 09) and a worse one when your control-flow needs don't match the SDK's assumptions. The Java version shows the same seam (a validated ProposedAction before TOOL_IMPL runs) surviving translation into a completely different ecosystem — which is the actual claim this module is making: the typed proposal boundary is a design decision, not a library feature, and it should survive every framework migration you ever do.

§ 5.4 Full-stack: porting loopbot to LangGraph with checkpointing, an approval gate, and a live UI

Picking up the graph from §5.3, here is the actual reason a team reaches for LangGraph specifically: durable checkpointing plus a first-class human-approval interrupt, wired all the way to a frontend that can act on it — not just watch it.

Layer 1 — agent runtime: checkpointed graph with an approval interrupt (Python)

# pip install langgraph langgraph-checkpoint-postgres
from langgraph.checkpoint.postgres import PostgresSaver

checkpointer = PostgresSaver.from_conn_string(DATABASE_URL)
checkpointer.setup()   # creates the checkpoint tables once

app = graph.compile(
    checkpointer=checkpointer,
    interrupt_before=["tools"],   # pause before every tool call — HITL, not optional
)

def start_run(run_id: str, goal: str):
    config = {"configurable": {"thread_id": run_id}}
    app.invoke({"messages": [{"role": "user", "content": goal}], "turn": 0}, config)
    # execution is now paused at "tools" — the pending call is in app.get_state(config)

def resume_run(run_id: str, approved: bool):
    config = {"configurable": {"thread_id": run_id}}
    if not approved:
        app.update_state(config, {"messages": [{"role": "user", "content": "Tool call denied by reviewer."}]})
        return
    app.invoke(None, config)   # None input = resume from the checkpoint, not from scratch

Layer 2 — API/transport: SSE for state, REST for the decision (Python, FastAPI)

# pip install fastapi uvicorn sse-starlette
from fastapi import FastAPI
from sse_starlette.sse import EventSourceResponse
from pydantic import BaseModel
import json

app_api = FastAPI()

@app_api.get("/runs/{run_id}/events")
async def run_events(run_id: str):
    async def stream():
        config = {"configurable": {"thread_id": run_id}}
        state = app.get_state(config)
        pending = state.tasks[0].interrupts if state.tasks else None
        if pending:
            yield {"event": "interrupt", "data": json.dumps({"pending_call": state.values["messages"][-1].content})}
        else:
            yield {"event": "complete", "data": json.dumps({"result": state.values["messages"][-1].content})}
    return EventSourceResponse(stream())

class ResumeDecision(BaseModel):
    approved: bool

@app_api.post("/runs/{run_id}/resume")
def resume(run_id: str, decision: ResumeDecision):
    resume_run(run_id, decision.approved)   # Layer 1's function — no re-derivation here
    return {"status": "resumed", "approved": decision.approved}

The fourth layer — data/infra — is the checkpoint tables PostgresSaver.setup() creates: one row per (thread_id, checkpoint) pair holding the full graph state as JSON, so a killed API process and a browser tab that reloaded mid-approval both reconstruct the identical pending state from the same source of truth. This is the concrete difference between this module's approval flow and Module 01 §1.2's read-only PlanChecklist: that UI only ever displayed state a different system owned; this one's ApprovalCard is a legitimate write path into the agent's execution — which is exactly why the resume endpoint validates approved as a typed, narrow boolean rather than accepting free-form instructions back into the graph.

Failure modes — frameworks and migrations
  1. Checkpoint-format lock-in. A minor LangGraph upgrade changes the serialized shape of checkpointed state, and threads paused before the upgrade fail to resume. Symptom in the trace: resume_run raises a deserialization error only for thread_ids created before a specific deploy timestamp. Mitigation: version your state schema explicitly and write a migration step for in-flight checkpoints before any framework upgrade that touches state shape — treat it like a database migration, because it is one.
  2. The framework's control flow silently overrides your typed boundary. A conversational framework (AG2, CrewAI) coerces a malformed tool argument into "something plausible" instead of failing, because its internal message-passing assumes free-form text, not validated schemas. Symptom: an eval failure with no corresponding validation error anywhere in the logs — the framework accepted a bad shape you thought you'd guarded against. Mitigation: re-validate at the framework/tool boundary yourself; never assume a framework's internal parsing implements Module 01 §1.5's fail-closed rule the way your own code would.
  3. Hidden retries multiply cost without moving the accuracy needle. A framework's default node-level retry policy fires on a transient tool error, but the underlying tool is deterministic and the actual bug is upstream — so every retry reproduces the identical wrong call three times before surfacing. Symptom: token/cost dashboards show 3x spend on a task class with unchanged accuracy. Mitigation: know your framework's default retry policy before you ship it, and cap or disable it in favor of your own turn-cap logic (Module 00 §0.5) when a tool's failures are logic bugs, not flakiness.
Principal's Judgment Call — framework or no framework?

Reach for a framework when at least two of the six axes in §5.1 are both genuinely needed and non-trivial to build correctly — durable checkpointing across process restarts and a real HITL interrupt are the pair that most often clears this bar, because both have sharp edges (partial writes, race conditions between a resume call and a still-running node) that a battle-tested implementation has already found for you. Stay with no framework when you need at most one axis, or when your control-flow shape is simple enough that a graph library's abstraction costs you more legibility than it returns — a straight-line ReAct loop wrapped in a graph node for its own sake is pure overhead.

The tell in review: ask "which of these six capabilities would take my team more than a week to build correctly, including the edge cases?" If the honest answer is zero, you don't need a framework yet — you need to notice you're asking the question before the deadline, not after.

§ 5.5 Keeping any framework at arm's length

Whichever framework wins the axis comparison, the engineering move that actually protects you is the same one Module 02 argued for at the tool boundary: put your own interface between your business logic and the framework's API, so a framework swap touches one adapter module instead of every call site. Concretely, don't let ProposedAction, your tool implementations, or your eval harness (Module 11) import framework types directly — define them once, plain, and write a thin adapter per framework that maps its node/state/message shapes onto your types. It costs a day of ceremony up front. It's the difference between the "3 weeks, 60% of the codebase changed" migration story and a genuinely moderate one, because the 60% that had to change was business logic tangled with framework imports, not an adapter.

§ 5.6 Lab — port one workflow across two frameworks + no-framework; write the migration-cost memo

LAB 05 · Estimated 4–6 hours

Mission: take Module 00's loopbot (or your own §5.6-scale agent) and implement the identical task three ways — no framework (you likely already have this from Module 00), LangGraph with a Postgres checkpointer and an approval interrupt (§5.4), and one more framework of your choice from the §5.2 table that isn't LangGraph — then write a migration-cost memo grounded in what you actually measured, not vibes.

Spec:

  1. Confirm all three implementations pass the same 5-task eval you'd use from Module 11 — correctness first, comparison second.
  2. For the LangGraph version, wire §5.4's full stack end to end: start a run, watch the SSE stream show an interrupt event with the actual pending tool call, and confirm POST /resume with approved: false denies it without crashing the process.
  3. Kill the API process mid-run (after a checkpoint, before resume) and restart it. Confirm GET /runs/:id/events reconstructs the same pending interrupt from Postgres rather than losing the run.
  4. Port the identical task into your third framework choice. Count concretely: lines of business logic that had to change (not just syntax), and any of the six §5.1 axes your third framework simply doesn't offer that you had to build yourself anyway.

Acceptance criteria:

  • ✅ curl -N http://localhost:8000/runs/todo-review/events streams an interrupt event containing the real pending tool call as JSON, not a placeholder string.
  • ✅ A killed-and-restarted process resumes the same paused run — captured as a before/after log or screenshot.
  • ✅ A migration-cost memo (1 page) stating: total lines changed per framework switch, which of the six axes each framework covers vs. leaves to you, and a recommendation with the reasoning made explicit — not "framework X felt cleaner."
  • ✅ At least one induced failure per framework, captured in the transcript: force the LangGraph checkpoint-version mismatch (§5's first failure mode) by intentionally changing your state schema between two checkpointed runs, and force your third framework's weakest axis to fail visibly (e.g., kill a CrewAI run mid-task and observe there's no resume path).

Stretch: build the §5.5 adapter layer first, then re-measure the third-framework port's line count. Does the adapter pay for itself on a single migration, or only across multiple ones?

§ 5.7 Consultant's checklist

Questions to ask any team choosing (or defending) a framework
  • Which of the six §5.1 axes does this team actually need in the next two quarters — and which are they adopting "just in case"?
  • Is there an adapter layer between business logic and the framework's types, or does a framework swap today mean touching every call site?
  • What is the framework's default retry/error policy, and has anyone read it, or is it running as shipped?
  • For any HITL interrupt: does the human reviewer see the actual pending action, or a generic "approval needed" with no specifics — the difference between a real gate and rubber-stamping (Pillar 9's approval-UX territory)?
  • If this framework were deprecated tomorrow, what's the honest estimate of the rewrite — and does it match the 50–80% figures teams have reported migrating off CrewAI, or is this team's state genuinely more portable?
  • Is the chosen framework's typed-I/O story strong enough to carry Module 01 §1.5's fail-closed verification rule, or does the team believe it is and never checked?

§ 5.8 Design-review questions (staff+)

Interview / design-review bank
  1. A team wants to migrate from CrewAI to LangGraph because "CrewAI can't checkpoint." What do you ask before approving the migration? Strong answers ask which specific axis is actually blocking them (checkpointing, most likely, per §5.2), whether an adapter layer exists to bound the blast radius, and whether the team has measured what fraction of their current code is business logic vs. CrewAI-specific role/crew wiring — since that ratio predicts whether this is a 3-week or 3-month project.
  2. Your org standardized on the Claude Agent SDK for coding agents. A new team wants to build a customer-support agent on it too. Approve or push back? Strong answers separate "same vendor, different shape": a coding agent's hook-based lifecycle and a support agent's typed-escalation-boundary needs (Module 14) are different axis priorities — provider-native lock-in is fine when the provider's model is genuinely the right one for both use cases, but the SDK choice shouldn't be assumed to transfer just because it worked once.
  3. A framework upgrade silently changed checkpoint serialization and broke resume for in-flight runs. How do you prevent a recurrence? Strong answers propose versioning the state schema explicitly, writing a migration path (or a compatibility shim) as part of any framework version bump that touches state, and adding a canary check — resume a known paused thread — to the upgrade's release checklist, not just a green test suite.
  4. Argue both sides: "No framework is always the right starting point; frameworks are for teams that already know exactly what they need." Strong answers concede the no-framework default is right for exploration and for teams under two axes of real need, then push back that "know exactly what you need" is a luxury — teams under real deadline pressure sometimes correctly buy a framework's opinions to avoid building and hardening checkpointing themselves, and the actual skill is naming the tradeoff, not avoiding it.
  5. Design the decision process for choosing a framework across Python, TypeScript, and Java teams in the same org. Strong answers resist forcing one framework across all three ecosystems — LangGraph/PydanticAI for Python, Mastra or the Claude Agent SDK's TS bindings for TypeScript, Spring AI + LangGraph4j for Java — and instead standardize the adapter-layer contract (§5.5) and the typed-I/O expectations, so the six axes are met consistently even though the library isn't the same across languages.
