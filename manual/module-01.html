Reasoning & Thought Processes

ReAct three years on, plan-and-execute DAGs with a typed proposal boundary, reflection's error-lamination trap, extended/adaptive thinking, and how to verify a model's output without trusting it.

PUBLISHED 2026-07-10 DEPTH ~5,000 words PREREQS Module 00 LAB same task, three reasoning strategies, evidence-backed pick

§ 1.0 Orientation

Module 00 defined the loop — context in, one inference pass, tool or stop — and named the "reason" step without opening it up. That step is not one thing. How the model decides what to do next is itself an architectural choice, with cost, latency, and accuracy consequences that show up directly in your p50/p99 dashboards. This module opens the reasoning step: the four dominant strategies (ReAct, plan-and-execute, reflection, extended thinking), when each earns its cost, and — because a reasoning strategy that can't be checked is just an opinion — how to verify what comes out the other end.

The throughline is Module 00's typed proposal boundary, generalized: whatever strategy produces the next action, that action should cross into your deterministic code as a validated, typed object, never as a trusted string. Everything below is in service of that one rule.

§ 1.1 ReAct, three years on

Yao et al.'s 2022 ReAct paper gave the field its default reasoning pattern: interleave Thought (a short private reasoning step), Action (a tool call), and Observation (the tool's result), repeating until done. If that sounds familiar, it should — the loop you built in Module 00's lab is ReAct. Every modern tool-calling API (Claude's tools parameter, OpenAI's function calling) is ReAct with the "Thought:" / "Action:" text parsing replaced by structured tool-use blocks the model emits natively. The paper's contribution wasn't the loop shape — it's that letting the model narrate a reason before acting measurably reduces hallucinated actions compared to acting with no visible intermediate reasoning at all. We don't cite the paper for the mechanism anymore; we cite it for that empirical result, which is why "Thought" survived as extended thinking (§1.4) even after the scratchpad format died.

Three years on, the honest limitation is well understood: ReAct is myopic by construction. Each Thought is conditioned on the context accumulated so far, and it picks the next action — it has no representation of the whole task's shape, no lookahead, and no way to notice "step 4 makes step 7 impossible" until it arrives at step 7. On short, tool-rich, exploratory tasks (Module 00's file-QA loopbot) that's a feature: the model adapts every turn to what it just learned. On long-horizon tasks with real structure — "migrate this schema across 12 services" — pure ReAct tends to drift, re-derive the same sub-plan differently each run, and burn turns rediscovering constraints a one-time upfront plan would have caught.

Nothing to build here — you already did (Module 00 §0.5). The engineering question ReAct forces onto every design review is: does this task have enough structure to plan upfront, or is discovery the point? §1.2 answers the first case; ReAct stays correct for the second.

§ 1.2 Plan-and-execute: trading reactivity for structure — and shipping it as a real service

Plan-and-execute inverts ReAct's granularity: one call asks the model to decompose the whole goal into a DAG of steps with explicit dependencies, then deterministic code — not the model — schedules and runs that DAG, executing independent nodes in parallel and replanning only when a node fails or a downstream step invalidates an assumption. You pay one extra planning call up front and get three things ReAct can't offer: the plan is inspectable before anything executes, independent branches run concurrently instead of serially, and a human (or an approval gate) can review the DAG before the agent touches anything irreversible.

The plan itself is the clearest instance yet of Module 00's typed proposal boundary: the model proposes a plan as structured output; your scheduler disposes — validates it against a schema, checks it's acyclic, and only then runs it. A plan that fails validation never executes; it triggers a re-prompt with the validation error, which is a far cheaper failure than a scheduler crash mid-run.

Layer 1 — agent runtime: typed plan + dependency-respecting scheduler (Python)

# pip install anthropic pydantic
from pydantic import BaseModel, Field
from typing import Literal
from concurrent.futures import ThreadPoolExecutor, wait
import anthropic, json

class PlanNode(BaseModel):
    id: str
    action: Literal["read_file", "grep", "summarize"]
    args: dict
    depends_on: list[str] = Field(default_factory=list)

class Plan(BaseModel):
    goal: str
    nodes: list[PlanNode]

client = anthropic.Anthropic()

PLAN_SYSTEM = """Decompose the goal into a DAG using only read_file, grep,
summarize. Respond with JSON matching:
{"goal": str, "nodes": [{"id": str, "action": str, "args": object, "depends_on": [str]}]}"""

def propose_plan(goal: str) -> Plan:
    resp = client.messages.create(
        model="claude-sonnet-5",
        max_tokens=1024,
        system=PLAN_SYSTEM,
        messages=[{"role": "user", "content": goal}],
    )
    raw = json.loads(resp.content[-1].text)
    return Plan.model_validate(raw)          # typed proposal boundary — raises, never "hopes"

ACTIONS = {
    "read_file": lambda a: open(a["path"], encoding="utf-8").read()[:20_000],
    "grep": lambda a: "\n".join(l for l in open(a["path"]) if a["term"] in l),
    "summarize": lambda a: a["text"][:500],
}

def run_plan(plan: Plan) -> dict[str, str]:
    done: dict[str, str] = {}
    remaining = {n.id: n for n in plan.nodes}
    with ThreadPoolExecutor(max_workers=4) as pool:
        while remaining:
            ready = [n for n in remaining.values() if all(d in done for d in n.depends_on)]
            if not ready:
                raise RuntimeError(f"cycle or unmet dependency: {list(remaining)}")
            futures = {pool.submit(ACTIONS[n.action], n.args): n for n in ready}
            for f in wait(futures)[0]:
                node = futures[f]
                try:
                    done[node.id] = f.result()
                except Exception as e:            # a node failure triggers replanning, not a crash
                    done[node.id] = f"ERROR: {e}"
                del remaining[node.id]
    return done

plan = propose_plan("Find every TODO under src/ and summarize the top 3.")
results = run_plan(plan)

Layer 2 — API/transport: streaming plan execution over SSE (Python, FastAPI)

Same process, same language deliberately — a separate transport hop between planner and scheduler is a Module 9 concern (harness boundaries), not this one. FastAPI just wraps Plan and run_plan from Layer 1 in a streaming endpoint.

# pip install fastapi uvicorn sse-starlette
from fastapi import FastAPI
from sse_starlette.sse import EventSourceResponse
import asyncio, json

app = FastAPI()
GOALS = {"todo-sweep": "Find every TODO under src/ and summarize the top 3."}

@app.get("/plans/{plan_id}/events")
async def plan_events(plan_id: str):
    async def stream():
        plan = propose_plan(GOALS[plan_id])
        remaining = {n.id: n for n in plan.nodes}
        done: dict[str, str] = {}
        yield {"event": "plan", "data": json.dumps(plan.model_dump())}
        while remaining:
            ready = [n for n in remaining.values() if all(d in done for d in n.depends_on)]
            for node in ready:
                yield {"event": "node_status", "data": json.dumps({"id": node.id, "status": "running"})}
                done[node.id] = await asyncio.to_thread(ACTIONS[node.action], node.args)
                del remaining[node.id]
                yield {"event": "node_status", "data": json.dumps({"id": node.id, "status": "done"})}
        yield {"event": "complete", "data": "{}"}
    return EventSourceResponse(stream())

The fourth layer — data/infra — is one table: persist each node's status and result so a killed process resumes instead of restarting the DAG from scratch (create table plan_nodes (plan_id uuid, node_id text, status text check (status in ('pending','running','done','failed')), result jsonb, updated_at timestamptz default now(), primary key (plan_id, node_id))). Module 9 builds the full checkpoint/resume harness on top of that table; here it's enough to know the scheduler's writes and the SSE stream's reads both go through it, so a reconnecting client sees consistent state instead of racing the scheduler.

Failure modes — plan-and-execute
  1. Stale plans. Node 6 assumed a file existed that node 2 was supposed to create, but node 2 failed silently upstream of validation. Symptom in the trace: a node succeeds against inputs that don't match what an earlier node actually produced. Mitigation: nodes consume prior results, not the original plan's assumptions — re-check preconditions at execution time, not just at planning time.
  2. Over-eager parallelism. Two "independent" nodes both write to the same file or call a rate-limited API simultaneously. Symptom: intermittent failures that don't reproduce serially. Mitigation: the scheduler needs a resource-conflict model, not just a data-dependency graph.
  3. Replan thrashing. One flaky node triggers a full replan, which regenerates a slightly different DAG, which fails on a different node, forever. Symptom: rising turn/cost counters with no forward progress across "plan" events. Mitigation: cap replan attempts and fall back to a narrower manual-review path, exactly like Module 00's turn-cap guard.
Principal's Judgment Call — plan-and-execute or ReAct?

Reach for plan-and-execute when the task has real structure (steps are knowable, some are parallelizable, or a human needs to approve the shape before anything irreversible runs) and the planning call's cost is small relative to the work it schedules. Stay with ReAct when discovery is the task — you don't know what step 2 needs until step 1's observation comes back, and forcing an upfront plan just means paying for a plan you'll immediately replan.

The tell in review: if you can sketch the DAG on a whiteboard without running anything, plan-and-execute is strictly better — you get parallelism and an approval gate for free. If sketching it requires "well, it depends what we find," you're describing ReAct and dressing a plan up won't fix that.

§ 1.3 Reflection loops and error lamination

Reflection (also called self-refine or, when it persists critiques across attempts, Reflexion) adds a critic step inside a single agent: generate a draft, ask the same or a second call to critique it against the task, and retry incorporating the critique — capped at a fixed number of rounds. This is not the same mechanism as Module 00's evaluator-optimizer workflow: there, code owns the generate↔evaluate loop and the evaluator is architecturally separate. Reflection is what happens when that pattern gets folded into a single agent's own reasoning, and it inherits a failure mode evaluator-optimizer's separation was designed to avoid.

messages = [{"role": "user", "content": task}]
for attempt in range(MAX_REFLECTIONS):
    draft = generate(messages)
    critique = generate(messages + [draft, {"role": "user", "content": CRITIQUE_PROMPT}])
    if "APPROVED" in critique:
        break
    messages += [draft, {"role": "user", "content": critique}]   # every failed attempt stays in context

That last line is error lamination: every rejected draft and its critique stays in context rather than being discarded, so context grows layer by layer with failed attempts. Two consequences compound. First, cost and latency grow with every round even though only the final draft matters — you're paying full inference cost to carry dead drafts forward. Second, and more insidious, the model can anchor on its first framing: round 3's critique is conditioned on round 1's mistaken approach still sitting in context, so revisions polish a wrong direction instead of abandoning it. Reflection improves outputs when the critique is genuinely orthogonal information (a different model, a rubric, a tool-verified check); it plateaus or degrades when the critic is the same model marking its own homework with no new signal.

§ 1.4 Extended thinking and test-time compute: when it buys accuracy

Extended thinking spends additional tokens on a visible reasoning pass before the model commits to a final answer or tool call — more test-time compute traded for accuracy on tasks that benefit from working through steps rather than pattern-matching to an answer. As of this writing, current-generation models are moving from a fixed budget_tokens parameter toward adaptive thinking, where the model decides how much to reason per turn rather than you pre-committing a token budget:

resp = client.messages.create(
    model="claude-sonnet-5",
    max_tokens=4096,
    thinking={"type": "adaptive"},   # model decides depth; budget_tokens is the older fixed-budget form
    messages=[{"role": "user", "content": hard_reasoning_task}],
)

The mistake teams make is reaching for extended thinking first when a task underperforms. It's the most expensive lever in the reasoning toolbox and the least likely to fix the actual problem. Work the cheaper levers first, in this order: tighten the prompt; add a structured-output schema so the model can't wander in its answer format; add retrieval if the failure is "the model doesn't know this," not "the model can't reason about what it knows." Reach for extended/adaptive thinking only when your trace shows the model had the right information in context and still reasoned to a wrong conclusion — multi-step math, subtle logic, long tool-use trajectories where earlier steps must be reconciled. It adds real latency; don't spend it on retrieval's job.

§ 1.5 Verification: the typed boundary, deterministic checks, and LLM-as-judge

Every reasoning strategy above ends the same way: a probabilistic component produces something that a deterministic system must act on. The question isn't whether to verify — it's which of three verification tools fits the failure mode you're guarding against, ordered cheapest-and-most-reliable first.

Typed output boundary — the non-negotiable first gate. A schema check answers "is this even shaped correctly," and it should reject, not coerce: a plan with a missing field or an out-of-range confidence is a bug to surface, not a value to guess at.

from pydantic import BaseModel, Field, ValidationError
from typing import Literal

class Decision(BaseModel):
    action: Literal["approve", "escalate", "reject"]
    confidence: float = Field(ge=0, le=1)

try:
    decision = Decision.model_validate_json(resp.content[-1].text)
except ValidationError:
    decision = Decision(action="escalate", confidence=0.0)   # malformed output escalates, never proceeds silently
// npm install zod   (v4 — folds validation errors into a single `error` param)
import { z } from "zod";

const Decision = z.object({
  action: z.enum(["approve", "escalate", "reject"]),
  confidence: z.number().min(0).max(1),
});

const parsed = Decision.safeParse(JSON.parse(raw));
if (!parsed.success) escalate();   // same rule: fail closed, never coerce

LLM-as-judge — the last resort, for what schemas structurally can't check. "Is this summary faithful to the source" is not expressible as a Pydantic constraint. A judge call earns its keep only when calibrated against human labels on a golden set (Module 11) and given a narrow rubric, not "rate this 1–10" — vague rubrics reproduce the judge's own biases (position bias toward the first option shown, self-preference when judge and generator share a model family) rather than measuring the thing you care about.

Principal's Judgment Call — don't reach for a judge first

Every verification problem should try deterministic checks before an LLM judge, not after. If you can write the check as code — a regex, a range, a foreign-key lookup, an allowlist — write it as code; it's free, instant, and doesn't hallucinate its own verdict. Reserve LLM-as-judge for genuinely subjective axes: tone, faithfulness to a source, whether an explanation would satisfy a human reader.

The anti-pattern I see most in reviews: teams ship an LLM judge to check something that was always a deterministic property — "did the agent cite a file that exists" is os.path.exists, not a rubric. An LLM judge for a checkable fact is strictly worse than the check: slower, costs money, and can be wrong in ways a unit test can't.

§ 1.6 Lab — same task, three reasoning strategies, evidence-backed pick

LAB 01 · Estimated 3–5 hours

Mission: take Module 00's loopbot task family (answer a question about a real repo) and implement it three ways — ReAct (you already have this), plan-and-execute (§1.2's scheduler), and reflection (§1.3) — then measure and argue for a winner with numbers, not vibes.

Spec:

  1. Pick one task class with real structure, e.g. "find every function matching pattern X across the repo, summarize what each does, and flag the two most likely to be dead code." Run it as ReAct, as plan-and-execute, and as reflection-wrapped-ReAct (one critique round).
  2. Instrument all three identically: total tokens, wall-clock latency, number of tool calls, and an accuracy score against a golden answer you write by hand for 5 repos.
  3. Wire the plan-and-execute variant to §1.2's SSE endpoint and a minimal HTML page (or the React component as-is) so a run is observable live, not just in logs.

Acceptance criteria:

  • ✅ All three strategies run against the same 5 repos and produce a table: strategy × (accuracy, cost, p50 latency).
  • ✅ curl -N http://localhost:8000/plans/todo-sweep/events streams node_status events you can watch land in order, including at least one node that ran in parallel with another (timestamps overlap).
  • ✅ A written verdict, 5–8 sentences, naming which strategy you'd ship for this task class and citing the specific numbers that justify it — not "plan-and-execute felt more organized."
  • ✅ At least one induced failure per strategy, captured in the transcript: force a ReAct error spiral, a plan-and-execute cycle/unmet-dependency error, and a reflection round that laminates onto a bad first draft.

Stretch: add adaptive thinking to the plan-and-execute variant's planning call only, and re-measure. Does thinking move the needle on plan quality, or just cost? That's the §1.4 judgment call, with your own evidence.

§ 1.7 Consultant's checklist

Questions to ask any team choosing a reasoning strategy
  • Can the task's steps be sketched as a DAG on a whiteboard before anything runs? If yes, why is it still ReAct?
  • Where does the model's raw output first get schema-validated — and does a validation failure fail closed (escalate) or silently coerce?
  • Is there an LLM-as-judge in the pipeline checking something that's actually a deterministic fact? Which one, and what would the equivalent unit test look like?
  • For any reflection or self-critique loop: is the critic getting genuinely new signal, or is it the same model re-reading its own laminated context?
  • Where extended/adaptive thinking is enabled, what cheaper lever (prompt, schema, retrieval) was ruled out first, and with what evidence?
  • What's the replan cap on any plan-and-execute loop, and what happens when it's hit — silent failure, or a surfaced escalation?

§ 1.8 Design-review questions (staff+)

Interview / design-review bank
  1. A team wants to add a "self-review" step where the agent critiques its own output before returning it. What do you push back on? Strong answers name error lamination and same-model self-preference bias, ask what new signal the critique step adds versus a deterministic check, and propose capping rounds with a hard budget rather than "until approved."
  2. Your plan-and-execute service's cost per task tripled after a migration, with no accuracy change. Diagnose. Strong answers suspect replan thrashing first (check the ratio of "plan" events to "complete" events per task_id), then check whether adaptive thinking got silently enabled on every planner call instead of the flagged-failure subset.
  3. Design the verification layer for an agent that files support tickets on a customer's behalf. Strong answers put schema validation first (valid ticket fields, existing customer ID), a deterministic allowlist second (can this account file this ticket type), and reserve an LLM judge only for tone/appropriateness of the ticket text — never for whether the account or ticket type is valid.
  4. When is a 200-line plan-and-execute scheduler the wrong architecture versus a graph framework with built-in checkpointing? Strong answers separate "do we need the scheduling logic" (usually no framework needed) from "do we need durable resume across process restarts" (that's the data-layer table in §1.2, and past a certain scale, a framework's checkpointing earns its keep) — not framework preference.
  5. Argue both sides: "LLM-as-judge is a crutch that teams reach for instead of building real evals." Strong answers concede it's true when judges check deterministic facts (misuse), then defend it for genuinely subjective axes when calibrated against human-labeled golden sets — landing on "judge quality is only as good as its calibration evidence," not a blanket yes/no.
