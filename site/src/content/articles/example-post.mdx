This file is not a real news article — it is a working template. Copy it,
rename the slug, fill in real content, and delete the parts you don't need.
It exists so the Astro pipeline has one real, buildable, end-to-end example
committed to the repo, proving the shared layout and components actually
render correct output rather than existing only on paper.

## What this template proves

Every article gets its SEO metadata (canonical URL, Open Graph, Twitter
card, and `BlogPosting` JSON-LD) automatically from `ArticleLayout` — an
author only fills in the front-matter fields above. That directly replaces
the old workflow, where every legacy `articles/*.html` page hand-authored
its own `<head>` and, in practice, never got a real `<meta description>`
tag at all (see `docs/technical-debt/backlog.md` #1).

<Judgment>
  Reach for a new MDX file under `site/src/content/articles/` when the piece
  is genuinely new full-stack content; keep editing the legacy
  `articles/*.html` files directly when it's a small correction to
  already-published content. Don't backport old articles into MDX just to
  "finish the migration" — do that deliberately, one article per run, only
  once duplicate-content handling (redirects or removal of the old page) is
  decided in a follow-up ADR.
</Judgment>

<FailureBlock symptom="build fails with a Zod validation error naming a front-matter field">
  The content collection schema (`site/src/content/config.ts`) is strict —
  a missing or mistyped front-matter field (e.g. `difficulty: "Advanced"`
  instead of one of the three allowed values) fails `npm run build` loudly
  instead of silently shipping a page with missing SEO data. That's the
  point: it's the same protection a type system gives application code,
  applied to content.
</FailureBlock>
