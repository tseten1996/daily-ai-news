Streaming Agent Output to the Browser: SSE, WebSockets, and the Reconnect Problem

Getting tokens on screen fast is the easy 80%. The other 20% — surviving a dropped wifi connection without duplicating or losing part of the transcript — is where most agent chat products actually break in production.

PUBLISHED 2026-07-16 DIFFICULTY Practitioner READ ~9 min (~1,750 words) LAB reconnect a live SSE stream without dropping or duplicating a token

§1 Cold open

A support ticket comes in: "the assistant froze for 25 seconds, then the whole answer appeared at once, garbled." Your local testing shows tokens streaming smoothly, one every 20–40ms. The difference is the customer's corporate network sits behind a proxy that buffers HTTP responses until it sees a full chunk worth flushing — your token stream arrives at the proxy in near-real-time and leaves it in bursts. A second ticket, same day: a user's laptop sleeps mid-generation, wifi drops for three seconds, and when it reconnects the transcript shows a sentence twice, followed by a gap where a tool call's result should be. Neither bug is in your agent. Both are in the transport layer nobody designed on purpose.

§2 The problem this solves — and what engineers did before it

Before token-level streaming was standard, agent and chatbot UIs had two options, both bad in the same way: block until the full response was ready and show a spinner, or poll a status endpoint every second or two asking "done yet?" Blocking full-response UIs make time-to-first-token indistinguishable from time-to-last-token — a 12-second agent turn with three tool calls looks identical to the user whether it's about to fail or about to succeed, and there's no way to show intermediate tool-call progress. Polling fixes the visibility problem but multiplies request volume by however many clients are waiting, forces you to persist and diff state on every poll, and still has visible latency jitter bounded by your poll interval.

Streaming protocols — Server-Sent Events (SSE) and WebSockets — collapse both problems: the server pushes each token, tool call, and status change to the client as it happens, over one long-lived connection instead of N repeated ones. That part is well understood by 2026. What's still routinely under-built is what happens when that one long-lived connection dies mid-turn — which, at any real scale, it will, constantly: mobile networks hand off, laptops sleep, corporate VPNs cycle, load balancers recycle idle connections. A streaming design that only works on the happy path isn't a streaming design; it's a demo.

§3 How it actually works

The mechanism has four moving parts across the stack, and the object that ties them together is a single typed event — {id, type: 'token' | 'tool_call' | 'done', payload} — with a monotonically increasing id per session. That id is the entire reconnect story: everything below exists to let a client say "resume after event 482" and get exactly the events it missed, no more, no fewer.

Layer 1 — agent runtime: publish events into a durable, resumable stream (Python)

# pip install redis anthropic
import redis, json

r = redis.Redis(host="localhost", port=6379, decode_responses=True)

def publish_event(session_id: str, event_type: str, payload: dict) -> str:
    """Append one event; Redis assigns a monotonic stream id we never invent ourselves."""
    stream_key = f"session:{session_id}:events"
    entry_id = r.xadd(stream_key, {"type": event_type, "payload": json.dumps(payload)})
    r.expire(stream_key, 3600)  # bound retention — see failure mode 3 below
    return entry_id

def run_turn(session_id: str, user_message: str):
    with anthropic_client.messages.stream(
        model="claude-sonnet-5",
        max_tokens=1024,
        messages=[{"role": "user", "content": user_message}],
    ) as stream:
        for event in stream:
            if event.type == "content_block_delta" and event.delta.type == "text_delta":
                publish_event(session_id, "token", {"text": event.delta.text})
            elif event.type == "content_block_start" and event.content_block.type == "tool_use":
                publish_event(session_id, "tool_call", {"name": event.content_block.name})
    publish_event(session_id, "done", {})

Layer 2 — API/transport: SSE endpoint that resumes from Last-Event-ID (Node/Express)

// npm install express ioredis
import express from "express";
import Redis from "ioredis";

const app = express();
const redis = new Redis();

app.get("/sessions/:id/stream", async (req, res) => {
  const streamKey = `session:${req.params.id}:events`;
  // Prefer the header (native reconnect); fall back to a query param for the
  // first manual connect. "0" means "from the beginning of what's buffered."
  let lastId = (req.header("Last-Event-ID") ?? (req.query.lastId as string) ?? "0") as string;

  res.setHeader("Content-Type", "text/event-stream");
  res.setHeader("Cache-Control", "no-cache");
  res.setHeader("Connection", "keep-alive");
  res.setHeader("X-Accel-Buffering", "no"); // tell nginx/most proxies: don't buffer this route
  res.flushHeaders();

  req.on("close", () => { closed = true; });
  let closed = false;

  while (!closed) {
    // Blocking read: wait up to 15s for new entries after lastId, replaying
    // anything the client missed before falling through to a live tail.
    const results = await redis.xread("BLOCK", 15000, "STREAMS", streamKey, lastId);
    if (!results) continue; // timeout, no new events — loop (keeps connection alive)
    const [, entries] = results[0];
    for (const [id, fields] of entries) {
      const [, type, , payloadJson] = fields;
      res.write(`id: ${id}\nevent: ${type}\ndata: ${payloadJson}\n\n`);
      lastId = id;
      if (type === "done") { closed = true; break; }
    }
  }
  res.end();
});

Layer 3 — frontend: EventSource with typed reconnect (TypeScript/React)

type AgentEvent =
  | { type: "token"; payload: { text: string } }
  | { type: "tool_call"; payload: { name: string } }
  | { type: "done"; payload: Record<string, never> };

function useAgentStream(sessionId: string) {
  const [transcript, setTranscript] = useState("");
  const [status, setStatus] = useState<"connecting" | "streaming" | "done" | "reconnecting">("connecting");
  const lastIdRef = useRef<string>("0");

  useEffect(() => {
    let source: EventSource;
    let retryDelay = 500; // exponential backoff, capped below

    function connect() {
      // Native EventSource replays Last-Event-ID automatically on its own
      // reconnect; we also pass it explicitly on the very first connect.
      source = new EventSource(`/sessions/${sessionId}/stream?lastId=${lastIdRef.current}`);
      setStatus((s) => (s === "connecting" ? "streaming" : "reconnecting"));

      source.addEventListener("token", (e: MessageEvent) => {
        lastIdRef.current = e.lastEventId;
        const { text } = JSON.parse(e.data) as AgentEvent["payload"] & { text: string };
        setTranscript((t) => t + text);
        retryDelay = 500; // reset backoff on any successful event
      });
      source.addEventListener("tool_call", (e: MessageEvent) => {
        lastIdRef.current = e.lastEventId;
        setTranscript((t) => t + `\n[tool: ${JSON.parse(e.data).name}]\n`);
      });
      source.addEventListener("done", () => {
        setStatus("done");
        source.close();
      });
      source.onerror = () => {
        source.close();
        setStatus("reconnecting");
        setTimeout(connect, Math.min(retryDelay *= 2, 8000)); // capped exponential backoff
      };
    }
    connect();
    return () => source?.close();
  }, [sessionId]);

  return { transcript, status };
}

Trace the object across layers: the Redis Stream assigns entry_id once (Layer 1); the SSE endpoint forwards it verbatim as the id: field (Layer 2); the browser's EventSource reads it as e.lastEventId and replays it as Last-Event-ID on reconnect (Layer 3). No layer re-derives or guesses the id — that's the whole trick, and it's also the whole failure surface if any layer breaks the chain.

§4 Failure modes

What you'll actually see in production
  1. Proxy buffering turns a stream into a burst. Symptom: local dev streams smoothly, but behind a corporate proxy or a misconfigured nginx (proxy_buffering on) or when gzip compression is applied to the route, tokens arrive in one or two large bursts instead of continuously — the trace shows the agent runtime writing at 30ms intervals while the client-observed timestamps cluster at connection-open and connection-drain. Fix: disable buffering and compression specifically on the streaming route (X-Accel-Buffering: no, exclude the path from gzip), and confirm with a raw curl -N against production, not just localhost.
  2. Reconnect without replay loses or duplicates events. Symptom: after any network blip, the transcript has a silent gap (events published during the gap were never delivered because the new connection started a fresh tail instead of resuming) or a duplicated sentence (the client re-rendered from scratch instead of appending from Last-Event-ID). This is what happens when a team ships SSE without the Redis Stream (or equivalent durable buffer) behind it — the events only ever exist in the socket, so once the socket closes, they're gone, and "resume" is a UI feature with no backing data.
  3. Backpressure from slow clients grows the agent runtime's memory. Symptom: a backgrounded mobile tab stops reading its SSE connection, but the server keeps generating and buffering events for it; multiply by thousands of idle tabs and worker RSS climbs until the process is OOM-killed. The Redis Stream absorbs this for *reconnect* (bounded by your XTRIM/expiry policy) but does not absorb it for a live, open-but-stalled connection unless your API layer's write loop itself yields or times out slow consumers — the fix is a write deadline per connection, not more memory.
  4. WebSocket sticky-session mismatch after a deploy or scale-down. Symptom: a client reconnects and the load balancer routes it to a different pod that never held that session's in-memory state; the connection opens, then errors on the first message because there's no local WebSocket-specific session to resume against. SSE sidesteps this by design if the durable state lives in Redis rather than in-process — any pod can serve the resume because the state isn't pinned to the one that opened the original connection.
Principal's Judgment Call — SSE vs. WebSockets

Reach for SSE when the traffic is fundamentally one-directional (server pushes tokens, tool status, progress) and you want to inherit ordinary HTTP semantics for free: auth cookies just work, HTTP/2 multiplexes it cheaply, and every proxy, CDN, and load balancer in your stack already understands "long-lived GET response" without special-casing a new protocol. Reach for WebSockets when you need genuinely low-latency client-to-server signals inside the same connection — a mid-generation cancel that must land in milliseconds, multi-user presence, or multiplexing many independent logical channels over one socket where opening N SSE connections would be wasteful.

For most agent chat products, the pragmatic shape is SSE for output plus a plain POST /sessions/:id/cancel for interrupt — it's simpler to operate, easier to debug with curl, and doesn't require you to reinvent reconnect semantics that HTTP already gives you. Don't reach for WebSockets because streaming "feels more real-time"; reach for it only once you've hit a concrete limit SSE can't clear, like sub-100ms bidirectional turn-taking.

§5 Try it in 20 minutes

LAB · Estimated 20 minutes

Mission: stand up the three layers above against a local Redis, then prove the reconnect path actually works — not just the happy path.

  1. Run redis-server locally. Implement publish_event from Layer 1 as a standalone script that pushes 10 fake token events one per second into session:demo:events, followed by a done event.
  2. Implement the Express endpoint from Layer 2 exactly as shown, pointed at that stream key.
  3. In one terminal: curl -N "http://localhost:3000/sessions/demo/stream" and watch events arrive roughly once per second with visible id: lines.
  4. Re-run the publisher script, then mid-stream (after ~4 events), kill the curl with Ctrl-C, note the last id: printed, and reissue: curl -N -H "Last-Event-ID: <that id>" "http://localhost:3000/sessions/demo/stream".
  5. Wire the React hook from Layer 3 to the same endpoint and confirm the transcript renders live and survives a manual Wi-Fi toggle without a gap or duplicate.

Acceptance criteria:

  • ✅ The first curl -N shows events arriving incrementally, not in one burst.
  • ✅ The reconnected curl with Last-Event-ID resumes from the next event after the one you last saw — no repeated event, no skipped one.
  • ✅ Toggling Wi-Fi in the browser mid-stream produces a visible "reconnecting" status in the UI and the transcript continues without a duplicated sentence.

§6 Design-review question

Staff+ interview / design-review bank

Your agent chat product needs to support a user cancelling generation mid-turn, and product wants it to feel instant. You're on SSE for output. How do you add cancellation, and does it change your transport choice?

Strong answer sketch: cancellation is client-to-server, which SSE doesn't carry — but that doesn't require switching transports, because SSE and the interrupt channel don't have to be the same connection. Add a plain POST /sessions/:id/cancel that sets a cancellation flag (e.g. a Redis key with a short TTL) the agent runtime's generation loop checks between tokens/tool calls; the SSE stream simply stops emitting once the loop observes the flag and publishes a final cancelled event so the UI can distinguish "cancelled" from "errored" from "completed." The only latency cost is the polling granularity of the runtime's cancellation check (checking once per token is effectively instant; checking once per tool call adds up to one tool's latency). This resolves the "feels instant" requirement without WebSockets, and it keeps the resume protocol from §3 completely unaffected — cancellation is just another event type in the same stream, with its own id, so a reconnecting client sees it like any other event. Switching the whole output channel to WebSockets to get a cancel button is treating a one-bit signal as if it required a bidirectional protocol; it doesn't.

AS-OF 2026-07-16 · Stable: the SSE wire format and Last-Event-ID reconnect mechanics have been unchanged since the spec's original publication and are not expected to move. Verify before shipping: native EventSource cannot set custom request headers (no bearer-token auth on the initial connect) — most production setups pass the session/auth token as a signed query parameter or rely on same-site cookies, and any fetch-based SSE polyfill you adopt for header support should be checked against its current maintenance status.
