The No-Framework Option: A 200-Line Harness You Fully Own

The bug was four lines wide. Finding it took a week inside the framework's checkpointer, its reducer semantics, and the superstep boundary that decides when partial state commits. A hand-rolled loop wouldn't have hidden the bug behind machinery nobody on the team had needed to read before that week.

PUBLISHED 2026-07-19 DIFFICULTY Foundations READ ~9 min (~1,650 words) LAB run a minimal harness, crash it mid-tool-call, resume without re-executing

§1 Cold open

Three weeks into debugging a stuck agent, the on-call engineer finds the bug: a tool call gets silently dropped whenever two graph nodes race to write the same state key. The defect itself is four lines wide. Finding it meant first learning the framework's checkpointer, its reducer semantics, and the superstep boundary that decides when a partial state actually commits — a debugging detour longer than writing the whole agent loop from scratch would have taken. Nobody on the team had ever called checkpointer.put() directly before that week. Nobody had needed to, until the framework's own internals became the thing standing between them and their bug.

§2 The problem this solves — and what engineers did before it

Before agent frameworks, teams called a chat completions API directly inside a hand-written loop: send messages and tool schemas, get a response, dispatch any tool calls, append the results, repeat. Every team reinvented persistence, retries, and multi-step tool dispatch from scratch, which is real, wasted effort. LangChain and its successors — LangGraph, CrewAI, AutoGen — genericized those primitives and added real capability on top: graph-based orchestration, built-in checkpointing with replay, and a wide connector ecosystem.

That capability has a cost, and Anthropic's own December 2024 field notes on the subject make the opposite-sounding argument from the same evidence: across their customer engagements, the most successful implementations used simple, composable patterns rather than frameworks, because a framework adds a layer of abstraction between you and the raw prompts and responses — precisely the layer this article's cold open got lost inside. Both things are true at once. A framework buys real capability at a real abstraction tax; for one agent with a bounded, known tool set, that tax routinely costs more than the loop it replaces would have.

§3 How it actually works

A harness, stripped to its mechanism, is a loop with four moving parts: call the model with the running message list and tool schemas; if the response contains no tool-use blocks, you're done; if it does, run each proposed call through a policy gate before touching the real world, execute the ones that pass, and append the results as the next turn; then persist the message list before looping again. "200 lines" isn't the loop itself — that's under 30 — it's the loop plus a tool registry, the checkpoint calls, and basic error handling, all of it code your team wrote and can read top to bottom in an afternoon.

Layer C — agent runtime: the loop itself (Python)

import anthropic, json

client = anthropic.Anthropic()
MAX_ITERATIONS = 12  # cost governor / loop-detection backstop — see §4.1

def run_harness(session_id, messages, tools, tool_registry, hook, checkpoint):
    for step in range(MAX_ITERATIONS):
        response = client.messages.create(
            model="claude-sonnet-5", max_tokens=4096,
            messages=messages, tools=tools,
        )
        messages.append({"role": "assistant", "content": response.content})

        tool_uses = [b for b in response.content if b.type == "tool_use"]
        if not tool_uses:
            checkpoint(session_id, step, messages, status="complete")
            return response

        results = []
        for call in tool_uses:
            decision = hook(call.name, call.input)   # the PreToolUse chokepoint
            if decision == "deny":
                results.append({"type": "tool_result", "tool_use_id": call.id,
                                 "content": "denied by policy", "is_error": True})
                continue
            output = tool_registry[call.name](**call.input)
            results.append({"type": "tool_result", "tool_use_id": call.id, "content": output})
        messages.append({"role": "user", "content": results})
        checkpoint(session_id, step, messages, status="in_progress")  # every step,
                                                                        # not just the end

    raise RuntimeError(f"session {session_id} exceeded {MAX_ITERATIONS} iterations")

Layer D — data/infra: a checkpoint table that survives a crash (SQL + Python)

-- CREATE TABLE agent_checkpoints (
--   session_id TEXT NOT NULL,
--   step       INT  NOT NULL,
--   status     TEXT NOT NULL,        -- in_progress | complete
--   messages   JSONB NOT NULL,
--   updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
--   PRIMARY KEY (session_id, step)
-- );

def checkpoint(session_id, step, messages, status):
    db.execute(
        """INSERT INTO agent_checkpoints (session_id, step, status, messages)
           VALUES (%s, %s, %s, %s)
           ON CONFLICT (session_id, step) DO UPDATE
             SET status = EXCLUDED.status, messages = EXCLUDED.messages, updated_at = now()""",
        (session_id, step, status, json.dumps(messages)),
    )

def resume(session_id):
    row = db.query_one(
        "SELECT step, messages FROM agent_checkpoints WHERE session_id=%s ORDER BY step DESC LIMIT 1",
        (session_id,),
    )
    return (row["step"], row["messages"]) if row else (0, [])

Wiring this to a frontend is a thin API route: POST /sessions/{id}/messages calls resume(), appends the new user turn, calls run_harness(), and returns the result — or, for a chat UI, streams it the way Streaming Agent Output to the Browser already covers. That article's SSE and reconnect plumbing sits on top of this loop's step-by-step checkpoint writes, not instead of them.

§4 Failure modes

What you'll actually see in production
  1. No iteration cap. Symptom: a cost or token-usage spike traced to one session, with a trace showing a dozen-plus execute_tool spans calling the same tool with near-identical arguments — the model stuck re-proposing a call that never satisfies whatever it's checking for. §3's MAX_ITERATIONS exists precisely to turn that into a bounded, loud failure instead of an unbounded, silent one.
  2. Checkpointing after the tool executes instead of before. Symptom: a crash mid-step, then a resumed session that re-executes a tool call already reflected in the real world — a duplicate charge, a duplicate ticket. This is exactly the gap Tool Design as API Design's idempotency keys exist to close; a harness without them turns "resume from checkpoint" into "replay side effects."
  3. No single chokepoint for policy. Symptom: a new compliance rule ships as a patch scattered across every function in tool_registry instead of one gate, and the next audit finds one call site where it was missed. Hooks as Deterministic Guardrails is built entirely on the hook() call in §3 already existing — retrofit it later and every policy change becomes a hand-audited sweep instead of one edit.
  4. An unregistered tool name crashes the loop. Symptom: an unhandled KeyError in the logs, the whole session lost, because the crash happens between the model's response and the next checkpoint() call. A model calling a tool name that drifted out of the registry (renamed, removed) is routine, not exotic — the loop needs to catch it and checkpoint an error state, not fall over.
Principal's Judgment Call — hand-rolled harness vs. a framework

Reach for the hand-rolled harness when you have one agent, a bounded and largely known tool set, and a team that wants full visibility into retries, checkpointing, and cost governance — the code in §3 is the whole surface area you have to reason about when something breaks, and there's no framework release to track or upgrade around.

Reach for a framework — LangGraph, CrewAI, or similar — when you need built-in multi-agent graph orchestration, a wide library of prebuilt connectors, or your team doesn't have the bandwidth to own retry, checkpoint, and cost-governor code itself and is knowingly trading the abstraction tax for velocity. That tax is real, not hypothetical: it's the cold open. Pay it on purpose, for a capability you actually need, not by default.

§5 Try it in 20 minutes

LAB · Estimated 20 minutes

Mission: run the §3 harness with one real tool, crash it mid-session on purpose, and prove resuming from the checkpoint doesn't replay a tool call that already ran.

  1. Set up Postgres (or SQLite for a quicker path) and create the agent_checkpoints table from §3's DDL comment.
  2. Wire tool_registry = {"get_time": lambda: str(datetime.now())} and a hook that always returns "allow". Wrap run_harness in a one-route script: curl -X POST localhost:8000/sessions/s1/messages -d '{"text":"what time is it?"}' should return a final answer.
  3. Confirm a row exists in agent_checkpoints for session_id='s1' with status='complete'.
  4. Force a crash: add a second turn ("now add 100 to that") but kill the process (Ctrl-C) right after the tool executes and before the final checkpoint() call — comment out that one line temporarily to simulate it reliably.
  5. Call resume("s1") and re-run the loop from the returned step. Confirm the harness produces a final answer without calling get_time again — the resumed messages list already contains that turn's tool result.

Acceptance criteria:

  • ✅ The curl call in step 2 returns a model-generated final answer, not raw tool output.
  • ✅ Step 3's row exists with the full message history in messages — inspectable with SELECT messages FROM agent_checkpoints WHERE session_id='s1'.
  • ✅ Step 5 resumes without a second execute_tool span for get_time — confirmed by adding a print statement inside the tool function and observing it fire exactly once across both runs.

§6 Design-review question

Staff+ interview / design-review bank

A teammate proposes adopting LangGraph on day one for a single-agent internal support bot with three tools, arguing "we might need multi-agent later." What's the right call?

Strong answer sketch: build the §3-style harness now. The complexity a framework brings isn't avoided by adopting it early, it's deferred — you pay the abstraction tax (learning the state-graph model, debugging through its checkpointer) starting today, for a multi-agent capability you don't have a concrete use for yet. For three tools and one agent, that tax exceeds what a well-isolated ~200-line loop costs to run and debug. The "might need it later" risk is also smaller than it sounds: migrating a harness into a framework later is a bounded, mechanical port — wrap the existing tool registry, drop the loop, keep the checkpoint schema — as long as the tool-registry and checkpoint boundaries were kept clean from the start, which is a cheap discipline to hold onto regardless of which path you're on. Adopt the framework the day you actually have two coordinating agents, not in anticipation of maybe having them.
