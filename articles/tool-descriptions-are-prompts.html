Tool Descriptions Are Prompts: Why the Schema Field Nobody Reviews Controls Tool Selection

A tool's description field looks like documentation. To the model, it's tokens in the context window, conditioning the exact same decision a system prompt does — which is why an "improved" sentence that never touches a type or a required field can silently reroute production traffic overnight.

PUBLISHED 2026-07-22 DIFFICULTY Practitioner READ ~9 min (~1,700 words) LAB golden-set eval a description change and prove the accuracy delta over curl

§1 Cold open

A support-tooling engineer tightens a tool's description from "Process a refund for an order" to "Use this for any refund-related request, including disputes about being double-charged." The edit ships inside an unrelated PR that adds a response field to the same tool — nobody flags the one-line description change in review, because it isn't code, it's a string. Two days later the on-call trace dashboard shows refund_order firing against password-reset tickets and billing-address-change tickets, anywhere a customer's message happens to contain the word "charged." Nothing in the code path changed. The model started routing on the new sentence's vocabulary instead of the account's actual state, and not one eval had run against the new wording before it reached production.

§2 The problem this solves — and what engineers did before it

JSON Schema's description field was, for years, treated the way a docstring is treated: helpful for a human skimming an API spec, inert with respect to program behavior. Early function-calling systems reinforced that assumption — with two or three tools in a prompt, ambiguity rarely surfaced, so a slightly sloppy sentence cost nothing. Type constraints, required fields, and enums were "the contract" (see Tool Design as API Design for that half of the story); the sentence explaining what a tool was for got polished by whoever last touched the file, with no review weight, no version pin, and no test.

That assumption breaks once agents scale past a handful of tools, often built across separate MCP servers owned by separate teams. The description field stops being documentation about the tool and becomes the primary signal the model uses to decide which tool to invoke at all — it is literally text injected into the model's context immediately before it emits a tool_use block, no different in kind from any other prompt the model conditions on. Teams that had already learned to gate system-prompt edits behind eval suites — because a chat product's system prompt is obviously a prompt — didn't apply the same discipline to tool descriptions, because those live in a schema file, not a prompts/ directory. The wording slipped under a review process that already existed, just aimed at the wrong file.

§3 How it actually works

Treat a tool's description exactly like a system-prompt change: version it, eval it against a golden set of paraphrased user requests before it ships, and gate the merge on a measured accuracy delta — not a "reads fine to me" comment. That spans four layers: the MCP server that serves the description text and a version tag for it, the agent runtime that assembles the description into the model's context and observes which tool actually gets picked, a golden-set eval store that turns "did this wording work" into a number, and a review surface where a human sees the diff and the accuracy delta side by side before promoting it.

Layer 2 — API/transport: an MCP server that versions its own descriptions (TypeScript)

// Each tool's wording carries a version tag in MCP's generic `_meta`
// extension point, so the eval harness below can say exactly which
// wording produced which accuracy number.
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { z } from "zod";

const DESCRIPTIONS = {
  cancel_subscription: {
    version: "v3",
    text: "Permanently end a customer's subscription at the end of the " +
          "current billing period. Do not use this for a temporary pause " +
          "— use pause_subscription instead.",
  },
  pause_subscription: {
    version: "v3",
    text: "Temporarily suspend billing and access for 1-3 months while " +
          "keeping the subscription record intact. Do not use this to end " +
          "the subscription permanently — use cancel_subscription instead.",
  },
};

const server = new McpServer({ name: "billing-tools", version: "1.4.0" });

for (const [name, { version, text }] of Object.entries(DESCRIPTIONS)) {
  server.registerTool(
    name,
    {
      description: text,
      inputSchema: { customer_id: z.string() },
      _meta: { descriptionVersion: version },
    },
    async ({ customer_id }) => handlers[name](customer_id)
  );
}

Layer 3 & 4 — agent runtime + data: golden-set eval, scored and stored (Python)

# Run in CI on every PR that touches a tool description. Fails the build
# if accuracy regresses versus the currently-deployed description_hash.
import anthropic, psycopg, hashlib, json

client = anthropic.Anthropic()

GOLDEN_SET = [
    ("I want to stop my subscription for good, I'm done with this service", "cancel_subscription"),
    ("Cancel my plan, I'm switching to a competitor", "cancel_subscription"),
    ("Can I skip the next couple months but keep my account?", "pause_subscription"),
    ("I'm traveling for 8 weeks, can you freeze billing until I'm back?", "pause_subscription"),
    # ... 6 more paraphrases per tool in the real set
]

def evaluate(tools: list[dict]) -> dict:
    description_hash = hashlib.sha256(
        json.dumps([t["description"] for t in tools]).encode()
    ).hexdigest()[:16]

    results = []
    for prompt, expected in GOLDEN_SET:
        resp = client.messages.create(
            model="claude-sonnet-5",
            max_tokens=256,
            tools=tools,
            messages=[{"role": "user", "content": prompt}],
        )
        called = next((b.name for b in resp.content if b.type == "tool_use"), None)
        results.append({"prompt": prompt, "expected": expected, "got": called, "hit": called == expected})

    accuracy = sum(r["hit"] for r in results) / len(results)
    with psycopg.connect() as conn:
        conn.execute(
            "INSERT INTO eval_runs (description_hash, accuracy, results) VALUES (%s, %s, %s)",
            (description_hash, accuracy, json.dumps(results)),
        )
    return {"description_hash": description_hash, "accuracy": accuracy, "results": results}

Notice what the panel gates: only the description string. A change to inputSchema's types or required fields is a different kind of change — it belongs behind the versioning discipline in Tool Design as API Design, not this eval gate, because a schema change breaks callers structurally where a wording change breaks them statistically.

§4 Failure modes

What you'll actually see in production
  1. Silent behavior drift from a "helpful" wording edit. Symptom: this article's cold open — a tool-call distribution graph shows one tool's call rate jump, correlated with a deploy timestamp that touched only the description string, while the code path, model, and input schema are all unchanged. If your alerting only watches error rate, this is invisible: nothing throws, the tool just gets called on inputs it shouldn't have, and returns a technically successful result for a request it was never meant to handle.
  2. Kitchen-sink descriptions as the tool count grows. Symptom: golden-set accuracy degrades measurably once live tool count crosses roughly a dozen, even though no individual description changed — a newly added tool's paragraph-length description ate context budget and pushed the disambiguating detail in an older tool's description further from the point where the model decides, and ties increasingly resolve toward whichever tool was declared last, a positional bias that worsens as you add tools, not improves.
  3. Overlapping descriptions between adjacent tools. Symptom: two paraphrases of the same underlying intent — "cancel my plan" versus "I'm done with this service" — route to different tools close to half the time, and your golden-set accuracy plateaus around 60-70% no matter how many times you rerun the eval, because the ambiguity lives in wording both tools share, not in any single request.
  4. An untrusted description as an injection surface. Symptom: an audit of tool-call payloads turns up a free-text or optional argument on an unrelated tool populated with content that never appeared anywhere in the user's conversation, because a third-party MCP server's description contained an embedded instruction the model treated as legitimate context. This is the sharp end of failure mode 1: if description text steers which tool gets picked, it can steer the arguments passed to it too, and a description you didn't author is a description you haven't reviewed.
Principal's Judgment Call — full eval-gate pipeline vs. PR-comment review

Reach for the full pipeline — versioned descriptions, a golden-set eval gate, a review panel that blocks merges on regression — once your tool count passes roughly a dozen, more than one team authors tool descriptions, or a tool's blast radius is expensive to misroute into (the refund and subscription tools here both qualify: a wrong call either moves money or promises a customer something support never intended). That's the point past which "someone will notice a bad description in code review" stops being true, because nobody scrutinizes a prose edit the way they scrutinize a type signature.

Stay with lightweight, PR-comment-level review when you own a small, single-team toolset — under about eight tools — where a misroute is cheap to notice and cheap to undo, like a search tool returning irrelevant-but-harmless results. Building golden-set CI for that toolset costs more engineering time than the failures it would ever catch; the eval infrastructure earns its keep only once the cost of a silent misroute exceeds the cost of building the gate.

§5 Try it in 20 minutes

LAB · Estimated 20 minutes

Mission: wrap the eval harness from §3 in a small HTTP endpoint, prove over curl that an ambiguous pair of tool descriptions scores badly, then fix the wording and prove the same endpoint now scores perfectly.

  1. Define two tools, cancel_subscription and pause_subscription, with deliberately overlapping first-draft descriptions (e.g. both saying only "manage a customer's subscription").
  2. Build a 10-prompt golden set: 5 paraphrases that should map to each tool (reuse the four examples in §3's code and extend them).
  3. Wrap the evaluate() function from §3 in a minimal FastAPI app with a single route: POST /eval/run, returning the JSON from evaluate().
  4. Run it: curl -X POST localhost:8000/eval/run. Record the accuracy field and note which prompts missed.
  5. Rewrite both descriptions using the disambiguating language from the §3 TypeScript snippet ("temporary" vs. "permanent," each pointing at the other tool by name).
  6. Restart the server and fire the exact same curl command again.

Acceptance criteria:

  • ✅ The first curl call returns accuracy at or below 0.7, with at least one entry in results where got != expected.
  • ✅ The second call, after the wording fix and a server restart, returns accuracy equal to 1.0 with no mismatched entries.
  • ✅ Both responses include the per-prompt breakdown (prompt, expected, got), not just the aggregate score — the reviewer needs to see which paraphrase flipped, not just that something did.

§6 Design-review question

Staff+ interview / design-review bank

Your team wants to let a support-ops PM — not an engineer — tweak tool descriptions directly in a no-code admin panel, to improve routing accuracy without waiting on a PR. What guardrails would you put in place, and what would you refuse to allow through that panel?

Strong answer sketch: description text is unreviewed prompt injection into the model's decision boundary regardless of who edits it, so every save through the panel has to go through the exact same gate a PR would — the panel's "Save" button calls the golden-set eval pipeline from §3, it doesn't bypass it. A save that regresses accuracy lands in a blocked state, same as the frontend snippet in §3 shows for a code reviewer; the PM sees the same diff-plus-accuracy-delta view an engineer would. What the panel should refuse outright is any edit to the inputSchema — types, required fields, enums — because that's a structural contract change belonging to the versioning discipline in Tool Design as API Design, not a wording tweak, and a PM editing it without engineering review reintroduces exactly the schema-drift failure that article describes. The panel should also never auto-publish above some blast-radius threshold: a wording change to a read-only search tool can promote on eval-pass alone, but a change to a tool that moves money or deletes state should still require a human engineer's sign-off in addition to a passing eval, mirroring the tool-call approval-surface pattern one layer up — from gating a tool call to gating a tool definition.
