Orchestrator-Workers and the Coordination Tax

Fanning a task out to N worker agents doesn't just divide the work by N — it adds a bill for keeping those workers from stepping on each other. Skip paying it deliberately and you pay it anyway, in silently lost results.

PUBLISHED 2026-07-20 DIFFICULTY Practitioner READ ~10 min (~1,900 words) LAB prove atomic task claiming under concurrency and reclaim a stuck task after timeout

§1 Cold open

A platform team kicks off an overnight job: audit 40 microservices for hardcoded secrets, one worker agent per service, orchestrated to run in parallel so the whole sweep finishes before standup instead of taking all night sequentially. Each worker reads its service's source, writes its findings as an appended block to a shared audit_findings.md in object storage, and exits. The next morning the document has 31 entries, not 40. No worker logged an error; no task shows as failed. What happened is a lost update: nine workers finished inside the same two-second window, and the orchestrator's "append a finding" step was read the whole document, add my paragraph, write the whole document back — not a true append. Whichever worker in each finishing cluster wrote last overwrote the document state the others had just written into, discarding their findings along with it. Nothing in the architecture ever noticed two workers were mid-flight on the same read-modify-write at once, because nothing was watching for it.

§2 The problem this solves — and what engineers did before it

The instinct once an agent workload can be split — "summarize each of these 40 services," "review each of these 200 files" — is to spawn one worker per unit of work and let them run concurrently, because wall-clock time is the whole point of not doing it sequentially. Anthropic's own taxonomy names this the orchestrator-workers pattern: a central agent decomposes a task and dispatches the pieces to workers, then synthesizes their outputs. What that framing doesn't spell out is that workers touching any shared resource — a shared output document, a shared task list, a shared rate-limited API, a shared budget — turns "independent parallel work" into a concurrency problem indistinguishable from any other distributed system's, whether or not an LLM is involved anywhere in it.

Before teams built real coordination, they mostly avoided the problem by construction: split work into a small, fixed number of shards known in advance, hardcode shard N to worker N, and never let two workers write to the same place. That works and it's simple, right up until the task count isn't known ahead of time (you're processing "every file matching a pattern" and won't know how many until you've scanned), or a worker can crash mid-task and something has to notice and retry it, or the shards are uneven enough that some workers finish in ten seconds and sit idle while one straggler runs for twenty minutes. At that point static assignment stops being enough, and you need the thing distributed systems have always needed for shared mutable state under concurrency: a way to atomically claim a unit of work, and a way to record a result without two writers clobbering each other.

§3 How it actually works

Replace both failure-prone habits from the cold open — a shared mutable document, and pushing task assignments out at spawn time — with two primitives borrowed straight from queue-backed job systems. First, workers pull and atomically claim a task row from a table, using a single conditional update that can only succeed for one caller even if a hundred fire at the identical instant — the same underlying database guarantee that makes INSERT ... ON CONFLICT race-free for idempotency keys in Tool Design as API Design, applied here to work distribution instead of retry safety. Second, each worker's result is its own insert-only row keyed by task ID, never a read-modify-write against one shared document — so two workers finishing in the same instant produce two rows, not one row where the second write erased the first. A claim also carries a timestamp, so a reaper query can find tasks claimed long enough ago with no matching result and put them back up for grabs, covering the worker-crashed case without any worker having to detect its own death.

Layer 4 & 2 — atomic claim and insert-only results (Postgres + TypeScript)

// CREATE TABLE tasks (
//   id TEXT PRIMARY KEY, payload JSONB NOT NULL,
//   worker_id TEXT, claimed_at TIMESTAMPTZ
// );
// CREATE TABLE results (
//   task_id TEXT NOT NULL REFERENCES tasks(id),
//   worker_id TEXT NOT NULL, output JSONB NOT NULL,
//   created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
//   PRIMARY KEY (task_id)   -- one result per task, ever
// );

app.post("/tasks/claim", async (req, res) => {
  const workerId = req.body.worker_id;
  // The WHERE clause is what makes this race-free: under N concurrent
  // callers, Postgres serializes row-level updates, so exactly one caller's
  // UPDATE sees worker_id IS NULL for a given row and the rest see zero rows.
  const claim = await db.query(
    `UPDATE tasks SET worker_id = $1, claimed_at = now()
     WHERE id = (
       SELECT id FROM tasks WHERE worker_id IS NULL LIMIT 1 FOR UPDATE SKIP LOCKED
     )
     RETURNING id, payload`,
    [workerId]
  );
  if (claim.rowCount === 0) return res.status(204).end(); // nothing left to claim
  return res.status(200).json(claim.rows[0]);
});

app.post("/results", async (req, res) => {
  const { task_id, worker_id, output } = req.body;
  try {
    // INSERT, never UPDATE — a second submission for the same task_id
    // fails on the primary key instead of silently overwriting the first.
    await db.query(
      `INSERT INTO results (task_id, worker_id, output) VALUES ($1, $2, $3)`,
      [task_id, worker_id, output]
    );
    return res.status(201).json({ status: "recorded" });
  } catch (err) {
    if (err.code === "23505") { // unique_violation
      return res.status(409).json({ error: { code: "ALREADY_SUBMITTED", retryable: false } });
    }
    throw err;
  }
});

// Run every 60s: tasks claimed too long ago with no matching result go
// back into the pool for another worker to pick up.
async function reapStuckClaims() {
  await db.query(
    `UPDATE tasks SET worker_id = NULL, claimed_at = NULL
     WHERE claimed_at < now() - interval '5 minutes'
       AND id NOT IN (SELECT task_id FROM results)`
  );
}

Layer 3 — worker agent: claim, work, submit (Python)

import requests, uuid

def run_worker(orchestrator_url: str):
    worker_id = str(uuid.uuid4())
    while True:
        claim = requests.post(f"{orchestrator_url}/tasks/claim", json={"worker_id": worker_id})
        if claim.status_code == 204:
            break  # no tasks left — this worker is done, others may still run
        task = claim.json()

        # Isolated work: this worker never reads or writes another
        # worker's output, and never touches a shared document.
        finding = audit_service_for_secrets(task["payload"]["service_name"])

        submit = requests.post(f"{orchestrator_url}/results", json={
            "task_id": task["id"], "worker_id": worker_id, "output": finding,
        })
        if submit.status_code == 409:
            # A reaper requeued this task and another worker already
            # finished it first — not an error, just lost the race.
            continue

§4 Failure modes

What you'll actually see in production
  1. Shared mutable output, read-modify-write instead of insert. Symptom: the cold open — a run reports fewer results than tasks dispatched, with zero errors logged anywhere, because nothing failed; a write just silently overwrote another write. This is the single most common coordination bug because it's invisible until someone counts.
  2. The claim itself isn't atomic — a SELECT followed by a separate UPDATE. Symptom: trace or log review shows two different worker IDs both submitting a result for the same task_id, one of them hitting the results table's unique-key rejection. The gap between "read who's unclaimed" and "write that I've claimed it" is exactly wide enough for two workers to both read the row as free before either writes — the fix is a single conditional statement (the WHERE worker_id IS NULL update, or FOR UPDATE SKIP LOCKED), never a check-then-act pair.
  3. No claim timeout, so a crashed worker's task is claimed forever. Symptom: the job dashboard sits at "39/40 done" indefinitely; querying the tasks table shows one row with a worker_id and claimed_at from hours ago and no matching row in results. Whatever killed that worker — an unhandled exception, an OOM, a container eviction — never got a chance to release its claim, and nothing else was watching for a claim that never resolved.
  4. The coordination tax exceeds the parallelism payoff. Symptom: a benchmark run splitting the same total workload into progressively smaller tasks shows total wall-clock time going up, not down, past some worker count — each task's claim round-trip and each worker's independent re-read of shared context (a schema file, a style guide) start to dominate over the actual work being done. This isn't a bug in the coordination primitives; it's a sign the task granularity is wrong. Fewer, larger shards reduce the number of claim/commit round-trips and the number of times shared context gets paid for redundantly.
Principal's Judgment Call — dynamic claiming vs. static assignment

Reach for the claim-table pattern in §3 when the task count isn't known until runtime (you're processing "every file matching X," discovered by a scan), when workers can plausibly crash or hang and need their work picked up by someone else without re-running the whole batch, or when task sizes are uneven enough that a worker finishing early should be able to pull more work instead of sitting idle while a straggler grinds — a pull-based claim queue self-balances load; a push-based static assignment doesn't.

Stay with static assignment — spawn exactly N workers, hardcode worker i to shard i, no claim table at all — when the split is fixed and known before dispatch and a failed worker is acceptable to just retry as a whole unit (re-run worker 7, not "some subset of worker 7's rows"). It's strictly less code, there's no database in the critical path, and for a small, evenly-sized, known-ahead-of-time batch the coordination machinery in this article is solving a problem you don't have yet.

§5 Try it in 20 minutes

LAB · Estimated 20 minutes

Mission: stand up the §3 claim/result endpoints and prove, under real concurrency, that a task is claimed exactly once and a stuck claim gets reclaimed.

  1. Run Postgres locally: docker run -d --name pg -e POSTGRES_PASSWORD=pw -p 5432:5432 postgres:16, then create the tasks and results tables from the DDL comment in §3, and insert 10 rows into tasks.
  2. Wire up the Express handlers from Layer 2 & 4, with audit_service_for_secrets stubbed to sleep 1 second and return a fake finding.
  3. Fire 5 concurrent claim requests at once: for i in 1 2 3 4 5; do curl -s -X POST localhost:3000/tasks/claim -H "Content-Type: application/json" -d "{\"worker_id\":\"w$i\"}" & done; wait. Confirm each returned a different id — no two responses claim the same task.
  4. Submit a result for one claimed task, then submit a second result for that same task_id with a different worker_id. Confirm the second call returns 409 ALREADY_SUBMITTED and the results table still has exactly one row for that task.
  5. Manually backdate one task's claimed_at to 10 minutes ago (UPDATE tasks SET claimed_at = now() - interval '10 minutes' WHERE id = '...') with no matching result row, then run the reapStuckClaims query from Layer 4. Confirm that task's worker_id is now NULL and a fresh /tasks/claim call picks it up again.

Acceptance criteria:

  • ✅ Five simultaneous claim requests return five distinct task IDs — no duplicate claims.
  • ✅ A second result submission for an already-completed task is rejected with 409, and the results table never has more than one row per task_id.
  • ✅ A task claimed past the timeout with no result is reclaimable by a new worker after the reaper runs.

§6 Design-review question

Staff+ interview / design-review bank

A teammate proposes skipping the tasks table entirely: the orchestrator just spawns exactly one worker process per unit of work at dispatch time, hardcoding each worker's assignment as a startup argument. When does that break down, and when is it actually the right call?

Strong answer sketch: static, push-based assignment is strictly simpler and it's correct precisely as long as three things hold — the task count is known before dispatch, task sizes are roughly even, and a failed unit of work is cheap to retry wholesale. Where it breaks down is any workload where the count is discovered at runtime rather than known up front (a recursive file-tree walk, "find every X and process it," where you don't know N until you've already partly done the scan), where task cost varies enough that fixed 1:1 assignment leaves fast workers idle while a slow one is still the long pole, or where a worker can die mid-task and you need to reassign just the unfinished remainder rather than rerunning everything that worker owned. A claim table trades a small amount of infrastructure — one table, one conditional update, one reaper query — for self-balancing load distribution and crash recovery at the granularity of individual tasks instead of whole workers. The judgment call isn't "claim tables are better" in the abstract; it's recognizing which of those three assumptions your specific workload actually violates, because if none of them are violated, the claim table is machinery solving a problem you don't have.
