OpenTelemetry for Agents: Tracing a Request Across the Whole Stack

Your APM says the API is fast. The user says it took eleven seconds. Both are true, and the gap between them lives in a part of the stack nobody instrumented — until one trace ID ties the browser, the gateway, the agent loop, and the database query together.

PUBLISHED 2026-07-17 DIFFICULTY Practitioner READ ~10 min (~1,850 words) LAB trace a real request tree across three layers in Jaeger

§1 Cold open

An incident channel lights up: "agent chat is slow for some users, fine for others — same model, same prompt length." The API gateway's dashboard shows p95 latency at 400ms, well within SLA. The support ticket says the customer waited eleven seconds and watched a spinner. Nobody can explain the gap: the frontend's session-replay tool shows one opaque fetch call, the backend's APM shows a fast HTTP handler, and the agent runtime's logs sit in a third system with timestamps that don't line up with either. The on-call engineer greps a request ID by hand across three log aggregators and finds it forty minutes later, sitting inside a single tool call that fanned out to a database query with no index on the filter column. The 400ms and the 11 seconds were both real, correctly measured, for two spans of the same request that nothing had ever connected.

§2 The problem this solves — and what engineers did before it

Before distributed tracing was standard practice, teams correlated a request across services the manual way: generate a request ID at the edge, thread it through every function call and log line as an extra argument, and grep for it after the fact across whatever logging backends each team happened to own. This works until the request crosses a boundary someone forgot to thread the ID through — a background job, a retry, a fire-and-forget call into the agent runtime — at which point the trail goes cold and you're back to correlating by timestamp and hoping the clocks agree. Each layer also tended to buy its own observability tool — a frontend RUM product, a backend APM agent, whatever the LLM SDK's default logger printed — and none of them knew the others existed, so "what happened during this one turn" meant opening three consoles and reconciling them by eye.

OpenTelemetry (OTel) standardizes the two things that fix this: a wire format for propagating trace context across process and service boundaries (the W3C traceparent header), and a data model — traces made of parent/child spans — that any instrumented layer can emit into and any backend can render as one waterfall. As of mid-2026, OTel also ships GenAI semantic conventions (the gen_ai.* attribute namespace) purpose-built for LLM and agent spans — standardized names for the model called, token counts, tool arguments, and agent hand-offs — so an agent's spans look the same whether your backend is Jaeger, Honeycomb, or Datadog. These conventions are still in Development status, not stable, and attribute names have moved before; treat the field names below as correct as of this writing, not frozen forever.

§3 How it actually works

The mechanism is one object — a trace_id — created once at the edge and carried forward, unchanged, by every layer it passes through. Each layer adds a span (an operation with a start time, an end time, and attributes) as a child of whatever span was active when it started. The parent-child nesting is what turns four unrelated systems' worth of logs into one causal tree.

Layer 1 — frontend: root span on the browser, propagated over the wire (TypeScript)

// npm install @opentelemetry/sdk-trace-web @opentelemetry/api
import { trace } from "@opentelemetry/api";
import { WebTracerProvider } from "@opentelemetry/sdk-trace-web";

const provider = new WebTracerProvider();
provider.register(); // installs the W3C tracecontext propagator by default
const tracer = trace.getTracer("agent-chat-ui");

async function sendTurn(sessionId: string, message: string) {
  return tracer.startActiveSpan("submit_turn", async (span) => {
    span.setAttribute("session.id", sessionId);
    try {
      const res = await fetch(`/agent/turn`, {
        method: "POST",
        // The registered propagator injects `traceparent` into this fetch's
        // headers automatically — this line is here to make that fact visible.
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ sessionId, message }),
      });
      const traceId = span.spanContext().traceId;
      console.log(`support reference: trace ${traceId}`); // surface for tickets
      return await res.json();
    } finally {
      span.end();
    }
  });
}

Layer 3 — agent runtime: nested spans with GenAI semantic-convention attributes (Python)

# pip install opentelemetry-sdk opentelemetry-exporter-otlp anthropic
from opentelemetry import trace

tracer = trace.get_tracer("agent-runtime")

def run_turn(agent_name: str, user_message: str, tools: list):
    # This span is opened as a CHILD of whatever context the API layer
    # extracted from the inbound traceparent header — see Layer 2 below.
    with tracer.start_as_current_span("invoke_agent") as agent_span:
        agent_span.set_attribute("gen_ai.operation.name", "invoke_agent")
        agent_span.set_attribute("gen_ai.agent.name", agent_name)

        with tracer.start_as_current_span("chat claude-sonnet-5") as chat_span:
            chat_span.set_attribute("gen_ai.operation.name", "chat")
            chat_span.set_attribute("gen_ai.request.model", "claude-sonnet-5")
            response = anthropic_client.messages.create(
                model="claude-sonnet-5", max_tokens=1024,
                messages=[{"role": "user", "content": user_message}], tools=tools,
            )
            chat_span.set_attribute("gen_ai.usage.input_tokens", response.usage.input_tokens)
            chat_span.set_attribute("gen_ai.usage.output_tokens", response.usage.output_tokens)

        for block in response.content:
            if block.type != "tool_use":
                continue
            with tracer.start_as_current_span(f"execute_tool {block.name}") as tool_span:
                tool_span.set_attribute("gen_ai.operation.name", "execute_tool")
                tool_span.set_attribute("gen_ai.tool.name", block.name)
                # Do NOT set full arguments/results as attributes in prod — see
                # failure mode 2. Redact or hash anything that isn't a metric.
                result = call_tool(block.name, block.input)  # DB span nests here
                tool_span.set_attribute("gen_ai.tool.call.result.size", len(str(result)))
        return response

The piece that makes Layer 1 and Layer 3 one trace instead of two is Layer 2: the API server (Express, Spring, whatever fronts the agent runtime) must extract the inbound traceparent header and re-enter that context before calling into the agent — a one-line propagation.extract() call most OTel HTTP auto-instrumentation packages do for you. Layer 4, the data layer, needs no code at all here: a standard psycopg2 or JDBC OTel instrumentor reads whatever span is currently active (the execute_tool span above) and nests its query span under it automatically, because context propagation is ambient within a process, not something you thread through function arguments by hand. That's the whole payoff — the database query in the incident from §1 would have shown up as a labeled child span, in the same waterfall as the browser's fetch, instead of requiring a forty-minute manual grep.

§4 Failure modes

What you'll actually see in production
  1. Context propagation breaks at an async boundary. Symptom: in your tracing backend, a tool call that clearly ran as part of a user's turn shows up as its own disconnected root trace instead of a child span. This happens whenever work hops onto a thread pool, a background task queue, or a fire-and-forget async call without explicitly carrying the OTel context along — the new thread or worker has no ambient context to attach to, so it starts fresh. Fix: explicitly capture context.get_current() before the hop and re-attach it (context.attach()) on the other side; for message queues, serialize the traceparent into the message payload itself and extract it in the consumer.
  2. Content capture as span attributes becomes a cost and privacy liability. Symptom: your tracing backend's ingest cost triples and a security review flags customer PII sitting in span attributes. This happens when a team follows the GenAI conventions' optional prompt/completion capture too literally and logs full user messages or tool arguments as attributes on every span, which then get exported, indexed, and retained by a third-party backend under its own data-retention policy. Fix: default content capture off, and where you need it for debugging, gate it behind an explicit environment flag and redact or truncate before setting the attribute — never pass raw user input into gen_ai.tool.call.arguments in production.
  3. Head-based sampling drops exactly the trace you needed. Symptom: an error is visible in your error-rate dashboard, but searching the tracing backend for a matching trace around that timestamp turns up nothing. Head-based probabilistic sampling decides whether to keep a trace at the moment it starts, before anything has gone wrong, so a 1%-sampling policy keeps roughly 1% of your failures too — which is almost never the one from the ticket in front of you. Fix: add a tail-based sampling policy (sample after the trace completes, based on whether it errored or exceeded a latency threshold) at the collector, or at minimum force-sample any trace whose root span ends in an error status.
  4. High-cardinality span names blow up the backend's indexing cost. Symptom: your tracing backend's bill or query latency degrades sharply after a rollout, with no change in traffic volume. This happens when span names include per-request values — a tool span named execute_tool lookup_order_48213 instead of execute_tool lookup_order with the order ID as an attribute — because most backends index on span name and treat each distinct name as a new time series. Fix: keep span names to a small, fixed vocabulary (operation + resource type, per the GenAI conventions' own pattern) and put anything request-specific in attributes instead.
Principal's Judgment Call — full distributed tracing vs. a correlation ID

Reach for full OpenTelemetry instrumentation once a request genuinely crosses more than one process or service boundary with branching — an orchestrator fanning out to subagents, a tool call that hits its own microservice, or a hand-off across a queue where you need causal, timed reconstruction of what happened and in what order. The parent-child span model is what earns its cost there: it's the only structure that answers "which of these five parallel tool calls was the slow one" without hand-correlating five separate log streams by timestamp.

Stay with a plain generated correlation ID threaded through structured logs when the agent is a single in-process loop calling one LLM API with no cross-service hops — a request ID in every log line, searchable in whatever log aggregator you already run, gets you 90% of the debugging value with none of the propagation plumbing, the collector infrastructure, or the sampling-policy tuning that full tracing demands. Don't adopt OTel because it's the standard; adopt it when a request's story genuinely spans systems that don't share a log stream.

§5 Try it in 20 minutes

LAB · Estimated 20 minutes

Mission: instrument a minimal agent turn across the API and agent-runtime layers, export to a local Jaeger instance, and prove one trace_id ties both layers' spans together.

  1. Run Jaeger locally: docker run -d --name jaeger -p 16686:16686 -p 4318:4318 jaegertracing/all-in-one:latest (UI at http://localhost:16686, OTLP HTTP receiver on 4318).
  2. Wire the Python snippet from Layer 3 into a small Flask or FastAPI handler for POST /agent/turn, configured with an OTLP exporter pointed at http://localhost:4318/v1/traces, and add one fake execute_tool call plus one real database or in-memory lookup nested inside it.
  3. Add OTel's auto-instrumentation for your web framework (e.g. opentelemetry-instrumentation-flask) so the server span extracts context from an inbound traceparent header automatically.
  4. Fire a request with a manually-crafted trace header: curl -X POST localhost:5000/agent/turn -H "traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01" -d '{"message":"test"}' -H "Content-Type: application/json".
  5. Open Jaeger's UI, search for trace ID 4bf92f3577b34da6a3ce929d0e0e4736, and confirm the waterfall shows invoke_agent → chat and execute_tool as nested children under one root.

Acceptance criteria:

  • ✅ Jaeger's search finds the exact trace ID you passed in the curl header — proving the server extracted, not regenerated, the incoming context.
  • ✅ The waterfall view shows at least three nested spans (server, invoke_agent, chat or execute_tool) under one root, with correct parent-child indentation.
  • ✅ No span attribute in the UI contains a raw user message or tool argument — confirming your redaction from failure mode 2 actually took effect.

§6 Design-review question

Staff+ interview / design-review bank

Your orchestrator hands a subtask off to a worker agent by publishing a message onto a queue — not a direct function call — and the worker may not pick it up for several seconds. How do you keep that hand-off connected in your tracing, and would you make it a child span or something else?

Strong answer sketch: a direct parent-child span assumes the child starts and ends while the parent is still open, which breaks down across a queue — the orchestrator's span would have to stay open for however long the message sits queued, which pollutes latency metrics with wait time that isn't the orchestrator's own work and can hold the trace open indefinitely if the worker never picks it up. The right primitive is a span link rather than strict parent-child: serialize the current trace context (the traceparent) into the queued message's metadata, close the orchestrator's publish span normally, and have the worker's consumer start a new span that links back to that trace context instead of nesting under it. Most tracing backends render links as a visible cross-reference between two otherwise-independent traces, so you can still jump from the publish event to the eventual worker execution, but neither side's duration lies about how long the other side took. The failure mode to watch for is teams skipping this distinction and forcing parent-child semantics across the queue anyway — it works fine in the demo, then produces multi-hour "spans" in production the moment a worker is delayed, which corrupts every latency percentile the backend computes for that operation.
