A2A Protocol Anatomy: Agent Cards, Task Lifecycle, and Discovery Between Independent Agents

Two agents built by two different teams, on two different stacks, behind two different trust boundaries, need to hand off a unit of work and both need to agree on its state days later. A2A doesn't replace MCP's agent-to-tool contract — it standardizes the agent-to-agent one.

PUBLISHED 2026-07-23 DIFFICULTY Foundations READ ~9 min (~1,650 words) LAB stand up a minimal A2A server + client and prove task state survives a disconnect over curl

§1 Cold open

Your procurement agent needs a vendor's reimbursement status. It calls a REST endpoint the vendor's platform team hand-rolled eighteen months ago, gets back {"status": "processing"}, and has no way to know whether that means five seconds or five days. Six minutes later your orchestrator's retry policy fires — the connection timed out on your side, not theirs — and the vendor's system, which was never given an idempotency key because nobody agreed on one, opens a second reimbursement case for the same invoice. Nobody on your team can see the vendor's task queue. Nobody on the vendor's team knows your retry logic exists. Neither agent is broken; they were never speaking a protocol built for this handoff, so every partner integration reinvents task states, polling cadence, and auth from scratch, badly, once per partner.

§2 The problem this solves — and what engineers did before it

Before a standard existed for this, teams had two bad options. The first was a bespoke webhook contract per partner: an ad hoc status string with meanings only one team's runbook explained, no standard way to resume after a dropped connection, and authentication negotiated over email. The second was pretending the remote agent was just another tool — wrapping it behind a synchronous HTTP call the way you'd call any internal function. That works for a single hop under thirty seconds and falls apart the moment the task needs a human's approval on the other side, runs for hours, or needs to survive your process restarting mid-call. OpenAPI, the closest existing standard, describes CRUD-shaped resources with a fixed set of verbs — it has no vocabulary for "this task is now waiting on a human," no concept of a capability an agent can reason about and choose to invoke, and no standard way for the remote side to say what it can do before you commit to calling it.

That last gap is the one worth being precise about, because it's easy to confuse with a problem MCP already solves. MCP standardizes how a host application calls tools inside its own trust boundary — the tools it registers are effectively extensions of its own runtime. A2A (originated at Google in April 2025, now governed by the Linux Foundation, with v1.0 shipped April 2026) standardizes something structurally different: how one autonomous agent discovers what a separate agent — running its own reasoning loop, on infrastructure you don't control, behind its own auth — can do, and how the two track a task's lifecycle across that boundary without a shared database, a shared deploy, or a shared team.

§3 How it actually works

A2A defines three things a caller needs and one thing both sides need to agree on. Discovery: a remote agent publishes an Agent Card — a JSON document at the well-known URI /.well-known/agent-card.json — declaring its name, its skills (each with an id, description, and supported input/output modes), the auth schemes it accepts, and the endpoint to call. Transport: JSON-RPC 2.0 over HTTPS, with a synchronous SendMessage for short tasks, SendStreamingMessage opening a Server-Sent Events stream of status and artifact updates for long-running ones, and — critically — GetTask, which lets either side ask "where does this task actually stand" independent of whatever request just failed. What both sides agree on is the task object itself: an id, a status.state, and a history of messages and artifacts. State moves through submitted → working → a terminal state (completed, failed, canceled, rejected), with input-required and auth-required as pause states — the task isn't done or dead, it's parked waiting for a follow-up message on the same task id.

GetTask is the piece that answers the cold open's question, if you actually call it before retrying: a task id is a durable handle to state that outlives any single connection, so the correct response to a timeout is "ask what happened," not "resend and hope." That reframes cross-org delegation as a four-layer problem, not just a network call: the frontend layer is wherever a human watches tasks that parked waiting on them; the API/transport layer is A2A's own wire format; the agent runtime layer is your orchestrator deciding to delegate and the remote agent's skill handler doing the work; and the data/infra layer is the local record that maps your side's idempotency key to the remote task id, so a retry becomes a lookup instead of a second SendMessage.

Layer 3 & 2 — remote agent runtime + transport: a minimal A2A server (Python)

# A single skill, "check-invoice-status", exposed over A2A's two
# required surfaces: the Agent Card (discovery) and a JSON-RPC endpoint
# (SendMessage, GetTask). Runs with: uvicorn a2a_server:app
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
import uuid, time

app = FastAPI()
TASKS: dict[str, dict] = {}

AGENT_CARD = {
    "name": "vendor-billing-agent",
    "description": "Looks up invoice and reimbursement status for a partner.",
    "url": "https://vendor.example.com/a2a",
    "skills": [{
        "id": "check-invoice-status",
        "description": "Given an invoice id, return its current status.",
        "inputModes": ["application/json"],
        "outputModes": ["application/json"],
    }],
    "securitySchemes": {"bearer": {"type": "http", "scheme": "bearer"}},
}

@app.get("/.well-known/agent-card.json")
def agent_card():
    return AGENT_CARD

@app.post("/a2a")
async def rpc(request: Request):
    body = await request.json()
    method, params, req_id = body["method"], body.get("params", {}), body["id"]

    if method == "SendMessage":
        task_id = str(uuid.uuid4())
        invoice_id = params["message"]["parts"][0]["text"]
        # Real handlers do real work here; this one resolves immediately.
        TASKS[task_id] = {
            "id": task_id,
            "status": {"state": "completed", "timestamp": time.time()},
            "artifacts": [{"parts": [{"text": f"invoice {invoice_id}: paid"}]}],
        }
        result = TASKS[task_id]

    elif method == "GetTask":
        task = TASKS.get(params["id"])
        if task is None:
            return JSONResponse(
                {"jsonrpc": "2.0", "id": req_id,
                 "error": {"code": -32001, "message": "task not found"}},
                status_code=404,
            )
        result = task

    else:
        return JSONResponse(
            {"jsonrpc": "2.0", "id": req_id,
             "error": {"code": -32601, "message": "method not found"}},
            status_code=400,
        )

    return {"jsonrpc": "2.0", "id": req_id, "result": result}

Layer 3 & 4 — local orchestrator: idempotency shadow row before the call (Python)

# The shadow row is written BEFORE the network call. If the process
# crashes or the connection times out after that write, the caller's
# next attempt looks the key up instead of minting a second task.
import httpx, psycopg, hashlib, json

A2A_ENDPOINT = "https://vendor.example.com/a2a"

def check_invoice_status(invoice_id: str) -> dict:
    idem_key = hashlib.sha256(f"check-invoice:{invoice_id}".encode()).hexdigest()[:24]

    with psycopg.connect() as conn:
        existing = conn.execute(
            "SELECT remote_task_id, last_state FROM task_shadow WHERE idempotency_key = %s",
            (idem_key,),
        ).fetchone()

        if existing:
            remote_task_id, last_state = existing
            if last_state in ("completed", "failed", "canceled", "rejected"):
                return _cached_result(conn, remote_task_id)
            # In-flight from a prior attempt: reconcile, don't resend.
            return _get_task(remote_task_id, conn)

        conn.execute(
            "INSERT INTO task_shadow (idempotency_key, remote_task_id, last_state) "
            "VALUES (%s, NULL, 'pending')",
            (idem_key,),
        )
        conn.commit()

    resp = httpx.post(A2A_ENDPOINT, json={
        "jsonrpc": "2.0", "id": idem_key, "method": "SendMessage",
        "params": {"message": {"parts": [{"text": invoice_id}]}},
    }, timeout=30.0).json()

    task = resp["result"]
    with psycopg.connect() as conn:
        conn.execute(
            "UPDATE task_shadow SET remote_task_id = %s, last_state = %s "
            "WHERE idempotency_key = %s",
            (task["id"], task["status"]["state"], idem_key),
        )
        conn.commit()
    return task

def _get_task(remote_task_id: str, conn) -> dict:
    resp = httpx.post(A2A_ENDPOINT, json={
        "jsonrpc": "2.0", "id": remote_task_id, "method": "GetTask",
        "params": {"id": remote_task_id},
    }, timeout=30.0).json()
    task = resp["result"]
    conn.execute(
        "UPDATE task_shadow SET last_state = %s WHERE remote_task_id = %s",
        (task["status"]["state"], remote_task_id),
    )
    conn.commit()
    return task

That's the same idempotency-key discipline as Tool Design as API Design, applied one trust boundary further out — inside your own system a lost update is a bug; across an org boundary it's a bug you can't even see the other half of.

§4 Failure modes

What you'll actually see in production
  1. Blind retry storms across the trust boundary. Symptom: the remote agent's task log shows two tasks with near-identical payloads seconds apart, and you find out from the vendor's invoice, not your own logs, because a retry that skips GetTask and calls SendMessage again looks identical to a legitimate new request from the far side of the boundary — there's no shared database to catch the duplicate the way there would be inside one team's system.
  2. Stale Agent Card, silently wrong. Symptom: task calls that worked yesterday start failing with an auth error or a schema mismatch immediately after a partner deploy, because your orchestrator cached the Agent Card and kept calling a skill whose inputModes or security scheme changed underneath it. Nothing on your side changed; the card you're reasoning from is no longer the one the endpoint actually implements, and there's no version field forcing you to notice.
  3. A task parked at input-required that nobody's watching. Symptom: a remote task sits paused indefinitely and eventually times out on the far side, while your system has no record it was ever waiting — because the orchestrator only handled the synchronous completed path and never wired the §3 pause states to anything a human sees. The task didn't fail; it just aged out while parked, and your side finds out only when the vendor asks why you never replied.
  4. Treating a remote skill's description as trusted tool text. Symptom: once the Agent Card's skills[].description is fed straight into your local model's tool list, it's exposed to exactly the injection surface Tool Descriptions Are Prompts describes — except now the description is authored by an organization you don't control, not a teammate's PR. A card you haven't reviewed is prompt text you haven't reviewed.
Principal's Judgment Call — A2A vs. a plain internal API

Reach for A2A when the two agents are owned by genuinely separate parties — different teams, different companies, different deploy cadences — and you need standardized discovery (what can the other side even do) plus a task lifecycle that survives disconnects, human-in-the-loop pauses, and retries without a shared database to referee them. That's the situation the protocol was built for, and rolling your own version of agent cards, task states, and reconciliation semantics per partner is the exact reinvention this article's cold open describes.

Stay with a plain internal API — or the coordination patterns in Orchestrator-Workers and the Coordination Tax — when everything lives inside one trust boundary and one deploy. Agent cards and JSON-RPC discovery buy you nothing over a typed internal client when you already control both ends' schemas and can just change them together; adding A2A there is protocol overhead paying for a problem — cross-org discovery — you don't have.

§5 Try it in 20 minutes

LAB · Estimated 20 minutes

Mission: run the §3 server and client, then prove over curl that GetTask — not a resend — is how you recover from a dropped connection.

  1. Save the FastAPI server from §3 as a2a_server.py and run uvicorn a2a_server:app --port 8000.
  2. curl localhost:8000/.well-known/agent-card.json — confirm the card lists the check-invoice-status skill.
  3. Send a task: curl -X POST localhost:8000/a2a -H 'content-type: application/json' -d '{"jsonrpc":"2.0","id":"1","method":"SendMessage","params":{"message":{"parts":[{"text":"INV-4471"}]}}}'. Record the returned result.id.
  4. Simulate the cold open's dropped connection: instead of resending step 3, call GetTask with that id: curl -X POST localhost:8000/a2a -d '{"jsonrpc":"2.0","id":"2","method":"GetTask","params":{"id":"<task id>"}}'.
  5. Wire the §3 Python client's check_invoice_status() against a local Postgres instance (or swap psycopg calls for an in-memory dict for the lab), and call it twice in a row with the same invoice_id.

Acceptance criteria:

  • ✅ Step 3's response and step 4's GetTask response both show status.state: "completed" for the same task id — proving state is queryable independent of the original request.
  • ✅ The server's in-memory TASKS dict contains exactly one entry after step 3, regardless of how many times you run step 4.
  • ✅ The two calls to check_invoice_status() in step 5 produce exactly one row in task_shadow and exactly one SendMessage hitting the server — verify by counting requests in the server's uvicorn log, not just checking the client's return value.

§6 Design-review question

Staff+ interview / design-review bank

Your company wants to expose an internal agent to an external partner over A2A for the first time. What do you refuse to put in the Agent Card, and how do you stop the task lifecycle itself from becoming an attack surface?

Strong answer sketch: the Agent Card should declare the minimum skill set the partner actually needs, not the agent's full internal capability — every skill listed is a capability an untrusted caller can now invoke by name, so "expose everything, gate with auth" is backwards from "expose the minimum, so a compromised or over-broad auth grant has a small blast radius." Concretely: no internal-only skills, no skill descriptions that leak implementation details (database names, internal service topology) the way a careless docstring would, and security schemes scoped per skill rather than one bearer token that unlocks all of them, so a leaked credential doesn't imply full access. On the task-lifecycle side, the attack surface isn't just the initial call: a caller that knows a task id can potentially GetTask or send follow-up messages on tasks it didn't create unless the server checks that the caller's identity matches the task's original requester — treat task ids as authorization-scoped handles, not bearer capabilities, the same discipline you'd apply to any object reference in a REST API. And rate-limit per authenticated caller at the task-creation path specifically, since an external partner controls when and how often SendMessage fires against your skill handler, and that handler is now attacker-adjacent infrastructure whether or not you think of it that way.
