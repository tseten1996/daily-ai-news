Tool Design as API Design: The Contract Your Agent's Tools Can't Skip

A tool function with a JSON schema bolted on is not an API. The moment something between the model and that function is allowed to retry — and in production, something always is — every side effect the tool performs needs the same contract a payments API has needed for a decade.

PUBLISHED 2026-07-17 DIFFICULTY Practitioner READ ~10 min (~1,950 words) LAB enforce idempotent tool calls end-to-end and prove it with curl

§1 Cold open

At 2:14am an on-call engineer gets paged: a customer was refunded $340 twice for one order. The support transcript shows a single refund_order tool call. The payments ledger shows two separate reversals, six seconds apart. The trace — instrumented last quarter, see OpenTelemetry for Agents — finally earns its keep: two execute_tool refund_order spans, same order_id, same amount, two different tool_call_ids. The orchestrator's HTTP client timed out waiting on the first call and silently retried it. The refund tool — a five-line Python function wrapping a payments SDK call — had no way to know the second request wasn't a brand-new refund. It did exactly what it was told, twice, because nothing had ever told it not to.

§2 The problem this solves — and what engineers did before it

Public APIs learned this lesson the expensive way, over roughly a decade. Early payment APIs double-charged customers whenever a client's network request succeeded on the server but the response never made it back, so the client — reasonably — retried. Stripe standardized the fix around 2012: every state-changing request carries an Idempotency-Key header, the server records the outcome of the first request under that key, and any retry with the same key gets the original response replayed instead of executed again. Google's API guidelines later generalized the same pattern as a request_id field (AIP-155) so it would work uniformly across REST and RPC. Alongside idempotency, mature APIs settled on versioned schemas so a breaking field change doesn't silently break every existing client, and structured error taxonomies — HTTP status codes, or a typed error field — so a caller could tell a transient failure worth retrying from a permanent one that isn't.

Agent tool integrations skipped all of it. A tool started life as a Python function decorated with a JSON schema and wired directly into the model's function-calling loop — no idempotency key, no version field, no distinction between "retry me" and "never retry me," because in a demo nothing fails halfway through. Then production reliability work arrived: a retrying HTTP client between the model and the tool (raw model calls time out sometimes), a self-correction loop that lets the model reissue a malformed call, an orchestrator that resumes a crashed run from its last completed step. Each of those is a correct reliability decision on its own. Stacked on a tool with no contract, together they reintroduce the exact double-execution bug the API world already spent ten years fixing — the tool function just never got the memo.

§3 How it actually works

Treat every tool with a side effect as a versioned API endpoint the model happens to call through a function-calling interface instead of curl. That means four ingredients, spread across four layers: a caller-supplied idempotency key that survives retries unchanged, a server that atomically claims that key before doing any work, a typed error shape that separates retryable from terminal failures, and — for anything with real blast radius — a human approval step that is itself idempotent, so a double-click doesn't approve twice.

Layer 3 — agent runtime: a key that survives retries (Python)

# The harness, not the model, owns the key — see the design-review
# question in §7 for why that boundary matters.
import hashlib

def idempotency_key_for(session_id: str, plan_step_id: str) -> str:
    # Deterministic: the SAME (session, step) always yields the SAME key,
    # even across process restarts, so a resumed run reuses it correctly.
    raw = f"{session_id}:{plan_step_id}"
    return hashlib.sha256(raw.encode()).hexdigest()[:32]

def call_tool_with_retry(tool_name, args, session_id, plan_step_id, max_attempts=3):
    key = idempotency_key_for(session_id, plan_step_id)  # minted ONCE
    last_err = None
    for attempt in range(max_attempts):
        try:
            return http_post(
                f"/tools/{tool_name}",
                json=args,
                headers={"Idempotency-Key": key},  # SAME key every attempt
                timeout=5,
            )
        except TimeoutError as e:
            last_err = e
            continue  # safe: the server will replay, not re-execute
    raise last_err

Layer 2 & 4 — API + data: claiming the key atomically (Node/Express + Postgres)

// CREATE TABLE idempotency_keys (
//   key TEXT PRIMARY KEY,
//   status TEXT NOT NULL DEFAULT 'processing',   -- processing | complete
//   response JSONB,
//   expires_at TIMESTAMPTZ NOT NULL DEFAULT now() + interval '24 hours'
// );

app.post("/tools/refund_order", async (req, res) => {
  const key = req.header("Idempotency-Key");
  if (!key) {
    // Structured, typed error — not a bare 400 — see failure mode 2.
    return res.status(400).json({ error: { code: "MISSING_IDEMPOTENCY_KEY", retryable: false } });
  }

  const claim = await db.query(
    `INSERT INTO idempotency_keys (key) VALUES ($1)
     ON CONFLICT (key) DO NOTHING RETURNING key`,
    [key]
  );

  if (claim.rowCount === 0) {
    // Key already claimed — either still processing or done. Poll/replay.
    const row = await db.query(`SELECT status, response FROM idempotency_keys WHERE key = $1`, [key]);
    if (row.rows[0].status === "complete") {
      res.setHeader("Idempotent-Replayed", "true");
      return res.status(200).json(row.rows[0].response);
    }
    return res.status(409).json({ error: { code: "IN_FLIGHT", retryable: true } });
  }

  try {
    const result = await refundViaPaymentsSDK(req.body.order_id, req.body.amount);
    await db.query(`UPDATE idempotency_keys SET status='complete', response=$2 WHERE key=$1`, [key, result]);
    return res.status(200).json(result);
  } catch (err) {
    // A declined card is terminal; a gateway timeout is retryable.
    // Distinguish or the caller retries a refund that will never succeed.
    const retryable = err.code === "GATEWAY_TIMEOUT";
    await db.query(`DELETE FROM idempotency_keys WHERE key = $1`, [key]); // free the key only if retryable
    return res.status(502).json({ error: { code: err.code, retryable } });
  }
});

Note that the frontend's approval action carries its own idempotency key (the approval ID), distinct from the tool call's key from Layer 3 — a double-click on "Approve" and a retried HTTP POST from the tool call are two different retry surfaces, and conflating them is how teams end up idempotent against the wrong request.

§4 Failure modes

What you'll actually see in production
  1. No idempotency key at all. Symptom: the cold open — two execute_tool spans with identical arguments and different tool_call_ids, both showing a successful result, and a downstream ledger with two rows for one logical action. Root cause is almost always a retrying HTTP client sitting between the model and a tool that was written as if it would only ever be called once.
  2. Errors aren't typed as retryable or terminal. Symptom: a trace shows five, ten, twenty consecutive execute_tool spans, all failing with the same argument, all logged as generic exceptions. A card decline is permanent; a gateway timeout is transient — if the tool returns an undifferentiated Error: refund failed for both, the orchestrator's blanket retry policy hammers a call that will never succeed, burning latency and, if the tool is metered, real money. The MCP tools spec's isError flag exists precisely to give callers this signal at the protocol level — use an equivalent structured field even outside MCP.
  3. A schema field changes without a version bump. Symptom: a spike in schema-validation-error rate immediately after a deploy, correlated in your trace tool with the deploy timestamp — the model keeps sending amount_cents because that's what it saw in the system prompt's tool description at the start of a long-running session, while the endpoint now expects amount. Long sessions and cached system prompts mean old and new tool contracts run concurrently; an unversioned rename breaks every in-flight session at once instead of failing loudly at the boundary.
  4. The idempotency key's TTL or derivation is wrong. Symptom one: a legitimate second refund on the same order, issued a week later for an unrelated reason, silently returns last week's cached response because the key was derived from order_id alone instead of a per-attempt identifier — "the tool said success but the ledger never moved." Symptom two: a key expires (Stripe's default window is 24 hours) mid-retry on a genuinely slow operation, so a legitimate retry after the TTL executes the side effect a second time, which is the original bug wearing a disguise.
Principal's Judgment Call — full contract vs. a bare schema

Reach for the full contract — idempotency keys, a versioned schema, a typed retryable/terminal error field — when the tool has side effects that cost money, mutate durable state, or are expensive to reverse: payments, infrastructure changes, sending an external message, deleting a resource. That's exactly the set of tools where "called twice" and "called once" produce different outcomes in the world.

Stay with a bare JSON-schema function and let the orchestrator retry freely when the tool is a pure read with no side effects — search, lookup, describe. Duplicate reads are free; building idempotency-key plumbing for a get_order_status tool is solving a problem that doesn't exist and adds a database round-trip to every call for no benefit. The dividing line isn't how important the tool feels — it's whether calling it twice changes the world twice.

§5 Try it in 20 minutes

LAB · Estimated 20 minutes

Mission: stand up the API-layer endpoint from §3 backed by a real database, and prove with curl that a retried call replays instead of re-executing.

  1. Run Postgres locally: docker run -d --name pg -e POSTGRES_PASSWORD=pw -p 5432:5432 postgres:16, then create the idempotency_keys table and a refunds ledger table (id, order_id, amount, created_at) from the DDL comment in §3.
  2. Wire up the Express handler from Layer 2, with refundViaPaymentsSDK stubbed to insert one row into refunds and return a fake transaction ID.
  3. Fire the first call: curl -X POST localhost:3000/tools/refund_order -H "Idempotency-Key: k-001" -H "Content-Type: application/json" -d '{"order_id":"ord_42","amount":340}'. Confirm it returns 200 with a transaction ID.
  4. Fire the exact same call again with the same key. Confirm the response body is byte-identical and the response includes Idempotent-Replayed: true.
  5. Query SELECT count(*) FROM refunds WHERE order_id = 'ord_42' — it must return 1, not 2.
  6. Fire a third call with a different key (k-002) and the same order_id. Confirm it executes again (the count becomes 2) — proving the key, not the order ID, is what's being deduplicated.

Acceptance criteria:

  • ✅ Two requests with the same Idempotency-Key produce one row in refunds and an identical response body.
  • ✅ The replayed response carries Idempotent-Replayed: true; the original does not.
  • ✅ A request with a missing Idempotency-Key header is rejected with {"error":{"code":"MISSING_IDEMPOTENCY_KEY","retryable":false}} and never reaches refundViaPaymentsSDK.

§6 Design-review question

Staff+ interview / design-review bank

A teammate proposes skipping harness-side key generation: instead, add "call each tool with a unique idempotency key, and reuse the same key if you're retrying your own call" to the system prompt, and let the model generate the key itself. What's wrong with that, and what would you do instead?

Strong answer sketch: idempotency is a guarantee about identity across separate attempts, and the model has no reliable notion of "this is the same attempt as before" — it's a stateless function of its context window, and a retry might arrive as a fresh turn where the model has no memory of having generated a key moments ago, so it mints a new one and defeats the dedup. The failure runs the other direction too: a model asked to "reuse keys for retries" can just as easily conflate two legitimately different calls that happen to look similar in context — over-collapsing a second, intentional refund into a replay of the first. Idempotency keys need to come from something that actually tracks attempt identity across retries: the harness, keyed off a stable (session_id, plan_step_id) pair that doesn't change no matter how many times that specific step is retried, exactly like §3's Python snippet. This is the general "agent proposes, deterministic engine disposes" boundary — the model decides what tool to call and with what arguments; a deterministic piece of code the model doesn't control decides the request's identity, because that decision has to be consistent to be worth anything, and consistency is not a property a non-deterministic generator can be trusted to hold on its own.

§7 Further reading

  • Stripe — Idempotent requests (the 24-hour key TTL and "save the first response, replay it on retry" pattern this article's data layer borrows directly)
  • Google AIP-155 — Request identification (the request_id/idempotency pattern generalized across REST and RPC API styles)
  • Model Context Protocol — Tools (the isError flag and structuredContent shape referenced in failure mode 2, for typing execution errors at the protocol level)
  • IETF httpapi WG — The Idempotency-Key HTTP Header Field (expired as a working-group draft, but documents the header-field convention this pattern is built on)
