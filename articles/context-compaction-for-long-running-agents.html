Compaction for Long-Running Agents: What Survives When the Context Window Doesn't

Summarizing old turns to stay under a token budget sounds like a storage problem. It's actually a lossy-compression problem, and the thing that gets lost is whatever the summarizer — a model with no idea what the agent will need forty turns from now — decided wasn't narratively important.

PUBLISHED 2026-07-21 DIFFICULTY Principal READ ~12 min (~2,420 words) LAB verify a tagged constraint survives compaction while stale facts don't, via curl against both transcript views

§1 Cold open

A migration agent was three hours into moving a monorepo's forty-odd services off a deprecated ORM. Turn 34: it tried the straightforward rewrite on the billing service's models, ran the test suite, and got nineteen failures from a circular-import pattern the old ORM had silently tolerated. It reverted, tried a second approach — splitting the models file in two — and that one passed. Turn 91: the context hit the harness's compaction threshold. The summarizer folded turns 1 through 70 into four paragraphs of prose, optimized for "what happened," and moved on. Turn 130, working through a related service, the agent circled back to re-examine the billing models — and re-tried the first, already-failed approach, because the compacted summary described the outcome ("billing models were migrated") without describing the specific thing that hadn't worked and why. It burned eleven minutes and a full test-suite run rediscovering a fact it already knew forty turns earlier, then quietly re-applied the fix it had already found, from scratch. Nobody looking at the final diff would ever know — but the token bill for that session was measurably higher than it needed to be, for pure rework.

§2 The problem this solves — and what engineers did before it

The earliest fix for "context window filled up" wasn't compaction at all — it was crashing, or silently dropping the oldest messages off the front of the list to make room, the same way a ring buffer overwrites its oldest slot. Both are honest about the tradeoff: dropped context is gone, full stop, and nobody pretends otherwise. The failure mode was obvious and constant enough that harness authors moved to the next idea — summarize instead of drop. An LLM call turns the aging portion of the transcript into a paragraph, the paragraph replaces the original messages, and the agent keeps going with a fraction of the tokens. This is now standard: the Claude Agent SDK's tool_runner compacts automatically past a configurable token threshold, and Anthropic's own context-engineering guidance frames the whole problem as "curating the smallest possible set of high-signal tokens" rather than maximizing what fits.

What "summarize instead of drop" doesn't solve on its own is which tokens are high-signal to a task the summarizer doesn't fully understand yet, because the task isn't finished. A human skimming a meeting for a report and an agent that will act on a transcript forty turns later have different definitions of "important" — the human wants narrative continuity; the agent needs the specific rejected approach, the specific constraint, the specific file path, verbatim, because a paraphrase of "billing models were migrated" is not the same fact as "the direct rewrite fails on a circular import — use the split-file approach." Free-form prose summarization optimizes for the former by default, because that's what summarization models are trained to produce. The fix that closes this gap has two parts, covered in §3: force the summary into a structure that has a dedicated slot for the kinds of facts an agent actually needs back, and let certain messages skip summarization entirely by being tagged as anchors the compactor is never allowed to touch.

§3 How it actually works

A compaction pass runs in four steps, spread across two layers of the runtime, with the result made visible in two more. First, the harness checks token usage against a budget before the next model call and triggers compaction if it's over threshold. Second, it segments the transcript into three parts: anchor messages (tagged by the harness as constraints or hard-won decisions, carried forward verbatim no matter how old), a recent window (the last N turns, always kept raw), and the compactable middle. Third, the compactable middle is sent to a summarization call constrained to a structured schema — not "write a summary," but "extract these specific fields" — so a rejected approach has somewhere to go that isn't at the mercy of the model's sense of what's narratively worth keeping. Fourth, the original messages are archived (never deleted) in the data layer, the active context is replaced with system prompt + anchors + structured summary + recent window, and that replacement is checkpointed before the agent's next model call — so a crash mid-task resumes from the compacted state instead of redoing the compaction or silently losing it.

Layer 3 — agent runtime: structured summary + anchor tags (Python)

from pydantic import BaseModel

class CompactionSummary(BaseModel):
    decisions: list[str]            # "billing service: use split-file approach"
    rejected_approaches: list[str]  # "direct model rewrite — fails on circular import"
    files_touched: list[str]
    open_questions: list[str]

ANCHOR_TAGS = {"constraint", "decision"}  # set by the harness when a message is written,
                                           # not inferred later — see failure mode 2
KEEP_RECENT = 12

def maybe_compact(session, model, token_budget):
    if session.token_count < token_budget * 0.8:
        return session

    anchors = [m for m in session.messages if m.tag in ANCHOR_TAGS]
    recent = session.messages[-KEEP_RECENT:]
    compactable = [
        m for m in session.messages[:-KEEP_RECENT] if m.tag not in ANCHOR_TAGS
    ]

    summary = model.generate(
        prompt=SUMMARIZE_PROMPT.format(transcript=render(compactable)),
        response_schema=CompactionSummary,  # structured — no free-prose escape hatch
    )

    session.archive(compactable)     # full originals kept, see data layer below
    session.messages = [session.system_prompt, *anchors, as_message(summary), *recent]
    session.checkpoint()             # persisted before the next model call fires
    return session

Layer 4 — data/infra: archive + checkpoint schema (Postgres)

CREATE TABLE session_messages (
  id BIGSERIAL PRIMARY KEY,
  session_id UUID NOT NULL,
  turn_index INT NOT NULL,
  role TEXT NOT NULL,
  content TEXT NOT NULL,
  tag TEXT,                     -- NULL | 'constraint' | 'decision'
  compacted_into UUID,          -- NULL until archived; FK to the checkpoint that folded it in
  created_at TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE session_checkpoints (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  session_id UUID NOT NULL,
  turn_index INT NOT NULL,      -- cutoff: messages up to here were folded in
  summary JSONB NOT NULL,       -- the CompactionSummary, queryable, diffable
  active_context JSONB NOT NULL, -- exactly what was sent to the model next — for replay/debug
  created_at TIMESTAMPTZ DEFAULT now()
);

Layer 2 — API/transport: serving both views (Node/Express)

app.get("/sessions/:id/transcript", async (req, res) => {
  const raw = req.query.view === "raw";
  if (raw) {
    const rows = await db.query(
      `SELECT * FROM session_messages WHERE session_id = $1 ORDER BY turn_index`,
      [req.params.id]
    );
    return res.json(rows.rows);
  }
  const checkpoint = await db.query(
    `SELECT * FROM session_checkpoints WHERE session_id = $1 ORDER BY turn_index DESC LIMIT 1`,
    [req.params.id]
  );
  const recent = await db.query(
    `SELECT * FROM session_messages WHERE session_id = $1 AND compacted_into IS NULL ORDER BY turn_index`,
    [req.params.id]
  );
  return res.json({ checkpoint: checkpoint.rows[0] ?? null, recent: recent.rows });
});

§4 Failure modes

What you'll actually see in production
  1. Free-prose summarization drops the fact the agent actually needed. Symptom: the cold open — a rejected approach is present in the archived session_messages rows but absent from the summary JSON, and forty turns later the agent re-tries the same failed approach, visible in the trace as two execute_tool spans with near-identical arguments separated by a compaction checkpoint. Forcing the summary into CompactionSummary's fixed fields (§3) doesn't guarantee nothing gets lost, but it does guarantee "rejected approaches" has a slot to be lost from — a diff against the raw transcript then tells you exactly what the summarizer skipped, instead of leaving you guessing at a paragraph's omissions.
  2. Anchor tagging is opt-in and someone forgets it. Symptom: a constraint stated once in an ordinary turn — never tagged constraint at write time — gets swept into the compactable middle and vanishes into the summary's judgment of what mattered, precisely the mechanism behind the wire-transfer failure in Hooks as Deterministic Guardrails. That article's fix is a deterministic gate that doesn't depend on context surviving at all; this one's fix is narrower — audit session_messages for rows matching known constraint patterns (dollar thresholds, "never," "do not") with tag IS NULL, and treat every hit as a tagging bug, not a one-off.
  3. Cascading compaction degrades a checkpoint that was already a summary. Symptom: a session with three or more checkpoints where files_touched and decisions in the latest checkpoint's JSON are a strict subset of the first checkpoint's, even though the agent did strictly more work in between — each pass summarizes the previous summary along with new turns, and prose-of-a-prose compresses like a re-encoded video: each generation loses detail the last one hadn't lost yet. The fix is to always re-summarize from the full archived originals plus the previous checkpoint's structured fields (which merge losslessly, being lists) rather than from the previous summary's rendered text.
  4. A checkpoint races an in-flight tool result. Symptom: a tool call's result row lands in session_messages with a turn_index just older than a checkpoint's cutoff but wasn't included when the checkpoint was computed — an off-by-one between "when compaction read the table" and "when the tool result was written." The active context sent to the model either duplicates that result (it's in both the summary's files_touched and the raw recent window) or drops it (compacted_into IS NULL forever, orphaned, never in either view). The fix is the same discipline as any other read-then-write race: compute the checkpoint's cutoff and mark rows compacted_into inside one transaction, not two round trips a tool call can land between.
Principal's Judgment Call — structured compaction vs. a plain sliding window

Reach for structured, anchor-aware compaction when sessions genuinely run long enough to approach the token budget more than once — multi-hour coding sessions, long research loops, anything racking up dozens of tool calls — and where losing one specific hard-won fact costs real rework, not just a slightly worse paragraph. The structured schema and archive-plus-checkpoint machinery in §3 exist to make that loss detectable and bounded, not to eliminate it entirely.

Stay with a plain sliding window — keep the last N turns, drop everything older, no LLM call involved — when a task is naturally recency-scoped by construction: a live monitoring agent whose decisions should only ever depend on the last few observations, or a bounded subagent dispatched for one unit of work in an orchestrator-workers fan-out, which finishes well inside N turns and never approaches a budget in the first place. A sliding window has none of §4's failure modes because it has no summarization step to go wrong — but it deletes rather than compresses, so anything relevant from more than N turns back is gone outright, not lossily available. Choose it only where "gone outright" is actually fine.

§5 Try it in 20 minutes

LAB · Estimated 20 minutes

Mission: run the compaction trigger from §3 against a seeded session, then use the §3 transcript endpoint to prove a tagged constraint survives compaction verbatim while an untagged, older fact gets folded into the summary and is only visible via the raw view.

  1. Run Postgres locally and create session_messages and session_checkpoints from the DDL in §3.
  2. Seed one session with 30 messages: message 2 tagged constraint ("never touch the legacy billing module"), message 10 an ordinary untagged turn stating a rejected approach, messages 25–30 as the "recent window."
  3. Wire up maybe_compact with a stubbed model.generate that returns a fixed CompactionSummary mentioning the rejected approach from message 10 but, deliberately, not the constraint from message 2 (constraints don't need to appear in the summary — they're carried as anchors, not summarized).
  4. Run the compaction pass, then wire up the Express endpoint from Layer 2.
  5. Fire: curl "localhost:3000/sessions/SESSION_ID/transcript?view=compact". Confirm the response's recent array includes the anchor message from step 2 verbatim, and the checkpoint.summary contains the rejected approach text.
  6. Fire: curl "localhost:3000/sessions/SESSION_ID/transcript?view=raw". Confirm all 30 original messages are present, including message 10 in full — proving nothing was deleted, only excluded from the active/compact view.

Acceptance criteria:

  • ✅ The compact view's active context includes the constraint message verbatim, unchanged by the summarization call.
  • ✅ The compact view's checkpoint.summary.rejected_approaches contains the fact from message 10.
  • ✅ The raw view returns all 30 original rows regardless of what the compact view shows — archiving never deletes.

§6 Design-review question

Staff+ interview / design-review bank

A teammate suggests skipping the anchor-tagging mechanism entirely: just re-run the structured summarization prompt on the full transcript every time, including previously summarized content, so nothing is ever permanently excluded from consideration. What's wrong with that, and what would you do instead?

Strong answer sketch: this sounds safer — "always re-derive from everything" — but it reintroduces cost and latency that scale with total session length rather than with the compactable window, which defeats the entire point of compacting at turn 90 of a session that might run to turn 400; by the end you're re-summarizing hundreds of turns on every single compaction pass instead of a bounded window. Worse, it doesn't actually fix the loss problem, because a summarization call over a longer transcript is not more faithful than one over a shorter transcript — it's the same lossy process with a bigger input, and nothing stops the same fact from being dropped again on pass two even though it "should" have another chance to be noticed. Anchors solve a different problem than "give facts more chances to be seen": they remove specific, harness-identified categories of fact (constraints, decisions) from the summarization process altogether, so their survival doesn't depend on any model's judgment, repeated or not. The right fix keeps re-summarization scoped to the bounded compactable window (§3), and if the team's real worry is "what if we tagged the wrong things as anchors," the answer is auditing tag coverage against the archived raw transcript, as failure mode 2 describes — not asking a summarizer to try harder over more tokens.
